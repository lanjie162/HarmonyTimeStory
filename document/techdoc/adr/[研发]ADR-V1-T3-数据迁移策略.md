# ADR-V1-T3 数据迁移策略

- **状态**：Accepted
- **日期**：2026-05-06
- **任务**：V1-T3
- **关联**：`document/plan/2026-05-06-V1-T3-迁移策略与ADR执行清单.md`

## 1. 背景与问题

V1 进入 V2 前，需要保证最小数据模型可稳定演进。当前工程尚未形成统一迁移机制，若直接改表或跳版本升级，将导致数据语义不可追溯、失败恢复困难、重复迁移风险不可控。

## 2. 决策

### 2.1 版本管理策略

- 使用整数 `schemaVersion` 管理结构版本。
- 迁移仅允许逐版本顺序执行，不允许跳步直升。
- 首批执行链路固定为 `v1 -> v2`，后续版本按同模式扩展。

### 2.2 兼容策略

- 新增字段必须提供默认值或允许空值，再在后续版本收紧约束。
- 新增索引采用可重复执行语义，重复执行不报错。
- 迁移步骤中禁止直接删除仍被业务引用的关键字段。

### 2.3 失败与回滚策略

- 每个迁移步骤前执行前置校验，失败即中断，不继续执行后续步骤。
- 迁移失败后进入安全停止状态，不写入目标版本号。
- 对可恢复步骤允许“修复后重跑”；重跑必须满足幂等。

### 2.4 幂等策略

- 每一步迁移按“是否已应用”标记判断，已应用则跳过。
- 迁移步骤重复执行不得重复改写同一结构或数据。
- 日志必须标记 `applied` / `skipped` / `failed`。

### 2.5 开发期与发版前口径

- 开发期允许清库以提升迭代效率，但迁移链路仍须可执行。
- 提测前与发版前冻结 schema，必须具备完整迁移链路和演练证据。

## 3. 约束与护栏

- 禁止跳过版本直接改表而不登记迁移步骤。
- 禁止在未完成演练证据前宣告可进入 V2。
- 禁止静默破坏历史数据语义（特别是引用关系与计数字段）。

## 4. 通过标准

- 迁移执行后版本号正确更新到目标版本。
- 迁移结构校验通过，关键数据对账通过。
- 失败分支可复现，且恢复后可重试成功。

## 5. 风险与处置

- **半更新风险**：步骤中断导致部分成功；处置为中断保护 + 版本号不推进。
- **非幂等风险**：重复执行破坏数据；处置为步骤级幂等检查与跳过。
- **样本不足风险**：仅空库演练造成假通过；处置为最小闭环样本强制纳入演练。

## 6. 结果

本 ADR 作为 V1-T3 冻结决策，约束后续迁移代码实现与验收判定。
