## 模块评审结论

### M1 · V2.2 数据模型补全版

**结论**：✅ **通过**

V2.2（9 任务，M+）补全 Person/Story 完整字段集（§6.1）、删除+级联（§6.3）、移除照片（PE-05/ST-05）、sortOrder 合规（§6.2）、孤儿 GC（§6.3）、导入面包屑/上限阻断（UI 原型 §4.5A/§4.5D）。

**评审意见**：
- sa：字段扩展为向后兼容，schema V2 迁移需注册新迁移步骤。T5 sortOrder 的 taken_at_cache 写入依赖 photoAccessHelper API——真机兼容风险容忍，DoD 允许有条件通过（模拟器不可测时以人工抽样为准）。
- dev：T3 删除是最大新接口（4 个新 API），与其他任务无竞合。
- qa：9 条 DoD 均可测试；T5 需真机。
- ued：T7 导入面包屑对齐 UI 原型 §4.5A；T8 对话框阻断对齐 §3。

---

### M2 · V2.3 异常与合规版

**结论**：✅ **通过**

V2.3（4 任务，M）补全失效态 ER-01（§8.6）、权限拒绝引导 ER-02（UI 原型 §6 + 交互评审 M6）、隐私说明页（§8.7 PR-01/02 + 交互评审 M1）、崩溃上报（§5.1 + §14）。

**决策点 D-M2-1**：崩溃上报方案

| 选项 | 说明 |
|------|------|
| 方案 A：系统 FaultLogger（内置） | 零外部依赖，MVP 阶段够用 |
| 方案 B：AppGallery Connect 崩溃服务 | 更完备但引入外部 SDK |

**结论**：✅ **采纳方案 A（内置 FaultLogger）**

**评审意见**：
- sa：UriHealthChecker 可用 photoAccessHelper.getAsset 区分"本地已删"与"云端不可访问"（不同错误码）。
- dev：T1/T2 可合并处理。T3 静态页独立无依赖。
- qa：ER-01 需构造 URI 已删除和云端不可访问两种条件。T2 需先拒绝媒体权限。
- ued：隐私页 6 板块对齐 UI 原型 §4.7。

**与 V1-D7 的一致性声明**：V1-D7（V1 不接崩溃 SDK）不构成 V2.3-T4 的阻碍——该决策时 V1 底座版的暂时排除，MVP §5.1 和 §14 明确要求崩溃上报纳入 MVP。

---

### M3 · V2.4 端侧AI管线版

**结论**：✅ **通过**

V2.4（5 任务，XL）全量实现人脸检测/聚类（IM-06）、导入条件真实逻辑（IM-04/05/07）、人脸开关双选项（IM-10）、形态三真实聚类（IM-03）、低端机降级（IX-02）。

**决策点 D-M3-1**：V2.4 退出条件

| 选项 | 说明 |
|------|------|
| 单设备真机功能通过 | MCP 证据记录，大规模性能验证留 V3 C7/C8 |
| 多设备降级验证 | 需至少 low/mid 两档设备 |

**结论**：✅ **退出条件为 MCP 单设备真机功能验证通过**；多设备降级和大规模性能验证归入 V3（C7/C8）。

**评审意见**：
- sa：T1 VisionImpl 可拆为 T1a（单图检测）+ T1b（聚类引擎）。HarmonyOS 12 不直接提供聚类 API，需应用层自行实现特征向量聚类或 pairwise 比对。实际工作量偏 L~XL 边界。
- dev：T2 导入条件段是 ImportPage 最大改动量（骨架 toggle -> 真实 DatePicker + 地点过滤 + AND 筛选）。
- qa：V2.4 测试依赖真机人脸能力，模拟器不支持 faceDetector。
- ued：全部 UI 已对齐——T2 条件页(§4.5B)、T4 建议页(§4.6)、T3 设置页(§4.8)。

---

**结论**：✅ **通过**
