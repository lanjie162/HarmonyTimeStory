## 1. 文档目的与阅读对象

本文档界定 **MVP 功能边界、数据关系、交互原则与验收要点**，与白皮书的关系为：**白皮书管战略，本文档管首版交付方向**。实现细节（具体 API、类名）以鸿蒙 SDK 与项目技术设计为准。

**§11** 为本轮评审已关闭决策摘要；**§14** 为原推荐补强项，**产品已确认**（仅崩溃上报纳入 MVP，见 §5.1 / §5.2）。

---

## 2. 产品目标

1. **人物**与**故事**是两个 **可独立使用** 的一级能力：允许仅使用人物、仅使用故事，或两者同时使用。
2. 人物与故事之间支持 **多对多** 关联，表达「谁与哪个故事相关」，**不依赖** 是否已有照片。
3. **照片**以 **媒体引用** 形式管理：**不复制用户原图、不向自有服务端上传原图**；通过系统 **PhotoPicker** 引用用户在系统相册中可见的资源（含 **本地与华为云相册** 中经系统统一呈现的资源）。App 层 **不调用华为云等云 SDK 直接读写云端原图**。单张照片的归属支持 **仅人物、仅故事、或同时关联人物与故事**（通过独立关联表实现）。
4. **相册批量导入 + 端侧智能关联** 支持 **三种形态**（见 **§8.4**）：**① 按故事定向**（指定 `storyId` + 条件，后台扫描后用户确认写 **`StoryPhoto`**）；**② 按人物定向**（指定 `personId` + 条件，后台扫描后用户确认写 **`PersonPhoto`**）；**③ 后台挖掘**（对已入库资源定时/闲时扫描，产出 **人脸→人物** 关联 **建议**，**必须用户确认** 后才写 **`PersonPhoto`**）。**不得**未经确认自动写边。

---

## 3. 设计原则

| 原则 | 说明 |
|------|------|
| 存储不动 | 应用不批量落盘用户原图；持久化以引用与业务元数据为主；**含云相册资源亦仅持系统引用，不另存副本。** |
| 模块独立 | 无人物也可建故事、加图；无故事也可维护人物与人物侧照片。 |
| 关联可选 | 人物—故事、人物—照片、故事—照片 均可为 0。 |
| 智能不越权 | 识别仅产生候选；**不得**未经确认批量写入。 |
| 人脸本地 | 人脸相关计算在端侧；单独开关与说明；特征若本地缓存须可清理并在隐私说明中描述。 |

---

## 4. 术语

| 术语 | 定义 |
|------|------|
| 人物 | 一条人物档案，可多存。 |
| 故事 | 一条故事单元，可多存。 |
| 人物—故事关联 | 多对多：一故事多人、一人物多故事；仅存关系，不代替照片挂载。 |
| 照片引用（PhotoRef） | 对 **单一系统媒体资源** 的一条应用内记录（系统 URI/资源标识、可选拍摄时间缓存等）；**同一系统资源在应用内对应一条 PhotoRef（去重）**，通过边表挂到故事/人物。**可指向本地或经系统相册呈现的华为云相册资源；App 层不感知物理存储位置。** |
| 相册导入与智能关联 | 批量选图后经 **导入页** 指定落点（故事或人物）与 **关联条件**，**后台扫描** 后 **候选 + 用户确认** 写边；另含对已入库资源的 **后台人脸挖掘建议** + **确认** 写边。详见 **§8.4**。 |

## 5. 范围

<a id="mvp-sec-5-1"></a>

### 5.1 本 MVP 包含

- 人物：列表、新建、编辑、删除；人物详情（含已关联故事、已关联照片）。
- 故事：列表、新建、编辑、删除；故事详情（元信息、有序照片、已关联人物）。
- 人物 ↔ 故事：多对多建立与解除；防重复。
- 照片：通过系统 **PhotoPicker** 手动选图；**StoryPhoto**、**PersonPhoto** 独立维护；同一 PhotoRef 可同时挂在 **多故事、多人物** 上（每侧各自 `sortOrder`，规则见 §6.2）。
- **§8.4 三种导入形态**：故事定向、人物定向、后台人脸建议；时间 / 地点 / 人脸等条件与 **AND** 规则、人脸双模式、进度 / 取消 / 重试、老机型降级。
- 隐私与说明页；系统删图、云端资源不可访问等 **失效态**。
- **崩溃与错误栈上报**：在 **不上传原图、可匿名/去标识化** 前提下采集崩溃栈（或厂商通道等价能力）；隐私说明须描述采集范围、保留周期与用户可控项（与 §14 一致）。
- **主题**：跟随系统深色/浅色模式（最低适配，见 §9）。

<a id="mvp-sec-5-2"></a>

### 5.2 本 MVP 不包含

App 层主动调用 **华为云 SDK** 或等价方式 **直接读写云端原图**；向自有服务端 **上传原图**；NAS 专有协议深度接入；**多人账号协作 / 邀请 / 权限**；跨设备同步；服务端 AI；自动成片 / PDF；疫苗与医学模板；向第三方上传原图用于识别。

**规划至 V1.1（本 MVP 不做）**：按人物显示名、故事标题的 **最小本地搜索**（单列表过滤）；业务元数据 **JSON 导出**（不含原图）。若路线图调整，须在升版需求中显式修订本节。

## 6. 数据模型（需求层必须遵守）

<a id="mvp-sec-6-1"></a>

### 6.1 实体

- **Person**：人物主数据  
  - **显示名**（必填）  
  - **类型**（枚举：宝宝 / 成人 / 长辈 / **其他**，默认 **其他**）  
  - **头像**（可选：从 PhotoPicker 选一张，写入 `PersonPhoto` 或单独头像引用字段由技术设计二选一并文档化；需求层要求「可选展示」）  
  - **生日**（可选）  
  - **性别**（可选枚举：**男 / 女 / 未填**）  
  - **备注**（可选）  

- **Story**：故事主数据  
  - **标题**（必填）  
  - **描述**（可选）  
  - **时间范围**（开始日期、结束日期；允许单日即起止相同）  
  - **封面**：默认取故事内首张（按 §6.2 展示顺序的第一张）；支持用户 **手选** 封面  
  - **地点**（可选：用户 **手填** 字符串）  
  - **自定义标签**（可选：字符串列表或等价，由技术设计锁定）  

- **PhotoRef**：单一系统媒体资源对应一条（按系统 URI/资源 ID 去重）；**不存原图文件副本**。

### 6.2 关系表

| 关系 | 含义 | 关键字段 |
|------|------|----------|
| **PersonStory** | 人物—故事多对多 | `personId`, `storyId`；可选 `role`、`sortOrder`；**唯一** `(personId, storyId)`。 |
| **StoryPhoto** | 故事—照片多对多 | `storyId`, `photoRefId`, `sortOrder`（故事内顺序）。**同一故事内** 同一 `photoRefId` 不重复。 |
| **PersonPhoto** | 人物—照片多对多 | `personId`, `photoRefId`, `sortOrder`（人物侧顺序）。**同一人物内** 同一 `photoRefId` 不重复。 |

**sortOrder 与展示顺序（已定稿）**  
- `StoryPhoto`、`PersonPhoto` 的列表顺序：**按拍摄时间降序（晚 → 早）** 写入与展示；若某条无拍摄时间元数据，则按 **添加时间降序** 作为次级排序（技术设计须明确字段来源）。  
- **MVP 不提供** 用户拖动重排。

**单故事照片上限（已定稿）**  
- 单故事 `StoryPhoto` 条数 **上限 1000**；达到上限后 **阻断继续写入**（含手动与 **§8.4 形态一** 经确认的写入），并给出友好提示（建议拆分故事）。

### 6.3 删除与级联（验收依据）

- 删除 **人物**：删除该人物所有 `PersonStory`、`PersonPhoto`；**不删除** Story、`StoryPhoto`、PhotoRef（若 PhotoRef 仍被某故事引用则保留 PhotoRef，仅删人物边）。
- 删除 **故事**：删除该故事所有 `StoryPhoto`、该故事在 `PersonStory` 中的行；**不删除** Person、`PersonPhoto`、仍被其他故事或人物引用的 PhotoRef。
- 从故事 **移除** 一张照片：仅删除对应 `StoryPhoto`。
- 从人物 **移除** 一张照片：仅删除对应 `PersonPhoto`。
- **PhotoRef 孤儿处理（已定稿）**：当 PhotoRef 不再被任何 `StoryPhoto` 或 `PersonPhoto` 引用时，**标记为孤立**；在应用 **启动后或闲时** 执行 **后台 GC** 物理清理孤立 PhotoRef（及关联的本地人脸缓存键，若有）。不要求在移除边的同一事务内立即物理删除。
- 删除 PhotoRef（若实现显式删除入口）：须同时清理所有 `StoryPhoto`、`PersonPhoto`。

**同一 URI 多挂（已定稿）**  
- 允许同一系统资源（同一 PhotoRef）出现在 **多个故事**、**多个人物** 中。

### 6.4 与「仅人物 / 仅故事」的一致性

- 允许存在 **无任何 StoryPhoto、也无任何 PersonStory** 的 PersonPhoto（纯人物相册）。
- 允许存在 **无任何 PersonPhoto、也无 PersonStory** 的 StoryPhoto（纯故事相册）。
- **PersonStory** 可与照片挂载 **完全独立**（例如先标记「谁参与团建」再慢慢加图）。

---

## 7. 信息架构与导航

- 主导航包含 **人物**、**故事**（文案以实现为准）。
- 不强制「先建人物」或「先建故事」；首次启动可用轻引导说明两种入口（可选）。
- **列表默认排序**：故事列表、人物列表默认按 **最近更新降序**（`updatedAt` 或等价字段，由技术设计锁定）。

## 8. 功能需求

### 8.1 人物模块（P0）

| ID | 需求 |
|----|------|
| PE-01 | 人物列表；新建、编辑、删除（删除规则见 §6.3）。 |
| PE-02 | 人物字段见 **§6.1 Person**（显示名、类型、头像可选、生日、性别、备注）。 |
| PE-03 | 人物详情：展示档案；**已关联故事**列表（来自 PersonStory）；**人物照片**列表（来自 PersonPhoto，顺序见 §6.2）。 |
| PE-04 | 人物侧 **添加照片**：系统选图 → 创建或复用 PhotoRef → 写入 `PersonPhoto`（可多张；顺序见 §6.2）。 |
| PE-05 | 人物侧 **移除照片**：仅删 `PersonPhoto`。 |
| PE-06 | 人物侧维护 **PersonStory**：关联/解除与故事的关联（可与故事侧入口二选一为主，但能力须覆盖）。 |

### 8.2 故事模块（P0）

| ID | 需求 |
|----|------|
| ST-01 | 故事列表；新建、编辑、删除（级联见 §6.3）。 |
| ST-02 | 故事字段见 **§6.1 Story**（标题、描述、时间范围、封面、地点手填、自定义标签）。 |
| ST-03 | 故事详情：元信息；**故事内有序照片**（StoryPhoto）；**已关联人物**（PersonStory）。 |
| ST-04 | 故事侧 **添加照片**：系统选图 → PhotoRef → `StoryPhoto`。 |
| ST-05 | 故事侧 **移除照片**：仅删 `StoryPhoto`。 |
| ST-06 | 故事侧维护 **PersonStory**：添加/移除人物；同一故事对同一人仅一条关联。 |
| ST-07 | 单故事 **StoryPhoto** 数量 **上限 1000**（见 §6.2），超出 **阻断写入** 并提示。 |

### 8.3 照片与双挂（P0 / P1）

| ID | 需求 |
|----|------|
| PH-01 | 同一系统资源对应 **一条** PhotoRef；重复添加同一资源到同一故事或同一人物时 **不重复建边**（可提示已存在）。 |
| PH-02 | 支持 **仅** 挂故事、**仅** 挂人物、**同时** 挂两者：通过分别写入 `StoryPhoto` 与 `PersonPhoto` 实现。**「加入故事时一键同时加入某人物」的 UI 为 P1**；MVP 数据模型与写入路径须已就绪，用户可在人物侧、故事侧分别添加同一 PhotoRef。 |
| PH-03 | 在故事/人物详情中，对同一 PhotoRef 的展示为 **同一底层引用** 的一致预览；一侧移除不自动移除另一侧（MVP 默认 **不联动**）。 |

### 8.4 相册导入与智能关联（P0）

**8.4.0 共性**

- 用户通过系统 **PhotoPicker** **一次选择多张** 图片进入 **导入页**（入口可来自故事侧、人物侧或统一「添加」入口，**交互稿锁定**）；仅持久化 **系统引用 + 业务元数据**，不落盘原图（§2、§6）。  
- 导入流程采用：**选图 → 导入准备（元数据导入）→ 条件页 → 后台扫描 → 候选确认**。导入准备阶段仅写入 `PhotoRef` 与元数据，不写业务边。  
- 凡依赖 **端侧扫描**（元数据、人脸等）的筛选，均在 **后台任务** 执行；须满足 **§8.5**（不长期阻塞 UI、可取消）与 **§9** 降级。  
- **智能不越权**：扫描与匹配只产生 **候选或建议**；**未经用户在界面中明确确认，不得写入** `StoryPhoto` / `PersonPhoto`。  

**8.4.1 形态一：按故事定向导入**

- 用户在导入页指定 **目标故事 `storyId`**，并配置 **关联条件**（至少包含 **时间**；**地点**、**人脸** 等按产品勾选组合，能力要求见下表 **IM-04～IM-07**）。  
- 流程：**选图** → **导入准备（元数据导入）** → **配置条件** → **后台扫描与筛选** → **候选网格**（全选 / 反选 / 单张剔除）→ 用户 **确认** → 批量创建或复用 **PhotoRef** 并写入 **`StoryPhoto`**。  
- 写入后故事内顺序须符合 **§6.2**；受 **ST-07**（单故事 **1000** 张上限）约束。  

**8.4.2 形态二：按人物定向导入**

- 用户在导入页指定 **目标人物 `personId`**，并配置 **关联条件**：以 **端侧人脸** 为主（**人脸簇**、**参考图** 两种模式须齐备，**默认人脸簇**，可切换；参考图可含 **与人物当前头像比对** 等由技术实现、验收可测的语义）。可 **辅以时间** 等元数据条件缩小候选（**具体可配项与默认值由交互稿锁定**）。  
- 流程与形态一相同（含导入准备阶段）→ 确认后写入 **`PersonPhoto`**（及必要时的 **PhotoRef** 去重，**PH-01**）。人物侧展示顺序 **§6.2**。  
- **不设**「仅在故事导入里顺带挂人物」的单一入口替代本形态；人物侧挂载以本形态或 **§8.3** 手动加图覆盖。  

**8.4.3 形态三：后台人脸关联挖掘（建议 → 确认）**

- 应用对 **已入库** 的 **PhotoRef**（已被挂载或已完成索引管线、具体范围由技术设计锁定并写入技术方案）做 **定时或闲时** 扫描，挖掘 **「某图可能属于某人物」** 的 **建议列表**（仅端侧、人脸开关受 **IM-10** 约束）。  
- **必须**提供独立界面或入口，使用户能 **逐条或批量审阅建议**；仅对 **用户明确接受** 的项写入 **`PersonPhoto`**；拒绝则丢弃该建议、**不**写边。  
- 本形态 **不从** 导入页批量选新图开始；与 **8.4.1 / 8.4.2** 互补。  

**8.4.4 条号需求（IM-01～IM-10）**

| ID | 需求 |
|----|------|
| IM-01 | **形态一（§8.4.1）**：须实现；入口须使用户能明确当前目标故事。 |
| IM-02 | **形态二（§8.4.2）**：须实现；入口须使用户能明确当前目标人物。 |
| IM-03 | **形态三（§8.4.3）**：须实现；**禁止**自动写 `PersonPhoto`；仅 **确认后** 写入。 |
| IM-04 | **时间**（形态一、二可选用）：至少 **自定义起止日期** 与 **单日**（或等价集合，交互锁定）。 |
| IM-05 | **地点**（形态一、形态二若启用）：基于可用位置信息过滤；无位置数据归入 **未知** 或仅按其余已启用条件评估，**不得崩溃**。 |
| IM-06 | **人脸**（形态一、二、三涉及扫描处）：端侧；**首次**独立开关与说明；**人脸簇** 与 **参考图** **均须实现**，**默认人脸簇**，用户可切换；**仅对用户确认的候选/建议写入边表**。 |
| IM-07 | 形态一、二在启用多条件时：**MVP 默认 AND（交集）**；**不提供 OR**（若未来需要，另起版本评审）。 |
| IM-08 | **候选 / 建议呈现与写入**：网格或等价列表支持 **全选 / 反选 / 单条剔除**；确认后批量 **创建/复用 PhotoRef** 并写 **`StoryPhoto` 或 `PersonPhoto`**（与当前形态一致）；写入后顺序符合 **§6.2**。 |
| IM-09 | **长任务**（三种形态凡触发后台扫描与批量写库）：**进度**、**取消**、**失败重试**；降级见 **§9** 与 [HarmonyOS平台能力调研表-v1.md](../techdoc/HarmonyOS平台能力调研表-v1.md#harmonyos-sec-2-5)。 |
| IM-10 | **关闭人脸开关**：停止 **新的人脸索引 / 聚类 / 形态三新建议生成**（实现须界定边界）；关闭时需提供双选项：**仅关闭后续分析** / **关闭并清空现有建议**（默认推荐清空）；已落库的引用与已写入的 **`PersonPhoto`/`StoryPhoto`** **保留**。 |

### 8.5 索引与任务（P0）

| ID | 需求 |
|----|------|
| IX-01 | 媒体/人脸索引类任务不长期阻塞 UI；可取消。 |
| IX-02 | 低端机降级（如弱化或关闭人脸、降低扫描范围）；具体阈值见调研表 **降级矩阵** 回填列。 |

### 8.6 异常与系统相册（P0）

| ID | 需求 |
|----|------|
| ER-01 | **本地已删除或权限失效**：凡引用该 PhotoRef 的展示均为 **失效态**，可单条移除边或删除引用；不崩溃。 **云端或网络原因暂不可访问**（如 URI 对应资源在华为云相册、当前断网或云端已删）：须与「本地已删除」区分文案（如「暂不可访问」「云端资源不可用」），同为可操作的失效态，不崩溃。 |
| ER-02 | 权限拒绝：明确提示，不死循环弹窗。 |

### 8.7 隐私与说明（P0）

| ID | 需求 |
|----|------|
| PR-01 | 静态页：不存原图副本、不上传自有服务端、仅系统相册引用（含云相册由系统呈现）、本地数据与人脸能力范围说明。 |
| PR-02 | 若缓存人脸特征/索引：提供清除或与卸载说明一致。 |

### 8.8 里程碑（P1，可选）

人物下「日期 + 标题 + 备注」非医疗记录；与故事、照片无强制绑定。纳入则单独列验收用例。

## 10. 验收检查清单（摘要）

- [ ] 仅人物：多人物、人物加图、无故事，可用。
- [ ] 仅故事：多故事、故事加图、无人物，可用。
- [ ] 人物—故事多对多：增删、防重复、删人物/删故事级联符合 §6.3。
- [ ] 同一 PhotoRef 同时存在于 `StoryPhoto` 与 `PersonPhoto`；单侧移除另一侧行为符合 PH-03。
- [ ] 同一 PhotoRef 可挂 **多个故事**；手动选图去重。
- [ ] **§8.4 形态一**：指定故事 + 条件 + 后台扫描 + 候选确认 → `StoryPhoto`；时间、地点、人脸及 **AND** 至少一条用例；**人脸簇 / 参考图** 可切换。
- [ ] **§8.4 形态二**：指定人物 + 条件（含人脸/参考）+ 后台扫描 + 确认 → `PersonPhoto`；与 **PH-01** 去重一致。
- [ ] **§8.4 形态三**：后台建议列表 → **仅确认项** 写入 `PersonPhoto`；**无**静默自动挂载。
- [ ] 三种形态确认写入后顺序均符合 **§6.2**（拍摄时间降序为主）；形态一受 **ST-07** 1000 张上限。
- [ ] 系统删图、本地失效态；**云端 / 网络不可访问** 失效态与文案区分。
- [ ] 首次安装、**拒绝媒体权限**、**零数据** 三种空态可走通。
- [ ] 单故事 **1000** 张上限：第 1000 张成功、第 1001 张 **阻断** 并提示。
- [ ] 杀进程后数据一致；不向自有服务端上传原图。
- [ ] **崩溃上报**：在不上传原图、可匿名前提下采集崩溃栈（或等价）；隐私页与系统设置路径可核对采集说明（与 §5.1 / §14 一致）。
- [ ] 人脸开关与说明；PR-02 清除能力（若实现缓存）。
- [ ] **约 5 万张** 媒体抽样下 §9.2 性能门槛（按调研表机型分档）。
- [ ] 卸载重装后：业务数据 **是否可恢复** 须在 PR/帮助页与验收预期 **一致**（默认不可恢复须写明）。

## 12. 风险与依赖

**MVP 最低系统能力**：**API version 12**（与工程 `compatibleSdkVersion` **5.0.0(12)** 一致，见 [build-profile.json5](../../build-profile.json5)）。低于该档位的设备 **不在 MVP 支持范围**（应用商店与帮助页须可述）。

鸿蒙媒体库、PhotoPicker、端侧人脸能力以 **当前 SDK + 真机** 为准；技术设计须附 **API 与降级矩阵**（模板见 [HarmonyOS平台能力调研表-v1.md](../techdoc/HarmonyOS平台能力调研表-v1.md#harmonyos-sec-2-5)）。人脸准确率依赖人工勾选；地点缺失率高，需文案与 UI 预期管理。

**依赖**：「通过 PhotoPicker 统一引用云相册」依赖系统对云相册的抽象；若调研发现需单独适配，须在调研表 **风险与假设** 节记录，并回修 §5.1 / §6.1 措辞。

**回填**：**最低 API** 已由 **V4.2** 裁定；**机型分档、云策略摘要、性能放宽系数** 以 **[TODO-跟踪 §C](../TODO-跟踪.md#todo-sec-c)** 执行完毕后调研表结论为准，并同步修订 **§9.4** 各行「待测试阶段」格（及必要时 **§8.4** / §8.5 的交叉引用）。**V4.3** 后 **§8.4** 含三种导入形态，技术方案须单列 **形态三** 的扫描范围、通知与确认 UX。
