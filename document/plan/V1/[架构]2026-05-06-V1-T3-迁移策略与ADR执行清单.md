# V1-T3 最小数据模型与迁移策略（ADR 执行清单）

**所属版本**：V1（底座版）  
**任务编号**：V1-T3  
**任务目标**：完成最小数据模型迁移策略落库，交付 `ADR + 可执行迁移骨架 + 一次 v1->v2 真实迁移演练`。

## 1. 交付边界与角色分工

### 1.1 本任务范围（In Scope）

1. 形成迁移 ADR，明确版本演进、回滚与风险策略。  
2. 落地可执行迁移骨架（具备版本判断、步骤执行、失败中断与日志）。  
3. 完成一次 `v1 -> v2` 真实迁移演练并留痕。  
4. 输出可用于联合验收的证据清单与判定口径。

### 1.2 非本任务范围（Out of Scope）

1. V2 全量功能实现。  
2. 与迁移无关的数据模型重构。  
3. 真机专项矩阵验证（仍归 C 区任务）。

### 1.3 角色责任

- `architect`：定义并冻结迁移策略（ADR）、验收口径、风险与升级机制。  
- `software-engineer`：实现迁移骨架并执行演练。  
- `qa-engineer`：验证迁移行为、异常分支与证据完整性。  
- `product-manager`：确认迁移结果不突破版本边界。

## 2. ADR 必备决策项（architect 主产出）

> ADR 建议文件：`document/techdoc/adr/ADR-V1-T3-数据迁移策略.md`（或等价路径）

### 2.1 问题定义

1. 为什么需要从 `v1` 迁移到 `v2`（业务与技术触发）。  
2. 迁移成功标准与失败标准。  
3. 迁移窗口（开发期 / 提测前 / 发版前）和适用环境。

### 2.2 决策结论（必须明确）

1. 版本管理策略：`schemaVersion` 规则与升级序列。  
2. 兼容策略：新增字段默认值、非空约束引入方式、索引演进策略。  
3. 回滚策略：失败时中断条件、回退方式、数据保护边界。  
4. 幂等策略：重复执行迁移不破坏数据一致性。  
5. 开发期与发版前口径：开发期允许清库，提测/发版前冻结 schema 并补齐迁移链路。

### 2.3 技术护栏

1. 禁止“静默破坏”历史数据语义。  
2. 禁止跳过版本直接改表而不登记迁移步骤。  
3. 禁止在未完成演练证据前宣告可进入 V2。

---

## 3. 可执行迁移骨架最小标准（架构验收口径）

迁移骨架至少包含以下能力：

1. 启动时可读取当前 DB 版本并判断目标版本。  
2. 迁移步骤按版本顺序执行（如 `v1_to_v2`）。  
3. 每步迁移具备前置校验与失败中断。  
4. 关键步骤输出日志（开始/结束/失败原因/耗时）。  
5. 支持最小回滚或安全停止策略。  
6. 迁移后执行结构与数据完整性校验。

---

## 4. `v1 -> v2` 真实迁移演练口径

### 4.1 演练前置

1. 准备 `v1` 数据样本（包含最小闭环数据：人物、故事、照片引用关系）。  
2. 固定演练环境与构建版本，避免结果不可复现。  
3. 明确演练脚本/命令与执行人。

### 4.2 演练步骤（最小）

1. 导入/构造 `v1` 数据。  
2. 执行迁移至 `v2`。  
3. 校验 schema 结果。  
4. 校验关键业务数据一致性（记录数、主键关系、唯一约束、引用计数等）。  
5. 触发一次异常分支演示（如中断后重试）并记录行为。

### 4.3 通过判定

1. 迁移成功完成且无未处理异常。  
2. 核心业务表结构符合目标版本定义。  
3. 关键数据对账通过（抽样与汇总均通过）。  
4. 日志与证据可追溯到具体版本与执行批次。

## 6. 风险清单与升级机制

| 风险 | 触发信号 | 影响 | 处置 |
|------|----------|------|------|
| 迁移中断导致数据半更新 | 迁移日志出现部分成功 | 数据不一致 | 启用中断保护，回滚或恢复后重试 |
| schema 演进无版本登记 | 发现直接改表 | 追溯失效 | 强制补齐迁移脚本与 ADR 记录 |
| 重复执行非幂等 | 二次迁移结果异常 | 线上风险 | 增加幂等校验并回归验证 |
| 演练样本覆盖不足 | 仅空库或单一数据 | 假通过 | 增补最小闭环样本后重跑 |

---

## 7. V2 输入条件（迁移维度）

进入 V2 前，T3 必须满足：

1. ADR 已冻结并经架构复核通过。  
2. `v1 -> v2` 可执行迁移骨架可在目标环境稳定运行。  
3. 已完成至少一次真实迁移演练，证据齐全且对账通过。  
4. 未关闭风险项需有明确责任人与关闭窗口。
