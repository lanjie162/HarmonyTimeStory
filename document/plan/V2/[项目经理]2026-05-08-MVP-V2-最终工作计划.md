## 【编排推断】

【编排推断】计划目标：基于现有MVP与V1放行证据，形成可执行、可验收、可追溯的V2版本交付计划；拟征询子代理：`pm -> sa -> dev -> qa -> ued`。

### 输入基线（document 全量参考后的V2聚焦）

- 需求基线：`document/requirement/时光故事-MVP需求说明-v4.md`（V4.4），V2主链需覆盖人物/故事P0主流程与`§8.4`导入链路关键状态，且满足`IM-09`取消、`IM-10`策略一致性。
- 技术基线：`document/techdoc/时光故事-MVP技术方案-v1.md`（V1.4.1），保持四层边界、`IImportService/ISuggestionService`职责分离、`PhotoOwnerLink`主路径与`ST-07/PH-01`规则不退化。
- 交互基线：`document/design/时光故事-MVP-UI原型-v1.md`（v1.2），导入流程必须遵循“选图->导入准备->条件->扫描->候选确认”，人物页顶栏建议入口与取消语义一致。
- 执行基线：`document/plan/2026-05-06-MVP研发工作计划-定稿.md`明确V2为“功能主链版（XL）”，并锁定`B2+C9`同版闭环。
- 质量基线：`document/evidence/V1/V1-QA/V1-QA-final-gate-20260506-2321.md`显示V1已通过并提供V2输入条件。
- 风险与待办：`document/TODO-跟踪.md`中`B2/C9`未关闭，`C`区真机矩阵未闭环，V2需采用“主链可交付 + 真机关键证据先闭环”策略。

### 计划策略

#### 目标与范围假设

- 目标：交付“可连续操作”的人物与故事主流程，以及“可取消、可重试、可确认写入”的导入主链，形成进入V3风险收口的稳定输入。
- 范围内：
  - 人物主流程：列表/详情/基础编辑/关联故事入口/人物侧手动加图与导入入口。
  - 故事主流程：列表/详情/基础编辑/关联人物入口/故事侧手动加图与导入入口。
  - 导入主链：形态一与形态二的“导入准备+条件+扫描+候选确认”主路径打通。
  - `B2`：`cancelRequest + 应用层协作取消 + UI取消语义`实现并可观测。
  - `C9`：API12真机验证`R-05`取消语义并产出证据。
- 非目标（V2不做）：
  - 形态三全量闭环与大规模参数调优（V3处理）。
  - 真机矩阵全覆盖与所有`C2~C8/C10`收口（V3/V4处理）。
  - 体验精修与P1增强（如一键双挂UI增强）。

#### 阶段划分逻辑（V2内）

- 阶段A（功能主链成形）：人物/故事双主链可用，路由与核心数据写入稳定。
- 阶段B（导入主链收口）：导入准备、扫描、候选确认、写入与去重/上限校验打通。
- 阶段C（取消语义与真机闭环）：`B2`实现 + `C9`真机证据 + 版本结项材料。

#### 协调策略（角色顺序与目的）

- `pm`：锁定V2可验收边界，防止范围漂移。
- `sa`：锁定关键路径与技术护栏，避免跨层耦合和返工。
- `dev`：确认任务拆分粒度、并行度、实现阻塞点。
- `qa`：定义V2最小可放行用例集与证据门槛。
- `ued`：锁定导入链路状态一致性与关键交互文案。

#### 版本切分原则

- 业务价值优先：先保障“能完成核心任务”的功能主链，再补高风险关键点。
- 风险前置：`B2+C9`在V2内强绑定，不后移。
- 依赖顺序：先功能主链，再导入链细化，最后真机关键验证。

#### 复杂度评估口径（S/M/L/XL）

- S：单模块、低依赖、验证面窄。
- M：跨2-3模块或中等验证成本。
- L：跨层协同、关键路径任务、验证面大。
- XL：高不确定/高风险/多角色耦合，必须拆解并前置验证。

#### 已知信息缺口

- `C3/C4/C6`真机矩阵未闭环，影响V2之后性能和降级策略定量阈值；本计划将其登记为V2外部风险输入，不阻断V2主链交付。
- 形态三“已入库扫描边界”仍待实现首版锁定；V2仅保持接口与入口兼容，不把其作为V2准出门槛。

#### 本轮 Task 征询

- 定稿会话内未随文打包；执行派发于 **2026-05-09** 通过 **`TR-20260509-02`** 覆盖 **V2-T1～V2-T8**（见 **§可选·打包推荐** 与 **§计划任务清单「task_request」列**）。
