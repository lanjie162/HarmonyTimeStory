## 验收流程（单任务 - 顺序执行）

**语义约定**：本节以下「子代理」「取用」→ 当前会话内加载对应角色 Skill 后，以该角色输出。

### 段一：验收守门（入口）

在载入验收角色 Skill 前，先完成以下必检项并公示结论：

| 守门项 | 检查内容 |
|--------|----------|
| A. 提交完整对照 | 执行提交人是否等于派发 `owner_role`（不匹配则直接 B-n，不进入验收） |
| B. DoD 逐条查验 | 对 DoD 每条逐项查验是否已有执行侧回应（缺条则阻塞 B-n 补全） |
| C. 高约束域 / L2 声称交叉校验 | 若任务 `DoD` 含 L2 声称或命中 governance Skill 之高约束域，核对执行**提交材料**是否与其声称同级（禁止假 L2） |

守门判决：
- 全部通过 → 进入段二
- 任一项不通过 → **立即** 判定 `不通过` + `B-n`，**停止**，不进入段二

### 段二：加载验收角色 Skill 与裁决

1. 按 `acceptor_role` 列表**顺序**，逐个加载对应 Skill 并以该角色视角逐一审阅提交材料
2. 若出现「acceptor-role 多位」：**一人一句** DoD 对齐意见，串行列出，不到上一人同意不推进到下一位
3. 验收角色在验收守门中不能描述 DoD 偏移 → 必须复用经理标量或升级 **裁决空格**（通过 / 有条件通过 / 不通过）

### 段三：验收判决与统一任务列表写入

- **通过（Pass）**：将对应 `T-<n>` 状态更新为 `已验收`
- **有条件通过（Conditional）**：在 §阻塞单中登记为 `B-n`（写明条件 → 责任人 → 限定范围），将状态更新为 `已验收（条件见B-n）`
- **不通过（Fail）**：理由升级为 `B-n`（阻塞实现门），状态保留 `已执行` 或降为 `阻塞（待修复）`，异步路径停盖

### 段四：证据补强（验收判据）

在验收结论下达前，从 `acceptor_role` 视角额外完成 3 维度证据核查并归档于验收判据：

| 维度 | 要点 |
|------|------|
| 1. DoD 覆盖率与合规检验 | `权威：<path>##<sec>` 引用准改；L1/L2 无缺级；缺项即不通过 |
| 2. 用例路径与运行覆盖 | 若提交含 `ohosTest` / hdc / Hypium 路径，核对「执行至少覆盖 DoD 断定路径，且无任意关键归结偏移」 |
| 3. 可追溯性与可复现性 | 提交路径可追溯可复现；缺失复现段标注为「证据链断裂」**
