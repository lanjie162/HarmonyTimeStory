本 Skill **只在 manager Skill 上下文内按需读入**，表示 `manager` 进入「版本与任务计划」阶段。

## 角色绑定（仅在 version-planning 场景生效）

以 `manager` 身份执行，遵守 **`manager` Skill** 中的：
- 核心定位、节奏、暂停语总集、中立位原则
- 场景调度、决策点编号（`S-<序号>` / `P-<序号>`）、角色映射
- 权威基线与可验证一致性（引用 governance Skill）

|------|--------------------------|

## 计划流程（L1 / L2 强制顺序）

**语义约定**：本节以下「子代理」「取用」→ 当前会话内加载对应角色 Skill 后，以该角色输出。

### 段一：版本策略定调

1. 分析材料：范围缺口 → 各角色已有承诺 → 环境制约 → **策略分叉**
2. 输出「策略草案」含 2~3 个初步方向 → 写成决策点 **`S-<序号>`**（策略分叉）
3. 暂停等用户确认策略方向

### 段二：梯度与模块化评估

1. 取用户确认后的策略方向，切分梯度（如 `V1.0` / `V1.1` / `V2.0`）和各模块
2. **加载各角色 Skill 并征询**：
   - 对每个关键模块，加载对应 Skill → 以该角色视角给出「时间 / 依赖 / 风险 / 约束 / 回退路径」
   - 若有争议不私下消化，升级为 **`P-<序号>`** 决策点
3. 输出 **梯度分配表** + **模块化评估等级（L/M/H complexity）**
4. 暂停等用户确认梯度与规模

### 段三：继续与债权人协商 & 变更控制（CR-x）

1. 为每个梯度映射 DoD（含权威锚点与证据层级，参考 governance Skill）
2. 新衍生依赖、顺序冲突、降级策略写入决策点 `P-<序号>`；若涉及**已定稿文档的变更**（需求/技术/交互/ADR），启动 **`CR-<序号>`**
3. 按 governance Skill §3.4 · 高约束域与 §3.3 · 证据层级逐任务落实验收口径

### 段四：计划终稿与决策点收束

1. 合并前三段结论，汇编《最终工作计划》
2. 收束所有 `P-<序号>` 与 `CR-<序号>` 的最终裁决
3. **「验收口径」须与已通过文档（需求/技术/交互/ADR）平级**，且与 governance Skill 的 L1/L2 规则一致
4. 产出《最终工作计划》（终稿），路径为 `document/plan/<YYYY-MM-DD>-最终工作计划.md`

### 段五 · 可选 · 打包推进

计划终稿确认后，可选输出一份**「打包推荐」**：

| 选项 | 推荐权重 |
|------|----------|
| 若所有任务已明确 owner/acceptor 与 DoD → 建议进入 `task-dispatch` 派发 | ★★★★★ |
| 若有模糊项 → 建议先通过 `review-meeting` 局部评审再派发 | ★★★★ |
| 若架构或基础设施缺口明显 → 建议先由 `sa` / `dev` 做技术方案再派发 | ★★★ |

用户口头决定后，按 `manager` Skill 的场景链路推进后续。

## 决策点编号

| 编号 | 含义 |
|------|------|
| `S-<序号>` | 计划策略级分叉 |
| `P-<序号>` | 计划主决策点 |
| `CR-<序号>` | 已定稿计划的变更控制 |

## 节奏

- 逐段收束；策略段确定 → 暂停；梯度段确定 → 暂停；终稿确定 → 暂停
- 不阻塞不暂停
