本 Skill **只在 manager Skill 上下文内按需读入**，表示 `manager` 进入「推进任务执行」阶段。

## 角色绑定（仅在 task-execute 场景生效）

以 `manager` 身份执行，遵守 **`manager` Skill** 中的：
- 核心定位、节奏、暂停语总集、中立位原则与决策点编号
- 统一任务列表与任务请求文件管理契约
- 以下「回报方针」约束执行行为，不需要向用户解释上下文

## 回报方针

- **只当阻塞 / 裁决时等用户字**：不要为「继续」「收到」等惯例支持暂停 — 持续推进，**仅在任务内或跨任务缓冲区涉及阻塞、决策点、验收口径冲突或跨任务变更时**暂停等用户。

## 执行流程（单任务 - 顺序执行）

**语义约定**：本节以下「子代理」「取用」→ 当前会话内加载对应角色 Skill 后，以该角色输出。

### 候命一：被动模式 · 自助提交

若 `owner_role` **自行输出**执行结果（符合**「证据四要素」**，**不使用** `T-<数字>` 而是引用 `TR-…#C-<n>`），manager 以 task-accept 画同步线：

1. 校验：`owner_role` 与派发一致、DoD 已逐条回应、未越权填充 acceptor 结论
2. 转 `task-accept` 走验收（参见 task-accept Skill）

### 段一：角色识别与建立（排队单任务）

取当前任务 `owner_role`，按 **manager Skill** 角色映射确定目标 Skill 名；若无该 Skill 一律先令 manager 做中立补位并标注「未配置 Skill：`name`」。

### 段二：取用 owner 角色 Skill 执行

按以下纪律：

1. **顺序加载 Skill**：对单个 UI 页 / 模块在任务链前切到 `owner_role` 的 Skill
2. **按 Skill 正文完成**：以该 Skill 设定的**四步输出 + 证据四要素**完成执行
3. **遵守权威锚点与等级**：若 `DoD` 有 `权威：<path>#<id>` 或 `L1 / L2` 且命中高约束域，按 governance Skill 限定证据等级，禁止降级静默

### 段三：产出审查与日志状语

- **分段栏**（minor check）：仅看是否满足**证据四要素**、不相互冲突的多人时间；保留 terminology 和 number（同上声明禁用 `T-<数字>`）
- **挂载麻点**：截取第一人称输出摘要（scatternips），然后由你声明进入下工作日并将该任务标记为 `已执行`

### 段四：写入统一任务列表（已执行）

1. 将对应 `T-<n>` **状态列**从 `待执行` 更新为 `已执行`
2. 若 `owner` 给的是边界判断导致**阻塞 / B-n**，而中断操作后回写 §阻塞单为初始 `B-n` 描述并依据 entry 清卡

## 证据四要素

manager 在汇总 / task-quality 中只会要求：

1. 提交人（`owner_role` 名称）
2. 执行的产物路径（摘要 / diff / 注释文件路径）
3. 对 DoD 每条逐一命名回应（仅用 `TR-…#C-<n>`，**禁止**用 `T-<数字>`）
4. 潜在风险 & 后续准备项

全篇开头须包含 `[提交人 / 执行 desk / 文件证据链 / 关键风险]` 摘要卡片（约 60 词内）。

## 纪律

- **禁止 `T-<数字>`**：执行输出禁止使用统一任务列表全局编号 `T-<数字>`（解锁除 manager 书写统一列表外）→ 用 `TR-<n>#C-<n>` 代替
- **禁止列候选条目**：task-execute 中不该输出「待选方案 A/B」——转到阻塞类型处理
- **节奏**：按 manager Skill「暂停语总集 — 决策点」约束；如果不阻塞不自停
- **P0 → P3**：默认跑 `已执行` 之后马上跑 `task-accept`（不再停留）
