# 权威基线与可验证一致性（治理专条）

**用途**：约束任务 `DoD`、版本计划「验收口径」、质检与验收守门，使 **权威基线 ↔ 声称 ↔ 可复核证据** 一致，避免执行侧或任务口径 **静默窄于** 已定稿文档。

**关联**：[`manager.md`](./manager.md) task 包 · [`task-dispatch`](./manager/skills/task/task-dispatch.md) · [`task-quality`](./manager/skills/task/task-quality.md) · [`version-planning`](./manager/skills/version-planning.md) · [`task-accept`](./manager/skills/task/task-accept.md)；变更控制见各计划 **`CR-x`**。

---

## 1. 目的

防止 **任务 DoD / 验收口径** 在强度或范围上 **窄于或偏离** 仓库内已定稿基线；与 **`manager`** 统一任务列表契约及 **`CR-x`** 配套使用。

---

## 2. 适用范围

- **`task-dispatch`** 写入的任务明细 `DoD`
- **`version-planning`** 计划任务清单中的 **验收口径**
- **`task-quality`**（Q2 及专条脚注）
- **`task-accept`** 验收守门（声称与证据同级）

---

## 3. 术语

### 3.1 权威基线

仓库内 **已定稿、可被引用** 的段落，包括但不限于：

- `document/techdoc/*`
- `document/requirement/*`
- `document/design/*`
- `document/techdoc/adr/*`（已定稿 ADR）

任务若依赖上述内容，须在 `DoD` 中给出 **显式锚点**（见 §4），禁止仅有执行侧口头复述。

### 3.2 声称

任务 `DoD` 或计划「验收口径」中对「完成态」的描述（功能、非功能、边界均可）。

### 3.3 证据层级（L1 / L2）

| 层级 | 含义 | 适用 |
|------|------|------|
| **L1** | 演示级、低成本验证：桩、Mock、单路径脚本、抽象上下文演练、开发期环境等 | **仅当** `DoD` **明文** 允许 L1 **且** 与权威基线不冲突 |
| **L2** | 与声称 **同级** 的可复核证据：真机/真数据/平台 API 实测、指标与日志、回归自动化、符合定稿交互的走查材料等 | 当权威基线对该能力规定了验收强度，或 `DoD` 使用「完成 / 上线就绪 / 符合定稿 / 通过门禁 / 满足 §x 指标」等 **强声称** 时 **默认需要** |

**禁止**：用 **L1** 级证据单独支撑 **L2** 级声称（除非 `DoD` 显式写成「仅 L1 演示通过」且 **`pm`/计划** 已确认砍 scope 或 **`CR-x`** 留痕）。

### 3.4 高约束域

**非固定关键词表**，而是判断规则：

> 若仅凭 **L1** 证据就会在验收上「看似通过」，但 **权威基线** 或产品 **Must** 实际要求 **更强** 的验证，则该任务落入 **高约束域**。

处置：**要么** 将 `DoD` 升级为 **L2** 证据要求 **要么** 通过 **`CR-x`** 削减范围或降级基线并留档。

### 3.5 占位

任何「临时实现 / 空壳 / 非最终真相源」须在 `DoD` 登记 **到期**（`Vx-Ty` 或版本名）与 **替换条件**；跨版本延续须 **`CR-x`**。

### 3.6 冲突

计划、任务与权威基线在 **字面或强度** 上不一致 → **禁止** 执行侧自行收窄理解 → 必须通过 **`CR-x`** 修订计划、任务或基线文档之一。

---

## 4. DoD 中的权威锚点（写法）

凡依赖已定稿需求/技术/设计/ADR 的任务，`DoD` 宜包含一行或多行：

```text
权威：<仓库相对路径>#<章节或锚点 id>
```

纯工程债、纯文档整理且无基线依赖时，可在 **任务摘要** 标明「无权威锚点」，避免误套 L2。

---

## 5. 高约束域判定（简决策树）

1. `DoD` 或 **Must** 是否隐含「交付级 / 上线级」完成态？  
   - 否 → 按 `DoD` 明示的证据层级执行。  
   - 是 → 继续。
2. 权威基线是否对该能力规定了 **验证强度**（性能、安全、交互像素级、数据语义等）？  
   - 是 → 默认 **L2**；缺少 L2 证据则 **高约束域** 未闭合。  
   - 否 → 以 `DoD` 明示为准；若仍模糊，派发轮澄清为 **TBD**。

---

## 6. 占位三要素（强制登记）

允许占位时 `DoD` 至少包含：

1. **占位到期**：`Vx-Ty` 或版本闸门  
2. **替换条件**：何种实现替代占位  
3. **延期路径**：到期未替换须走 **`CR-x`**

---

## 7. 示例锚点（非穷尽）

| 类型 | 示例 `权威：` 指向 |
|------|---------------------|
| 数据与迁移 | `document/techdoc/时光故事-MVP技术方案-v1.md`  §4 |
| 需求验收 | `document/requirement/时光故事-MVP需求说明-v4.md` 指定 § |
| 交互 | `document/design/时光故事-MVP-UI原型-v1.md` 指定 § |
| 调研结论 | `document/techdoc/HarmonyOS平台能力调研表-v1.md` 指定 § |
