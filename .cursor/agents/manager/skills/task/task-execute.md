# Skill: task-execute（公共技能）

> **Skill 元信息**
>
> - **属于**：公共技能（可由 `pm / sa / dev / qa / ued / manager` 按职责调用）
> - **层级**：task 包 · **执行原子**
> - **目的**：将统一任务列表 §当前在跑 中状态为 `待执行` 的任务推进为 `已执行`
> - **核心语义**：`owner_role` **直接提交**执行结果；本技能只做守门校验与状态扭转，不做 manager 中转调度
> - **入口**：
>   - 角色自助提交：「我提交 T-12 的执行结果」
>   - 用户点名提交：「执行 T-13，并按模板回执」
> - **专属编号**：不分配新编号；引用 `T-n` / `B-x`
> - **主入参**：单条 `T-n`（状态=`待执行`）+ 结构化执行结果
> - **主出参**：`执行结果` 列写入 + 状态 `待执行 → 已执行`

## 信息边界（强制）

> 引用 [`manager.md`](../../../manager.md) §任务请求契约 → 信息边界。本技能不向上游产物写入 `T-<数字>`；上游 `task_request` 字段由 dispatch/broadcast 维护。

## 角色自驱模型

| 角色 | 职责 |
|------|------|
| `owner_role` | 直接提交执行结果（证据、步骤、偏差） |
| 调用方（任意角色） | 触发本技能并传入材料，不改变责任归属 |
| 本技能 | 守门校验、写 `执行结果` 列、扭转状态、阻塞升级 |

## DoD 守门（全过才扭转）

`待执行 → 已执行` 必须通过以下 5 项：

1. **证据齐**：执行结果覆盖 DoD 要求的全部证据条目
2. **证据可寻址**：每条证据有路径/命令/输出定位（拒绝空泛描述）
3. **证据可复现**：关键证据附最小复现步骤
4. **证据时效**：证据时间晚于任务接收时间
5. **范围对齐**：执行内容覆盖任务摘要与 DoD，不超范围、不遗漏

任一不通过：状态保持 `待执行`，输出 `【守门未通过】T-<n>：<缺项>`。

## owner 执行标准流程（固定）

`owner_role` 执行 `T-n` 时，按以下 4 步固定推进：

1. **执行前检查**：
   - 从 §当前在跑 读取 `T-n`，确认 `状态=待执行`
   - 确认 `owner_role` 已明确且与提交责任一致
   - 核对 `DoD` 与 `来源任务请求反链`，避免偏离任务边界
2. **执行中产出**：
   - 按 `DoD` 完成实现/验证动作
   - 同步沉淀可寻址证据（路径/命令/输出/截图定位）
3. **提交前自检**：
   - 按 §守门前自检清单逐项自查，缺项先补齐再提交
4. **正式提交**：
   - 按 §执行提交模板填报结构化结果
   - 触发本技能进入守门与状态扭转

## 核心工作流

### 阶段一 · 读取任务与提交材料

1. 读取 §当前在跑 对应 `T-n` 行：
   - 状态必须为 `待执行`
   - `owner_role` 必须已明确（为空时拒绝受理并提示先由 dispatch 补齐）
   - 读取 `owner_role` / `DoD` / `来源任务请求反链`
2. 接收提交材料（见 §结果交付要素）
3. 校验提交责任：
   - 提交责任人必须与 `owner_role` 一致
   - 非 owner 代提时必须显式注明“代提交，责任归属不变”

### 阶段二 · 守门检查 + 状态扭转

1. 按 DoD 守门 5 项逐条校验
2. 不通过：
   - 保持 `待执行`
   - 输出缺项与补齐要求（分类提示：证据缺失 / 可寻址不足 / 复现缺失 / 时效不符 / 范围不齐）
3. 通过：
   - 写 `执行结果` 列：`已执行 by <owner_role> @ <date>; 证据: ...`
   - 状态扭转：`待执行 → 已执行`
   - 输出：`【已执行】T-<n>，下一环：验收（task-accept）`
   - 交接口径：保留 DoD 对照、证据定位、最小复现步骤，供 `acceptor_role[]` 直接会签

### 阶段三 · 阻塞升级

命中以下情况时生成 `B-x`：

- DoD 不可执行
- 证据不可获取
- 跨任务依赖未就绪
- owner_role 长时间未提交

升级路径：`→ manager`（仅裁决，不中转执行）。

## 结果交付要素（owner_role 必填）

| 要素 | 说明 |
|------|------|
| 任务编号 | `T-n` |
| 执行人 | 角色名（必须与 `owner_role` 一致） |
| 执行时间 | `YYYY-MM-DD` |
| 结果摘要 | 变更/实现说明 |
| DoD 对照 | 每条 DoD 的完成情况 |
| 证据列表 | 路径/命令/输出/截图定位 |
| 复现步骤 | 最小可复现步骤 |
| 偏差说明 | 与 DoD 差异及补救计划（如有） |

## 执行提交模板（可复制）

```text
我提交 T-<n> 的执行结果（owner 提交）

- 任务编号：T-<n>
- 执行人：<role>（必须与 owner_role 一致）
- 执行时间：YYYY-MM-DD
- 结果摘要：<变更/实现说明>
- DoD 对照：
  1) <DoD-1>：<完成情况>
  2) <DoD-2>：<完成情况>
- 证据列表：
  - E1：路径 <...> / 命令 <...> / 输出定位 <...>
  - E2：路径 <...> / 命令 <...> / 输出定位 <...>
- 复现步骤：
  1) <step-1>
  2) <step-2>
  3) 期望结果：<...>
- 偏差说明：<无 / 差异与补救计划>
```

## 守门前自检清单（owner 自测）

提交前必须自检以下 5 项（与 DoD 守门一一对应）：

1. **证据齐**：每条 DoD 都有对应证据条目
2. **证据可寻址**：每条证据都有路径/命令/输出定位
3. **证据可复现**：关键证据有最小复现步骤
4. **证据时效**：证据时间晚于任务接收时间
5. **范围对齐**：执行内容覆盖任务摘要与 DoD，无越界无遗漏

## 输出格式（默认）

```text
## 【execute 输入】
## 【守门检查】（5 项）
## §当前在跑 更新（执行结果列、状态）
## 【execute 结论】
## 阻塞单（若有）
```

## 注意事项

- 本技能是**公共提交技能**，不是 manager 调度技能
- 不跨列写：仅允许写 `执行结果` 与 `状态`
- 不跨任务：单次仅处理一个 `T-n`
- 责任归属不变：调用方不能替换 `owner_role`
- 质检打回到 `待执行` 时，按打回原因重新执行并再次走守门

> 通用注意事项见 [`manager.md`](../../../manager.md) §注意事项。
