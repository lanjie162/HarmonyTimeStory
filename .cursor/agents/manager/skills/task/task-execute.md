# Skill: task-execute（task 包 · 原子）

> **Skill 元信息**
>
> - **属于**：[`manager`](../../../manager.md)
> - **层级**：task 包 · **原子层**
> - **目的**：把统一任务列表 §当前在跑 中状态为 `待执行` 的任务推进为 `已执行`：
>   1. **主动模式**：调用 `owner_role` 子代理执行任务，回填执行结果
>   2. **被动模式**：接收 `owner_role` 自行执行后提交的结果
>   3. 按 DoD 完备度守门，通过则扭转状态 `待执行 → 已执行`
> - **入口**：
>   - 用户字面：「执行 T-12」「让 dev 跑 T-13」「我已经做完 T-14，结果在……」
>   - §当前在跑 中状态=`待执行` 的任意任务
> - **专属编号**：不分配新编号；引用 `T-n` / `B-x`
> - **来源选型触发**：**每条任务开始**（阶段一步骤 1）。规则按 [`manager.md`](../../../manager.md) §来源选型 §节奏 §暂停语总集 执行
> - **主入参**：单条 `T-n`（状态=`待执行`）+ 入口模式（主动 / 被动）+（被动模式下）执行结果材料
> - **主出参**：§当前在跑 该 T-n 的 `执行结果` 列填充 + 状态 `待执行 → 已执行`

## 信息边界（强制）

> 引用 [`manager.md`](../../../manager.md) §任务请求契约 → 信息边界。本 skill 不向上游产物写入 `T-<数字>`；上游 `task_request` 字段维护由其他原子 skill 负责。

## 执行口径

| 模式 | 触发 | 执行主体 | 本 skill 职责 |
|------|------|---------|--------------|
| **A · 主动调用** | 编排器 / 用户字面指定 | 本 skill 通过 Task 委派给 `owner_role` 子代理 | 写需求 → 等回执 → 守门 → 扭转 |
| **B · 被动接收** | `owner_role` 自行交付结果 | `owner_role` 自助 | 校验材料 → 守门 → 扭转 |

## DoD 完备度守门（核心）

`待执行 → 已执行` **必经**以下检查（全过才扭转）：

1. **证据齐**：执行结果包含 DoD 字段中要求的全部证据条目
2. **证据可寻址**：每条证据有具体路径 / 链接 / 文件名 / 命令输出（拒绝「已完成」「OK」等空泛描述）
3. **证据可复现**：关键证据附操作命令或步骤（如 `hdc shell …` 命令、构建命令、测试命令）
4. **证据时效**：证据生成时间晚于本任务接收时间
5. **范围对齐**：执行结果覆盖任务摘要 + DoD 描述的全部范围（不超范围、不遗漏）

任一项不通过 → 状态保持 `待执行`，输出 `【守门未通过】T-<n>：<缺啥项>，请 owner_role=<…> 补齐后重新提交。`

## 核心工作流

### 阶段一 · 任务读取与模式判定

1. **【条目 Tn · 来源】**（必达，用户暂停点）
   - 触发节点：每条任务进入本 skill 的第一步
   - 选项与暂停语：使用 [`manager.md`](../../../manager.md) §暂停语总集 第 1 行（`【来源】` 模板，前缀写为 `【条目 Tn · 来源】`）
2. 读 §当前在跑 该 `T-n` 行：
   - 状态必须 = `待执行`；否则提示「不在执行环节，跳过」并退出
   - 读取 `任务摘要` / `owner_role` / `acceptor_role[]` / `DoD` / `来源任务请求反链`
3. 模式判定：
   - 用户字面包含「让 / 派 / 调 owner」或编排器主动调度 → **A 主动模式**
   - 用户已附执行结果 / `owner_role` 自助提交 → **B 被动模式**
4. 公示：

   ```text
   【execute 模式】<A 主动调用｜B 被动接收>；T-<n>；owner_role=<…>；DoD 概要=<…>。
   ```

### 阶段二 · 执行（按所选模式）

#### 模式 A · 主动调用 owner_role

1. 公示「【调用 owner】T-<n> → <owner_role>」
2. 按下方 §Task 委派模板 切到 `owner_role` 子代理执行
3. 等回执：
   - **超时 / 拒绝 / 角色拒绝**：转 §阻塞单（类型：`角色不响应`），状态保持 `待执行`，本任务退出
   - **回执有结果**：进入阶段三守门
4. 接收 `owner_role` 的结构化回执（要素见 §结果交付要素）

#### 模式 B · 被动接收

1. 公示「【接收结果】T-<n> ← <owner_role 自助>」
2. 校验提交人是否在 `owner_role` 范围内（不在则提示并退出，不扭转状态）
3. 接收材料 → 进入阶段三守门

### 阶段三 · DoD 完备度守门 + 状态扭转

1. 按 §DoD 完备度守门 5 项逐条检查（必填 + 可寻址 + 可复现 + 时效 + 范围）
2. **任一不通过**：
   - 输出 `【守门未通过】T-<n>：<缺啥项 1>、<缺啥项 2>，请 owner_role=<…> 补齐：<具体期望>。`
   - 状态保持 `待执行`，本任务退出，等下次执行
3. **全通过**：
   - 写 §当前在跑 该 T-n 的 `执行结果` 列：`已执行 by <owner_role> @ <YYYY-MM-DD>; 证据：<路径 1>, <路径 2>; 备注：<…>`
   - 状态扭转 `待执行 → 已执行`
   - 输出 `【已执行】T-<n>，验收人=<acceptor_role[]>，下一环：验收。`

### 阶段四 · 收口

1. 本轮处理摘要：

   ```text
   ## 【execute 摘要】
   - 通过守门并扭转：N1 条
   - 守门未通过（保持待执行）：N2 条
   - 阻塞升级：N3 条
   ```

2. 阻塞 / 守门未通过 → 升级 / 留印 后退出。

## §结果交付要素（owner_role 视角）

`owner_role` 通过本 skill（主动 / 被动）交付结果时，须包含：

| 要素 | 说明 |
|------|------|
| `任务编号` | `T-n` |
| `执行人` | 子代理名（如 `dev` / `sa`） |
| `执行时间` | `YYYY-MM-DD` |
| `结果摘要` | 一段话说明做了什么 |
| `证据列表` | 对应 DoD 每条标准的证据：路径 / 命令 / 输出截图链接 |
| `操作步骤`（可复现项必填） | 关键命令 / 操作流程 |
| `偏差说明`（如有） | 与 DoD 的差异、原因、补救计划 |

## Task 委派模板（主动模式专用）

单条提示词模板要素：**任务编号（T-n）** ｜ **任务摘要 + DoD** ｜ **`subagent_type` = owner_role**（即 `dev` / `sa` / `pm` / `ued` / `qa`） ｜ **结果交付要素清单** ｜ **回填路径**（统一任务列表 §当前在跑 该 T-n 行 `执行结果` 列）。

```text
【委派方式】Task（任务：T-<n>）；subagent_type=<owner_role>；要求：按 §结果交付要素 提交结果，明确 DoD 对应的证据。
```

向 Task 写明：**「仅服务本任务 T-<n>；执行结果按结构化要素提交；不顺带改其他任务的执行结果列；不跨界写验收 / 质检 / 通晒列」**。

## 阻塞单格式

```text
## 阻塞单 B-<序号>
- 关联条目：T-<n>
- 阻塞类型：角色不响应 / 角色拒绝 / DoD 不可执行 / 证据无法获取 / 跨任务依赖未就绪
- 触发条件：…
- 已尝试：…
- 升级路径：→ manager（由 manager 决定升级到上游 skill / 调整任务 / 调整 DoD）
- 截止预期：…
```

阻塞单同步追加到统一任务列表 §阻塞单。

## 输出格式（默认）

```text
## 【execute 模式】公示
## 【调用 owner / 接收结果】公示
## §结果交付要素 摘要
## 【守门检查】结果（5 项）
## §当前在跑 T-n 行更新（执行结果列、状态）
## 【execute 摘要】
## 阻塞单（若有）
```

## 注意事项（task-execute 专属）

- **守门是核心职责**：宁可保持 `待执行` 状态等补齐，不可放水扭转 `已执行`
- **不跨列写**：仅写 §当前在跑 该 T-n 的 `执行结果` 列与 `状态` 列；不写 `验收结论 / 质检结论 / 通晒回填` 列（由其他原子 skill 写）
- **不跨任务**：单次执行只服务本 T-n；不顺带处理其他任务
- **角色守门**：`owner_role` 必须与 §当前在跑 登记一致；提交人不符则拒绝
- **信息边界**：本 skill 不修改任务请求文件 / 上游产物 / `task_request` 字段（由其他原子 skill 负责）
- **质检打回处理**：若 §当前在跑 出现 `状态=待执行 + 质检结论=打回执行（XXX）`，本 skill 按打回原因重新驱动（owner_role 优先针对原因补齐），守门通过后再次扭转 `已执行`
