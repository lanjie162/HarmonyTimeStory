# Skill: task-quality（task 包 · 原子）

> **Skill 元信息**
>
> - **属于**：[`manager`](../../../manager.md)
> - **层级**：task 包 · **原子层**
> - **目的**：把统一任务列表 §当前在跑 中状态为 `已验收` 的任务推进为 `已完成`：
>   1. 按 DoD 思想，对**执行结果**与**验收结论**做二次逻辑论证（与前序执行/验收环节的「材料完备度」不同，本 skill 做「逻辑正确性 + 一致性」）
>   2. 通过 → 扭转 `已验收 → 已完成`
>   3. 不通过 → 按问题定位打回：执行不合格 → `已验收 → 待执行`；验收不合格 → `已验收 → 已执行`
> - **入口**：
>   - 用户字面：「质检 T-12」「让 sa 质检 T-13」「检查 T-14 的执行 / 验收质量」
>   - §当前在跑 中状态=`已验收` 的任意任务
> - **专属编号**：不分配新编号；引用 `T-n` / `B-x`
> - **来源选型触发**：**每条任务开始**（阶段一步骤 1）。规则按 [`manager.md`](../../../manager.md) §来源选型 §节奏 §暂停语总集 执行
> - **主入参**：单条 `T-n`（状态=`已验收`）
> - **主出参**：§当前在跑 该 T-n 的 `质检结论` 列填充 + 状态扭转（`已验收 → 已完成` / `→ 待执行` / `→ 已执行`）

## 信息边界（强制）

> 引用 [`manager.md`](../../../manager.md) §任务请求契约 → 信息边界。本 skill 不向上游产物写入 `T-<数字>`。

## 与前序环节的差异（核心）

| 环节 | 守门视角 | 关注点 |
|------|---------|-------|
| 前序执行环节 | **材料完备度**（执行视角） | 证据齐 / 可寻址 / 可复现 / 时效 / 范围 |
| 前序验收环节 | **材料完备度**（验收视角） | 结论明确 / 依据可寻址 / 不通过可定位 / 条件可执行 / 范围 |
| 本质检环节 | **逻辑正确性 + 一致性** | DoD 推导链是否成立 / 证据是否真支持结论 / 验收是否真覆盖 DoD / 执行结果与验收结论是否对齐 |

通俗说：execute / accept 检查「东西交了没」；quality 检查「东西对不对」。

## 二次质检矩阵

对每条 T-n 按以下 6 项做逻辑论证（所有项都过 → `已完成`）：

| # | 检查项 | 检查内容 | 不过 → 归类 |
|---|--------|---------|------------|
| Q1 | **DoD 自洽** | DoD 字段本身是否可推出「完成」状态（不是空泛、不内部矛盾） | 上游问题 → 升级 manager（不打回到 execute / accept） |
| Q2 | **执行证据 → DoD** | 执行结果列出的证据，是否真能支撑 DoD 中的每一条标准 | **执行不合格** → 打回 `待执行` |
| Q3 | **验收依据 → 执行证据** | 验收结论引用的依据，是否真对应执行结果中的具体证据（不是想当然） | **验收不合格** → 打回 `已执行` |
| Q4 | **验收结论 → DoD 全覆盖** | `acceptor_role[]` 的全部结论加总，是否覆盖 DoD 全部标准（无遗漏） | **验收不合格** → 打回 `已执行` |
| Q5 | **执行 - 验收一致性** | 执行结果声称「已做」的事项，验收结论是否承认（无矛盾） | 看冲突方向：执行虚报 → `待执行`；验收误判 → `已执行` |
| Q6 | **范围对齐** | 执行结果与验收结论描述的范围，是否与任务摘要 + DoD 一致（无超范围、无遗漏） | 看冲突方向（同 Q5） |

## 打回口径表

| 不过项主要归类 | 打回到的状态 | 写入 `质检结论` 列示例 |
|---------------|-------------|---------------------|
| **执行不合格**（Q2 / Q5 执行虚报 / Q6 执行超范围或遗漏） | `已验收 → 待执行` | `打回执行 by quality @ <date>: <Q编号-原因>; 期望补齐：<…>` |
| **验收不合格**（Q3 / Q4 / Q5 验收误判 / Q6 验收超范围或遗漏） | `已验收 → 已执行` | `打回验收 by quality @ <date>: <Q编号-原因>; 期望补齐：<…>` |
| **执行 + 验收都不合格**（同时命中两类） | `已验收 → 待执行`（先解执行问题） | `双打回 by quality @ <date>: 优先解执行问题 <…>，再做验收 <…>` |
| **DoD 上游问题**（Q1） | 状态保持 `已验收` + 阻塞升级 | `阻塞 by quality @ <date>: DoD 自洽问题 <…>，升级 manager` |

## 核心工作流

### 阶段一 · 任务读取与质检主体确定

1. **【条目 Tn · 来源】**（必达，用户暂停点）
   - 触发节点：每条任务进入本 skill 的第一步
   - 选项与暂停语：使用 [`manager.md`](../../../manager.md) §暂停语总集 第 1 行（`【来源】` 模板，前缀写为 `【条目 Tn · 来源】`）
2. 读 §当前在跑 该 `T-n` 行：
   - 状态必须 = `已验收`；否则提示「不在质检环节，跳过」并退出
   - 读取 `任务摘要` / `DoD` / `执行结果` / `验收结论`
3. **质检主体选择**（用户暂停点）：
   - 默认：`sa`（适合架构 / 门禁 / 一致性类任务）
   - 可选：`pm`（适合需求范围 / 用户价值类任务）/ 其他角色
   - 公示：

     ```text
     【quality 主体】T-<n>；建议主体=<sa|pm|其他>；理由：<任务性质>。
     ```
   - 用户可指定其他角色或要求本 skill 自检（自检 = 由 manager 上下文直接走质检矩阵，不切角色子代理）

### 阶段二 · 二次质检（按 §二次质检矩阵 6 项）

1. 公示「【质检中】T-<n>，主体=<…>，按 6 项质检矩阵展开。」
2. 若主体 ≠ 自检 → 按 §Task 委派模板 切到主体子代理执行 6 项检查；等回执
3. 若主体 = 自检 → manager 上下文直接对照 6 项做论证
4. 接收质检材料：

   | 要素 | 说明 |
   |------|------|
   | `任务编号` | `T-n` |
   | `质检主体` | 子代理名 / `manager 自检` |
   | `质检时间` | `YYYY-MM-DD` |
   | `Q1 ~ Q6 逐项判定` | `通过` / `不通过` + 一段话理由 + 关键引用（执行结果证据 / 验收结论依据） |
   | `综合归类` | `通过` / `执行不合格` / `验收不合格` / `双不合格` / `DoD 上游问题` |
   | `打回期望` | 不通过时填 |

### 阶段三 · 写入质检结论 + 状态扭转

1. 按 §打回口径表 决定状态扭转方向：

   | 综合归类 | 状态扭转 |
   |---------|---------|
   | `通过` | `已验收 → 已完成` |
   | `执行不合格` | `已验收 → 待执行` |
   | `验收不合格` | `已验收 → 已执行` |
   | `双不合格` | `已验收 → 待执行`（优先解执行） |
   | `DoD 上游问题` | 状态保持 `已验收`，进入阻塞升级 |

2. 写 §当前在跑 该 T-n 的 `质检结论` 列（按 §打回口径表 示例格式）
3. 输出 `【质检完成】T-<n>，结论=<…>，状态：已验收 → <…>。`

### 阶段四 · 收口

1. 本轮处理摘要：

   ```text
   ## 【quality 摘要】
   - 通过（→ 已完成）：N1 条
   - 打回执行（→ 待执行）：N2 条
   - 打回验收（→ 已执行）：N3 条
   - 双打回（→ 待执行）：N4 条
   - DoD 上游问题（→ 阻塞）：N5 条
   ```

2. 阻塞 → 升级 manager 后退出。

## Task 委派模板（主体 ≠ 自检 时）

单条提示词模板要素：**任务编号（T-n）** ｜ **任务摘要 + DoD + 执行结果链接 + 验收结论链接** ｜ **`subagent_type` = 质检主体**（如 `sa` / `pm`） ｜ **§二次质检矩阵 6 项** ｜ **回填路径**（统一任务列表 §当前在跑 该 T-n 行 `质检结论` 列）。

```text
【委派方式】Task（任务：T-<n>，质检）；subagent_type=<架构师 / pm / 其他>；要求：按 §二次质检矩阵 6 项逐条判定，给出综合归类。
```

向 Task 写明：**「仅做逻辑论证 + 一致性检查；不做材料完备度检查（已由前置环节做过）；不顺带改其他任务；不跨界写执行 / 验收 / 通晒列」**。

## 阻塞单格式

```text
## 阻塞单 B-<序号>
- 关联条目：T-<n>
- 阻塞类型：DoD 自洽问题 / 主体角色不响应 / 双打回反复 / 质检主体不匹配
- 触发条件：…
- 已尝试：…
- 升级路径：→ manager（由 manager 决定调整 DoD / 调整任务 / 升级上游 skill）
- 截止预期：…
```

阻塞单同步追加到统一任务列表 §阻塞单。

## 输出格式（默认）

```text
## 【quality 主体】公示
## §二次质检矩阵 6 项判定
## 综合归类
## §当前在跑 T-n 行更新（质检结论列、状态扭转方向）
## 【quality 摘要】
## 阻塞单（若有）
```

## 注意事项（task-quality 专属）

- **逻辑视角而非材料视角**：本 skill 不重复做材料完备度检查（execute / accept 已守门）；只做「材料是不是真支撑结论」的逻辑论证
- **打回方向严格**：按 §打回口径表 严格区分执行不合格（`→ 待执行`）/ 验收不合格（`→ 已执行`），不混用
- **打回写入原因**：`质检结论` 列必须包含 `Q编号-原因` + `期望补齐` 两段，供下游 skill 复盘
- **不跨列写**：仅写 §当前在跑 该 T-n 的 `质检结论` 列与 `状态` 列；不写 `执行结果 / 验收结论 / 通晒回填` 列
- **不跨任务**：单次质检只服务本 T-n
- **DoD 上游问题不打回**：Q1 DoD 自洽问题不能打回到执行 / 验收（他们没法解），必须升级 manager 修 DoD（可能需要回到上游 skill）
- **信息边界**：本 skill 不修改任务请求文件 / 上游产物 / `task_request` 字段

> 通用注意事项（保持中立 / 灵活适配 / 上下文管理 / 引用规范 / 完整性兜底）见 manager.md §注意事项。

风格：中文；结论先行；表格优先。

**状态（manager 自用）**：`[主体选择 / 质检中 / 状态扭转 / 阻塞升级]`
