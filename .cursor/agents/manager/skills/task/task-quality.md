# Skill: task-quality（task 包 · 原子）

> **Skill 元信息**
>
> - **属于**：[`manager`](../../../manager.md)
> - **层级**：task 包 · **原子层**
> - **执行主体（固定）**：**仅 `manager`（本项目中的协调者子代理与同会话上下文）**。**禁止** 将 `task-quality` 再行委派给 `sa` / `pm` / `dev` / `qa` / `ued` 或使用 Task 充当「第二质检员」。
> - **目的**：把统一任务列表 §当前在跑 中状态为 `已验收` 的任务推进为 `已完成`，由 **manager 做一次收口式二次确认**：
>   1. **材料相对本任务可闭环**：执行结果列中的路径/表述能否支撑其对 DoD 的宣称；验收结论是否有据、是否覆盖 DoD —— **不代替** execute/accept 逐项重守门，但若发现 **对结论不可或明显缺口**，按矩阵归入打回项。
>   2. **一致性**：执行声称 vs 验收承认 vs DoD / 摘要是否对齐。
>   3. **逻辑性**：Q1～Q6 推导链是否成立。
> - **通过后**：扭转 `已验收 → 已完成`。**不通过**：按 §打回口径表（执行不合格 → `待执行`；验收不合格 → `已执行`；DoD 上游问题 → 阻塞）。
> - **入口**：
>   - 用户字面：「质检 T-n」「复检 T-n」「二次确认 T-n」「检查 T-n 的执行与验收闭环」
>   - §当前在跑 状态=`已验收`
> - **硬性前置**：**每条** `T-n` 展开论证前须完成 **【task-quality · manager 装载】**（§阶段一 · 步骤 0）；[`task-drive`](task-drive.md) 串流时与同回合 **`【task-drive · 装载】`**（quality 仅需 **一行** `manager.md`）一致。
> - **专属编号**：不新增；引用 `T-n` / `B-x`
> - **来源选型触发**：**每条任务开始**（阶段一步骤 **1**，且须在 **步骤 0** 完成后）。规则按 [`manager.md`](../../../manager.md) §来源选型 §节奏 §暂停语总集。
> - **主入参**：单条 `T-n`（状态=`已验收`）
> - **主出参**：§当前在跑 **`质检结论` + `状态`**（不写其它列）

## 信息边界（强制）

> 引用 [`manager.md`](../../../manager.md) §任务请求契约 → 信息边界。本 skill 不向上游产物写入 `T-<数字>`。

## 与前序环节的差异（核心）

| 环节 | 守门主体 | 关注点 |
|------|----------|--------|
| execute / accept | owner / acceptor | **按角色**做单层材料与结论守门 |
| **task-quality** | **仅 manager** | **中位二次收口**：对上述结果做 **相对本任务的闭环 + 一致性 + 逻辑**论证（Q1～Q6），**不追加** second-line 专业子代理复检 |

## 二次质检矩阵（Q1～Q6）

对每条 T-n 逐项判定（六项均 **通过** → 综合归类 **`通过`** → `已完成`）：

| # | 检查项 | 检查内容 | 不过 → 归类 |
|---|--------|---------|------------|
| Q1 | **DoD 自洽** | DoD 字段本身是否可推出「可验收/可完成」语义（非空泛、不自相矛盾） | **DoD 上游问题** → 保持 `已验收` + **阻塞**（登记 `B-x`；**不**打回执行/验收由其直接修 DoD 定义） |
| Q2 | **执行证据 → DoD** | 执行结果所列证据与路径，能否在逻辑上支撑 DoD 每一条；若 DoD 或权威基线要求 **L2** 级验证，执行证据不得仅为 **L1** 演示材料，除非 DoD 显式声明「仅 L1 演示通过」且与 **`CR-x`** / 计划 **Out** 一致 | **执行不合格** → `待执行` |
| Q3 | **验收依据 → 执行证据** | 验收结论所称依据是否真能对应到执行结果中的具体材料 | **验收不合格** → `已执行` |
| Q4 | **验收 → DoD 全覆盖** | `acceptor_role[]` 的结论合起来是否逻辑上覆盖 DoD 全条目 | **验收不合格** → `已执行` |
| Q5 | **执行—验收一致性** | 执行「已做」与验收「承认」是否矛盾 | 依方向：执行夸大 → `待执行`；验收误判 → `已执行` |
| Q6 | **范围对齐** | 执行与验收所述范围是否与 **任务摘要 + DoD** 一致 | 同上 |

> **专条与高约束域**：见 [`.cursor/agents/GOVERNANCE-权威基线与可验证一致性.md`](../../../GOVERNANCE-权威基线与可验证一致性.md)；**声称与证据不同级** 归入 **Q2 执行不合格**。

> **材料视角的落点**：显性路径缺失、表中应填而空、或与 execute/accept **已声称的守门结论明显冲突** → 分别按 Q2～Q5 归类，**不**单列「再跑一遍 execute 五项表」。

## 打回口径表

| 不过项主要归类 | 打回到的状态 | 写入 `质检结论` 列示例 |
|---------------|-------------|---------------------|
| **执行不合格**（Q2 / Q5 执行虚夸 / Q6 执行侧偏题） | `已验收 → 待执行` | `打回执行 by quality(manager) @ <date>: <Q-原因>; 期望补齐：<…>` |
| **验收不合格**（Q3 / Q4 / Q5 验收误判 / Q6 验收侧偏题） | `已验收 → 已执行` | `打回验收 by quality(manager) @ <date>: <Q-原因>; 期望补齐：<…>` |
| **双不合格** | `已验收 → 待执行`（先执行） | `双打回 by quality(manager) @ <date>: …` |
| **DoD 上游问题**（Q1） | 保持 `已验收` + 阻塞 | `阻塞 by quality(manager) @ <date>: DoD 自洽 <…>; 升级待决` |

## 核心工作流

### 阶段一 · 装载、来源与任务读取

0. **`manager` 装载（强制 · 实为读档）**  
   - 在本条 **任何** Q1～Q6 论证或改表 **之前**，须 **Read** [`.cursor/agents/manager.md`](../../../manager.md) 全文，并**单行公示**：  
     `【task-quality · manager 装载】已读 .cursor/agents/manager.md（中立位 / task 契约 / skill 调度已装载）。`  
   - **禁止**未读即写矩阵结论；禁止再装载第二份子代理 `.md` 作为「质检员」。

1. **【条目 Tn · 来源】**（必达）  
   - 使用 [`manager.md`](../../../manager.md) §暂停语总集：**`【来源】`**，前缀 **`【条目 Tn · 来源】`**。

2. **读 §当前在跑 `T-n`**  
   - 状态须 = `已验收`；否则退出。  
   - 读取：`任务摘要`、`DoD`、`执行结果`、`验收结论`。

3. **固定公示（无选项）**  

   ```text
   【task-quality · 执行主体】T-<n>；质检执行者=fixed(manager)；不进行 sa/pm/qa/dev/ued 分包或 Task 复检。
   ```

### 阶段二 · manager 中位质检（Q1～Q6）

1. 公示：`【质检中】T-<n>；执行者=manager；按 §二次质检矩阵逐项展开。`

2. 以 **`【manager · 质检论证】`** 为标题，按 Q1～Q6 表格输出每项 **通过/不通过**、一句理由、关键引用（执行结果 / 验收结论中的路径或原句）。

3. 产出 **综合归类**，并映射 §打回口径表。

### 阶段三 · 写入质检结论 + 状态扭转

1. 按综合归类扭转 `状态`（见矩阵上表）。
2. 写 `质检结论` 列（含 `Q编号`、打回期望若适用）。
3. 输出：`【质检完成】T-<n>，综合=<…>，状态：已验收 → <…>。`

### 阶段四 · 收口

```text
## 【quality 摘要】
- 通过（→ 已完成）：N1
- 打回执行 / 验收 / 双打回 / DoD 阻塞：…
```

需阻塞时同步 §阻塞单模板。

## 阻塞单格式

```text
## 阻塞单 B-<序号>
- 关联条目：T-<n>
- 阻塞类型：DoD 自洽（Q1）/ 双打回反复 / 其它
- 触发条件：…
- 升级路径：→ 用户裁决或上游计划/任务修订（仍由 manager 路由，不转嫁子代理质检）
```

## 输出格式（默认）

```text
【task-quality · manager 装载】…
【条目 Tn · 来源】…
【task-quality · 执行主体】fixed(manager)
【质检中】…
【manager · 质检论证】（Q1～Q6）
综合归类 + §当前在跑 更新
【质检完成】…
```

## 注意事项（task-quality 专属）

- **单角色**：全链路 **只允许 manager** 读本 skill 并完成论证；与用户沟通「希望 sa 把一眼」之类 **不改变**本条——可作 **下一轮 review-meeting / 单列征询**，**不属于** task-quality。
- **与 execute/accept 分工**：不重走 owner/acceptor 的逐项材料表；仅从 **闭环 + 一致性 + 逻辑** 二次确认。
- **打回用语**：质检结论署名 **`quality(manager)`** 或与 `质检结论` 列内文一致即可。
- **不跨界写列**：只动 `质检结论`、`状态`。

> 通用注意事项见 [`manager.md`](../../../manager.md) §注意事项。
