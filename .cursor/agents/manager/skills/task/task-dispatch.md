# Skill: task-dispatch（task 包 · 原子）

> **Skill 元信息**
>
> - **属于**：[`manager`](../../../manager.md)
> - **层级**：task 包 · **原子层**
> - **目的**：把上游产物（评审纪要 / 计划 / 任意命题）或已生成的任务请求文件，转写为统一任务列表中的可执行任务明细：
>   1. **打包模式**：从上游产物抽取候选 → 视角转换 → 生成任务请求文件 → 反向回填上游 `task_request` 字段
>   2. **接收模式**：读取已存在的任务请求文件 → 要素补齐 → 写入 §当前在跑（状态=`待执行`）+ 回填任务请求 §接收记录
> - **入口**：
>   - 用户字面：「打包 1,3 / 打包候选 N 为任务请求」「派发 TR-… / 接收任务请求 / 入站」「按本纪要发任务请求」
>   - `document/task/requests/<source_type>/` 下未接收（`status: requested`）的任务请求文件
> - **专属编号**：分配 `T-n`（统一任务列表全局自增）/ `TR-<YYYYMMDD>-<seq>`（任务请求 ID）/ `C-<n>`（任务明细在请求文件内编号）
> - **来源选型触发**：**每个任务请求开始**（阶段二步骤 1）。规则按 [`manager.md`](../../../manager.md) §来源选型 §节奏 §暂停语总集 执行
> - **主入参**：上游产物（评审纪要 / 计划）/ 用户开放命题 / 已生成的任务请求文件
> - **主出参**：任务请求文件（路径 / frontmatter / §派发概要 / §任务明细 / §接收记录）+ 统一任务列表 §当前在跑 新行 + 上游 `task_request` 字段回填

## 信息边界（强制）

> 任务请求文件 §接收记录 与统一任务列表 §当前在跑 是 **`T-n` 与 `C-<n>` 之间的唯一映射点**。
>
> 上游产物 `task_request` 字段仅写 `TR-…#C-<n>(<status>)` 两层信息，**严禁写入 `T-<数字>`**。详见 [`manager.md`](../../../manager.md) §任务请求契约 → 信息边界。

## task 目录与契约（引用 manager.md）

本 skill 依据 [`manager.md`](../../../manager.md) §任务请求契约 / §统一任务列表契约 操作以下物件：

```text
document/task/
├── [任务经理]统一任务列表.md
├── requests/<source_type>/<YYYY-MM-DD>-[<source_id>]-<meaningful_phrase>.md
└── archive/<YYYY-MM>/...
```

子目录由本 skill 在打包时**按需自动创建**；不预创建空目录。

## 文件命名（强制 · 引用 manager.md）

```text
document/task/requests/<source_type>/<YYYY-MM-DD>-[<source_id>]-<meaningful_phrase>.md
```

字段规则（`source_type` / `source_id` / `meaningful_phrase` 强制要求与示例）详见 [`manager.md`](../../../manager.md) §任务请求契约 → 任务请求文件命名。

## 文件结构（强制 · 引用 manager.md）

frontmatter（`request_id` / `source_type` / `source_doc` / `source_anchor` / `created_at` / `created_by` / `status` / `scope` / `item_count`）与正文章节（§派发概要 / §任务明细 / §接收记录）详见 [`manager.md`](../../../manager.md) §任务请求契约 → 任务请求文件结构。

## 任务明细要素（强制扩展）

§任务明细 表头**强制包含**以下列（在 manager.md 基础列上扩展）：

```text
| 明细编号 | 任务摘要 | owner_role | acceptor_role[] | DoD | 复杂度 | 优先级 | 依赖 | 来源回链 |
```

字段约束：

- `明细编号`：`C-<n>`，文件内自增（与 manager.md §任务请求契约 一致）
- `owner_role`：**必填**，唯一责任执行人（取自 manager.md §可用角色映射；可多选时用 `,` 分隔，但建议主责单选）
- `acceptor_role[]`：**必填**，验收人列表（数组形式 `[qa, pm]`，至少 1 人；多人时按列表顺序为主验收 / 复验收）
- `DoD`：**必填**，「完成意味着什么」的可验证标准（一段话或编号清单）；缺失则填 `TBD` 并在文件末尾 `🟡 待补充字段` 段记录
- `复杂度`：`S / M / L / XL`
- `优先级`：`P1 / P2 / P3`
- `依赖`：本文件内的 `C-<n>` 反链（`-` 表示无依赖）
- `来源回链`：上游产物的具体段落锚点

## 核心工作流

本 skill 有两条主路径，按入口方式分流：

- **A · 打包模式**：从上游产物 / 命题 → 写新任务请求文件
- **B · 接收模式**：读已生成的任务请求文件 → 写 §当前在跑

### 阶段一 · 模式判定与公示

1. 读用户字面 / 入口路径，判定模式：
   - 命中「打包候选 N / 按本纪要发任务请求 / 把这条计划做任务请求」→ **A 打包模式**
   - 命中「派发 TR-… / 接收任务请求 / 入站」或入口路径是已存在的 task 请求文件 → **B 接收模式**
2. 公示一行：

   ```text
   【dispatch 模式】<A 打包｜B 接收>；source_type=<review|plan|ad-hoc>；source_id=<…>；目标=<TR-…>。
   ```

3. 编排模式 / 全流程执行方式按 [`manager.md`](../../../manager.md) §节奏 §来源选型 §全流程默认 处理；未公示则在阶段二步骤 1 单独选择。

### 阶段二 · 执行（按所选模式）

#### 模式 A · 打包（manager 路由进入后由本 skill 执行 5 步动作）

> 承接 [`manager.md`](../../../manager.md) §上游产物 → 任务请求 转换协议 §阶段 B：manager 字面捕获后切到本 skill，由本 skill 完成以下 5 步。

1. **公示**：

   ```text
   【打包】source_type=<review|plan|ad-hoc>，source_id=<…>，本次派发候选=<上游编号集>，目标文件=document/task/requests/<source_type>/<YYYY-MM-DD>-[<source_id>]-<meaningful_phrase>.md，request_id=TR-<YYYYMMDD>-<seq>。
   ```

2. **抽取与视角转换**：去除评审 / 计划视角文案，转写为执行视角；DoD 缺失时标 `dod_hint: TBD` 并在请求文末记 `🟡 待补充字段`。

3. **分配明细编号**：本次派发的每条任务明细在该文件内分配 `C-1 / C-2 / …`（自增）；建立「上游编号 → C-编号」映射作为派发轨迹（写入文件 §派发概要 末尾）。

4. **要素补齐子流程**（**本 skill 核心增强**）：
   对每条明细 `C-<n>` 必须按以下顺序补齐字段。任一缺失时优先「主动加载关联材料」其次「向用户最短一问」，不轻易写 `TBD`：
   - **业务含义理解**：读 `source_doc` + `source_anchor` 上下文，明白这条任务到底要解决什么问题
   - **DoD 拆解**：把「完成意味着什么」拆为 1~3 条可验证的具体标准
   - **依赖关系**：识别本文件内 `C-<n>` 之间的前后依赖（建图）
   - **`owner_role` 指派**：结合 manager.md §可用角色映射，决定唯一主责执行人
   - **`acceptor_role[]` 指派**：识别验收人，常见组合：
     - 实现类条目：`acceptor_role[] = [qa]`（单人验收）
     - 范围 / 验收类条目：`acceptor_role[] = [pm, qa]`（双人验收）
     - 设计 / 体验类条目：`acceptor_role[] = [ued, pm]`
     - 架构 / 门禁类条目：`acceptor_role[] = [architect, qa]`
   - **验收标准**：DoD 落到 `acceptor_role` 视角的可执行验收清单（即「验收时要检查的具体证据」）

5. **写入任务请求文件**（按 §文件结构）：frontmatter + §派发概要 + §任务明细（含上述补齐字段）+ §接收记录（空表，待 B 模式回填）。

6. **反向回写上游产物**：每条上游条目的 `task_request` 列填充 `TR-…#C-<n>(requested)`（拆分则写 `C-n,C-m`；合并则多条上游同填一个 `C-n`）。**严禁**写入 `T-n`。

7. **结束公示**：

   ```text
   【打包完成】TR-…，N 条明细已写入 <path>；上游 task_request 字段已更新。下一步可由用户决定是否启动接收（路由 B 模式）或编排驱动。
   ```

#### 模式 B · 接收（写入统一任务列表 §当前在跑）

> 承接 [`manager.md`](../../../manager.md) §上游产物 → 任务请求 转换协议 §阶段 C：用户主动启动接收后由本 skill 执行。

1. **首次启动 · 自动创建统一任务列表**（若 `document/task/[任务经理]统一任务列表.md` 不存在）：

   ```yaml
   ---
   maintained_by: manager · task 包
   last_updated: <YYYY-MM-DD>
   active_count: 0
   archived_count: 0
   ---
   ```

   正文初始化为四个空 section：`## §当前在跑` / `## §待入站` / `## §归档索引` / `## §阻塞单`。

   §当前在跑 表头按 [`manager.md`](../../../manager.md) §统一任务列表契约 与本 skill §任务明细要素 的扩展规则采用：

   ```text
   | T-n | 任务摘要 | owner_role | acceptor_role[] | DoD | 状态 | 来源任务请求反链 | 执行结果 | 验收结论 | 质检结论 | 通晒回填 | 阻塞标记 |
   ```

2. **入站登记**：把任务请求文件登记到 §待入站（指向 TR-… 文件，附 `item_count` + frontmatter `status`）。

3. **逐条 `C-<n>` 与用户确认决策**（用户暂停点）：
   - **accepted**：进入 §当前在跑
   - **rejected**：仅在 §接收记录 留印 `rejected`
   - **拆分为多条**：原 `C-<n>` 接收后再分配 `T-n1, T-n2`，备注「拆分自 `C-<n>`」

4. **分配 `T-n`** 并写入 §当前在跑：每条 accepted 明细分配统一任务列表全局 `T-n`（自增），状态置 `待执行`，把 §任务明细 表的字段（owner_role / acceptor_role[] / DoD / 来源回链）映射到 §当前在跑 对应列；执行结果 / 验收结论 / 质检结论 / 通晒回填 / 阻塞标记 列**留空**（由后续原子 skill 写入）。

5. **回填任务请求 §接收记录**：

   ```text
   | 明细编号 | 决策 | 统一任务列表编号 | accepted_at |
   |----------|------|------------------|-------------|
   | C-1      | accepted | T-12         | 2026-05-08  |
   ```

6. **更新任务请求 frontmatter `status`**：全 accepted → `accepted`；部分 → `partially_accepted`；全 rejected → `rejected`。

7. **同步上游 `task_request` 字段**：`#C-<n>(requested) → #C-<n>(accepted)`（保留 `C-n`，**严禁**写入 `T-n`）。

8. **结束公示**：

   ```text
   【接收完成】TR-…，N1 条已 accepted（T-x1 ~ T-xN1），N2 条 rejected。统一任务列表 §当前在跑 已更新。
   ```

### 阶段三 · 完备度自检（本 skill 守门标准）

任务明细写入 §当前在跑 之前**必须**通过以下检查（本 skill 核心守门职责）：

1. **要素齐**：`owner_role` / `acceptor_role[]` / `DoD` / `来源回链` 四项**必填**；任一缺失则退回阶段二步骤 4 补齐
2. **依赖闭包**：依赖的 `C-<m>` 必须在本文件存在；跨文件依赖须显式标注「外部依赖：TR-…#C-<m>」
3. **验收人合规**：`acceptor_role[]` 中每个角色必须存在对应 `.cursor/agents/<role>.md`；不存在则按 manager.md §可用角色映射 处理
4. **DoD 可验证**：`DoD` 字段不可为「实现该功能」「按要求完成」等空泛描述；必须包含至少 1 个可证据化的标准（如「执行 `hdc shell ...` 输出包含 X」「单测覆盖率 ≥ 90%」）

任一不通过 → 当前明细暂留 §待入站，不进入 §当前在跑；输出 `【自检未通过】C-<n>：<缺啥>，请补齐后重试。`

## Task 委派模板

单条提示词模板要素：**任务请求路径** ｜ **目标 source_type / source_id** ｜ **本次派发候选编号集** ｜ **`subagent_type`=manager skill=task-dispatch** ｜ **禁止改其他文件**。

```text
【委派方式】Task（任务请求：<TR-… / 文件路径>，模式：<打包 / 接收>）；回退：按 manager.md §来源选型 失败回退。
```

向 Task 写明：**「仅服务本任务请求；禁止顺带改其他任务请求文件 / 上游产物 / 统一任务列表的非本请求行；输出限于本次派发的明细写入与回填」**。

## 阻塞单格式

```text
## 阻塞单 B-<序号>
- 关联条目：T-<…>（若已分配）/ TR-…#C-<n>（接收前）
- 阻塞类型：上游材料缺失 / 视角转换分歧 / 验收人无法指派 / 依赖闭环 / 用户决策待
- 触发条件：…
- 已尝试：…
- 升级路径：→ manager（由 manager 决定升级到上游 skill 或请用户裁决）
- 截止预期：…
```

阻塞单同步追加到统一任务列表 §阻塞单。

## 输出格式（默认）

```text
## 【dispatch 模式】公示
## 【打包公示】 / 【接收公示】（按模式）
## §任务明细 表（打包模式）
## §接收记录 表（接收模式）
## 上游 task_request 字段回填明细
## 自检结果（通过 / 退回项）
## 阻塞单（若有）
```

## 注意事项（task-dispatch 专属）

- **要素守门**：`owner_role / acceptor_role[] / DoD / 来源回链` 四项缺一不写入 §当前在跑；不轻易标 `TBD`，优先主动加载或最短一问
- **不私改上游产物正文**：除 `task_request` 字段（按本 skill 与 manager.md 信息边界规则）外，不修改评审纪要 / 计划的其他内容
- **打包行为单向写**：本 skill 写入 task 请求文件后**不再修改其内容**（除 §接收记录 在 B 模式下回填、frontmatter `status` 同步外）；正文 §派发概要 / §任务明细 一旦写入即冻结
- **信息边界守门**：上游产物 `task_request` 字段只能保留 `C-<n>` 维度，**严禁**写入 `T-<数字>`
- **不感知后续编排**：本 skill 写完 §当前在跑 后即结束；任务的执行 / 验收 / 质检 / 通晒由 task 包其他原子 skill 各自承担，本 skill 不调用、不引用它们的 skill 名

> 通用注意事项（保持中立 / 灵活适配 / 上下文管理 / 引用规范 / 完整性兜底）见 manager.md §注意事项。

风格：中文；结论先行；表格优先。
