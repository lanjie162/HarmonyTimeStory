# Skill: task-drive（task 包 · 编排器）

> **Skill 元信息**
>
> - **属于**：[`manager`](../../../manager.md)
> - **层级**：task 包 · **编排层**（5 个原子 skill 之上的指挥官）
> - **目的**：对**一组任务**基于当前状态进行流程编排，串接 4 个被编排原子 skill：[`task-execute`](task-execute.md) / [`task-accept`](task-accept.md) / [`task-quality`](task-quality.md) / [`task-broadcast`](task-broadcast.md)。
> - **入口**：
>   - 上下文指定任务集合（如 `T-12, T-13`、某 `TR-…`、某任务请求文件路径）
>   - 统一任务列表筛选任务集合（`document/task/[任务经理]统一任务列表.md`）
>   - 用户字面：「驱动一组任务 / 推进一批 / 按状态推一批 / 跑完一批」
> - **不在编排范围**：派发入站环节（任务还未到 §当前在跑 时本编排器无对象可驱动）
> - **常见参与角色**：`pm` / `architect` / `dev` / `qa` / `ued`（具体角色由各原子 skill 在内部决定）
> - **专属编号**：本 skill **不定义新编号**；引用 `T-n`（统一任务列表全局）/ `B-x`（阻塞）。详见 [`manager.md`](../../../manager.md) §决策点编号 - 总集
> - **来源选型触发**：**每条目开始**（阶段三步骤 1，每条任务交给原子 skill 前一步）。规则按 [`manager.md`](../../../manager.md) §来源选型 §节奏 §暂停语总集 执行，本文不重述
> - **主入参**：一组任务（上下文指定 / §当前在跑 筛选）
> - **主出参**：编排日志 + 各任务最终状态摘要表 + 阻塞升级清单（不直接扭转任务状态、不写 §当前在跑 业务列）

## 信息边界（强制）

> 统一任务列表的全局编号 **`T-n` 仅在 task 包内部物件可见**：
> 1. 统一任务列表（`document/task/[任务经理]统一任务列表.md`）的 §当前在跑 / §归档索引
> 2. 任务请求文件（`document/task/requests/<source_type>/<file>.md`）的 §接收记录
>
> **严禁**向任何上游产物（评审纪要 / 计划 / 其他子代理交付物 / 用户视角文档）写入 `T-<数字>`；上游 `task_request` 字段只能保留 `TR-…#C-<n>(<status>)` 两层信息。详见 [`manager.md`](../../../manager.md) §任务请求契约 → 信息边界。

## 编排器边界（守住自己的角色）

本 skill **只做调度**，不做：

- ❌ DoD 完备度检查（由对应原子 skill 守门）
- ❌ 业务列写入（`执行结果 / 验收结论 / 质检结论 / 通晒回填` 列由各原子 skill 写）
- ❌ 任务请求文件 / 上游产物的修改（由派发与通晒环节各自负责）
- ❌ 状态扭转的最终落字（由对应原子 skill 在守门通过后落字）
- ❌ 内容性裁决（角色冲突时只做升级，不做折中）

## 编排契约

### 输入解析（两类来源）

| 来源 | 描述 | 示例 |
|------|------|------|
| **A · 上下文指定** | 用户直接给任务集合 | `T-12, T-13, T-14` ｜ `TR-20260507-01` ｜ `document/task/requests/review/2026-05-06-…缓存键标准化.md` ｜「本周未完成的 5 条」 |
| **B · 统一任务列表筛选** | 从 §当前在跑 按条件筛选 | `状态∈{待执行, 已执行, 验收中, 已验收, 已完成}` ｜ `owner_role=dev` ｜ `来源任务请求=TR-…` ｜ `非阻塞` |

固定公示一行：

```text
【任务集】来源：<A 上下文指定｜B §当前在跑筛选>；条件：<…>；命中：T-<x1>, T-<x2>, …（共 N 条）。
```

### 执行模式（两种）

| 模式 | 描述 | 节奏伪代码 |
|------|------|----------|
| **A · 按任务串行** | 一条任务从当前状态一路推到 已通晒（或阻塞），再下一条 | `for task in 集合: while 状态≠已通晒 且 非阻塞: 选原子 skill → 调用 → 检查` |
| **B · 按状态分组串行** | 先把所有「待执行」推进到「已执行」；再把所有「已执行」推到「已验收」… | `for state in [待执行, 已执行/验收中, 已验收, 已完成]: 分组 = 集合.filter(状态=state); for task in 分组: 调对应原子 skill` |

固定公示一行：

```text
【执行模式】<A 按任务串行｜B 按状态分组串行>；说明：<一句模式语义>。
```

### 状态 → 原子 skill 映射表（编排器内部使用）

| 任务当前状态 | 调度的原子 skill | 期望扭转 |
|-------------|------------------|---------|
| `待执行` | [`task-execute`](task-execute.md) | `→ 已执行` |
| `已执行` / `验收中` | [`task-accept`](task-accept.md) | `→ 验收中 → 已验收`（多人）/ `→ 已验收`（单人可直跨） |
| `已验收` | [`task-quality`](task-quality.md) | `→ 已完成`（通过）/ 打回 |
| `已完成` | [`task-broadcast`](task-broadcast.md) | `→ 已通晒` |
| `已通晒` | 终态，跳过 | - |
| `阻塞` / `待裁决` | 升级 manager，本任务挂起 | - |

### 暂停节奏

- **交互式（默认）**：每次原子 skill 完成一项后暂停一回，公示状态扭转结果，等用户「继续」
- **批量模式**（用户显式声明 `批量 / 一次性 / 不要每条停`）：连续推进直到任务集合全部 `已通晒` 或遇阻塞 / 打回；批量产出后逐项展示状态摘要

## 核心工作流

### 阶段一 · 任务集收集与公示

1. 读统一任务列表（不存在则提示并暂停 —— 编排器不自动创建，由入站派发环节在首次接收时完成）
2. 按输入来源解析任务集合：
   - **来源 A**：直接用用户给的编号 / 路径解析为 `T-n` 列表
   - **来源 B**：按筛选条件遍历 §当前在跑 提取
3. 输出 `【任务集】` 公示行（见上）+ 任务集合摘要表：

   ```text
   | T-n | 摘要 | 当前状态 | owner_role | acceptor_role[] | 阻塞标记 |
   |-----|------|---------|-----------|-----------------|---------|
   ```

4. 若集合为空，暂停并请用户调整筛选条件 / 重选来源。

### 阶段二 · 执行模式选择（用户暂停点）

1. 若用户已指定模式，按用户。
2. 否则给出建议：
   - 任务数 ≤ 5 + 各任务状态分散：建议 **模式 A 按任务串行**
   - 任务数 > 5 或状态高度集中：建议 **模式 B 按状态分组串行**
3. 输出 `【执行模式】` 公示行 + 等用户回复 `按推荐执行` / `用 A` / `用 B`。
4. 节奏选择（交互式 / 批量）按 [`manager.md`](../../../manager.md) §节奏 处理。

### 阶段三 · 主循环（按所选模式调度原子 skill）

通用步骤（每次推进一条任务、调用一个原子 skill）：

1. **【条目 Tn · 来源】**（必达，用户暂停点）  
   - 触发节点：每条任务交给原子 skill 之前  
   - 选项与暂停语：使用 [`manager.md`](../../../manager.md) §暂停语总集 第 1 行（`【来源】` 模板，前缀写为 `【条目 Tn · 来源】`）；其余规则按 §来源选型 全文执行

2. **状态读取与映射决策**：从 §当前在跑 读 T-n 当前状态 → 按 §状态 → 原子 skill 映射表 决定本步要切到哪个原子 skill（若状态=`已通晒` 则跳过该任务）

3. **调度公示**：输出一行 `【调度】T-<n> 状态=<…> → 切 <原子 skill 名>；理由：<状态映射>`

4. **切换原子 skill 执行**：按 manager.md §调用方式说明 切到对应原子 skill 文件，由原子 skill 完成本步守门 + 状态扭转 + 业务列写入

5. **回检状态扭转**：原子 skill 执行结束后，回读 §当前在跑 该 T-n 的状态：
   - 状态已扭转到下一档 → 记录 `成功推进`
   - 状态未变（守门未通过 / 退回） → 记录 `未推进 + 缺啥`
   - 状态变为 `阻塞` / `待裁决` → 记录 `阻塞升级` + 本任务挂起，进入 §阻塞升级清单
   - 状态被 `task-quality` 打回（`已验收 → 已执行` 或 `→ 待执行`） → 记录 `打回` + 在模式 A 下继续推进（重新发执行 / 验收）；模式 B 下放回对应分组等下一轮

6. **节奏判断**：交互式 → 暂停等用户「继续」；批量模式 → 直接进入下一项

#### 模式 A · 按任务串行 主循环伪代码

```text
for task in 任务集合:
    while task.状态 ∉ {已通晒, 阻塞, 待裁决}:
        skill = 状态映射表[task.状态]
        若 skill = 跳过 → break
        【条目 Tn · 来源】公示
        【调度】公示
        切到 skill 执行
        回检状态扭转
        若 阻塞 / 待裁决 → 升级清单，break
        交互式 → 暂停等继续
    汇总该任务最终状态
```

#### 模式 B · 按状态分组串行 主循环伪代码

```text
for state in [待执行, 已执行/验收中, 已验收, 已完成]:
    group = 任务集合.filter(状态 = state)
    若 group 为空 → 跳过本组
    【分组推进】公示：状态组=<state>；条目数=<n>；目标原子 skill=<…>
    for task in group:
        【条目 Tn · 来源】公示
        【调度】公示
        切到对应原子 skill 执行
        回检状态扭转
        若 阻塞 → 升级清单
        交互式 → 暂停等继续
    【分组完成】公示
    若所有任务都未推进或全阻塞 → 询问用户是否继续下一组
（一轮完后，被打回回到上轮状态的任务进入下一轮迭代或人工再编排）
```

### 阶段四 · 收口与升级

1. **汇总本轮结果**：

   ```text
   ## 【本轮编排摘要】
   - 推进成功：N1 条
   - 已到终态（已通晒）：N2 条
   - 打回（待重做）：N3 条
   - 阻塞 / 待裁决：N4 条
   - 跳过（终态 / 已编排过）：N5 条
   ```

2. **明细表**：

   | T-n | 起始状态 | 经历 skill | 结束状态 | 结果 | 备注 |

3. **阻塞升级清单**（若有）：把所有 `阻塞` / `待裁决` 任务统一抛给 manager：

   ```text
   ## 【阻塞升级清单】
   - B-<x>: T-<n>（原阻塞详情见 §阻塞单）
   - …
   → 由 manager 决定后续（升级到上游 skill / 请用户裁决 / 调整计划）
   ```

4. **节奏完成**：等用户回复 `结束本轮 / 进入下一轮 / 调整任务集` 等。

## Task 委派模板（编排器视角）

`task-drive` 本身**不向角色子代理发 Task**；具体的角色 Task 委派由各原子 skill 在自己的主循环内完成（按各原子 skill 的「Task 委派模板」）。

`task-drive` 仅在切到原子 skill 时输出 `【调度】` 公示，类似：

```text
【调度】T-12 状态=待执行 → 切 task-execute；详见 .cursor/agents/manager/skills/task/task-execute.md。
```

## 输出格式（默认）

```text
## 【任务集】来源与命中摘要
## 【执行模式】与节奏
## 【编排过程】（按条或按组的调度日志 + 状态扭转回检）
## 【本轮编排摘要】
## 【阻塞升级清单】（若有）
```

## 注意事项（task-drive 编排器专属）

- **守住编排器边界**：不读写 §当前在跑 业务列、不做 DoD 完备度检查、不修改任务请求与上游产物 —— 这些由对应原子 skill 承担
- **单向引用 4 个被编排原子 skill**：execute / accept / quality / broadcast；入站派发不属于本编排层的引用范围
- **被编排原子 skill 反向不感知本 skill**：原子 skill 文件正文严禁出现 `task-drive` 字面，否则违反「单向引用」原则
- **冲突冻结**：碰到打回 / 阻塞 / 多角色结论冲突等情况只做记录与升级，不做内容性折中
- **信息边界守门**：上游产物 `task_request` 字段只能保留 `C-<n>` 维度，**严禁**写入 `T-<数字>`；详见 manager.md §任务请求契约 → 信息边界

> 通用注意事项（保持中立 / 灵活适配 / 上下文管理 / 引用规范 / 完整性兜底）见 manager.md §注意事项。
