# Skill: task-accept（task 包 · 原子）

> **Skill 元信息**
>
> - **属于**：[`manager`](../../../manager.md)
> - **层级**：task 包 · **原子层**
> - **目的**：把统一任务列表 §当前在跑 中状态为 `已执行` / `验收中` 的任务推进为 `已验收`：
>   1. **主动模式**：调用 `acceptor_role` 子代理验收，回填验收结论
>   2. **被动模式**：接收 `acceptor_role` 自行验收后提交的结论
>   3. **多人验收**：第一位提交后即扭转为 `验收中`；全部验收人完成后扭转为 `已验收`
>   4. 按 DoD 完备度守门（验收视角），通过则扭转
> - **入口**：
>   - 用户字面：「验收 T-12」「让 qa 验 T-13」「我已经验完 T-14，结论是……」
>   - §当前在跑 中状态=`已执行` 或 `验收中` 的任意任务
> - **专属编号**：不分配新编号；引用 `T-n` / `B-x`
> - **来源选型触发**：**每条任务开始**（阶段一步骤 1）。规则按 [`manager.md`](../../../manager.md) §来源选型 §节奏 §暂停语总集 执行
> - **主入参**：单条 `T-n`（状态=`已执行` 或 `验收中`）+ 入口模式（主动 / 被动）+（被动模式下）验收材料
> - **主出参**：§当前在跑 该 T-n 的 `验收结论` 列填充 + 状态扭转（`已执行 → 验收中` 或 `验收中 → 已验收`）

## 信息边界（强制）

> 引用 [`manager.md`](../../../manager.md) §任务请求契约 → 信息边界。本 skill 不向上游产物写入 `T-<数字>`。

## 验收口径

| 模式 | 触发 | 验收主体 | 本 skill 职责 |
|------|------|---------|--------------|
| **A · 主动调用** | 编排器 / 用户字面指定 | 本 skill 通过 Task 委派给 `acceptor_role` 子代理 | 写要求 → 等回执 → 守门 → 扭转 |
| **B · 被动接收** | `acceptor_role` 自行交付结论 | `acceptor_role` 自助 | 校验材料 → 守门 → 扭转 |

### 多人验收语义

`acceptor_role[]` 长度 = 1：单人验收，提交后直接扭转 `已执行 → 已验收`

`acceptor_role[]` 长度 ≥ 2：

| 当前状态 | 第 k 位提交后 | 扭转动作 |
|---------|--------------|---------|
| `已执行` | k = 1（第 1 位） | `已执行 → 验收中`，记录第 1 位结论 |
| `验收中` | k < N | 状态保持 `验收中`，记录第 k 位结论 |
| `验收中` | k = N（最后一位） | `验收中 → 已验收`（前提：全部 `通过`） |

任一位结论=`不通过` → 状态保持当前（`已执行` 或 `验收中`），不扭转 `已验收`；输出阻塞建议（升级到 manager 决策是否打回执行）。

## DoD 完备度守门（验收视角）

`已执行/验收中 → 已验收` **必经**以下检查（全过才扭转）：

1. **验收结论明确**：每位 `acceptor_role` 必须给出 `通过` / `不通过` / `条件通过` 三选一（不接受「OK」「看起来行」等模糊表达）
2. **依据可寻址**：每条结论附 DoD 中对应标准的核对依据（哪些证据已对照过）
3. **不通过项可定位**：`不通过` 结论必须指出具体 DoD 标准编号 + 缺啥（供执行环节复盘）
4. **条件通过项可执行**：`条件通过` 结论必须列出条件清单 + 期望补齐时限
5. **范围对齐**：验收结论覆盖 DoD 描述的全部范围（不超范围、不遗漏验收点）

任一项不通过 → 状态保持 `已执行` 或 `验收中`，输出 `【守门未通过】T-<n>：<缺啥项>，请 acceptor_role=<…> 补齐结论后重新提交。`

## 核心工作流

### 阶段一 · 任务读取与模式判定

1. **【条目 Tn · 来源】**（必达，用户暂停点）
   - 触发节点：每条任务进入本 skill 的第一步
   - 选项与暂停语：使用 [`manager.md`](../../../manager.md) §暂停语总集 第 1 行（`【来源】` 模板，前缀写为 `【条目 Tn · 来源】`）
2. 读 §当前在跑 该 `T-n` 行：
   - 状态必须 ∈ `{已执行, 验收中}`；否则提示「不在验收环节，跳过」并退出
   - 读取 `任务摘要` / `acceptor_role[]` / `DoD` / `执行结果` / 已存在的 `验收结论`（若有）
3. 模式判定：
   - 用户字面包含「让 / 派 / 调 acceptor」或编排器主动调度 → **A 主动模式**
   - 用户已附验收结论 / `acceptor_role` 自助提交 → **B 被动模式**
4. 公示：

   ```text
   【accept 模式】<A 主动调用｜B 被动接收>；T-<n>；acceptor_role[]=<…>；当前状态=<已执行｜验收中>；已收 k/N 位结论。
   ```

### 阶段二 · 验收（按所选模式）

#### 模式 A · 主动调用 acceptor_role

1. **多位验收人调度策略**：
   - 默认按 `acceptor_role[]` 列表顺序逐位调度（第 1 位 → 第 2 位 → … → 第 N 位）
   - 每位调度结束后回到主流程检查状态（第 1 位完后扭转 `验收中`，可暂停等下一位；批量模式可连调）
2. 公示「【调用 acceptor】T-<n> → <第 k 位 acceptor>（k/N）」
3. 按下方 §Task 委派模板 切到 `acceptor_role` 子代理执行
4. 等回执：
   - **超时 / 拒绝 / 角色拒绝**：转 §阻塞单（类型：`角色不响应`），状态保持当前，本任务退出
   - **回执有结论**：进入阶段三守门
5. 接收 `acceptor_role` 的结构化回执（要素见 §结论交付要素）

#### 模式 B · 被动接收

1. 公示「【接收结论】T-<n> ← <acceptor_role 自助>（k/N）」
2. 校验提交人是否在 `acceptor_role[]` 范围内 + 是否已经提交过（重复提交按用户意图覆盖 / 拒绝）
3. 接收材料 → 进入阶段三守门

### 阶段三 · DoD 完备度守门 + 状态扭转

1. 按 §DoD 完备度守门 5 项逐条检查
2. **任一不通过**：
   - 输出 `【守门未通过】T-<n>：<缺啥项>，请 acceptor_role=<…>（k/N） 补齐结论：<具体期望>。`
   - 状态保持当前，本任务退出
3. **全通过**：
   - 把本位结论追加写到 §当前在跑 该 T-n 的 `验收结论` 列：`<acceptor> @ <YYYY-MM-DD>: 通过/不通过/条件通过 — <要点>; 依据：<…>`（多位以 `;` 分隔）
   - 状态扭转判定（按 §多人验收语义 表）：
     - 单人 + `通过` → `已验收`
     - 多人 · 第 1 位 + 任意结论 → `已执行 → 验收中`
     - 多人 · 第 k 位（1 < k < N） + 任意结论 → 保持 `验收中`
     - 多人 · 第 N 位 + **全部通过** → `验收中 → 已验收`
     - 多人 · 第 N 位 + 存在 `不通过` → 保持 `验收中`，触发阻塞升级（manager 决定是否打回执行）
   - 输出 `【验收推进】T-<n>，状态：<旧> → <新>，进度 k/N。`

### 阶段四 · 收口

1. 本轮处理摘要：

   ```text
   ## 【accept 摘要】
   - 扭转到 已验收：N1 条
   - 扭转到 验收中（多人推进中）：N2 条
   - 守门未通过（保持当前）：N3 条
   - 阻塞升级（不通过）：N4 条
   ```

2. 阻塞 / 守门未通过 → 升级 / 留印 后退出。

## §结论交付要素（acceptor_role 视角）

`acceptor_role` 通过本 skill（主动 / 被动）交付结论时，须包含：

| 要素 | 说明 |
|------|------|
| `任务编号` | `T-n` |
| `验收人` | 子代理名（如 `qa` / `pm`） |
| `验收时间` | `YYYY-MM-DD` |
| `结论` | `通过` / `不通过` / `条件通过` 三选一 |
| `结论要点` | 一段话说明判断依据 |
| `DoD 对照` | 对应 DoD 每条标准的核对结果（通过 / 不通过 / 条件通过 + 依据） |
| `不通过定位`（结论=不通过 时必填） | 具体 DoD 标准编号 + 缺啥 + 建议 |
| `条件清单`（结论=条件通过 时必填） | 待补条件 + 期望时限 |

## Task 委派模板（主动模式专用）

单条提示词模板要素：**任务编号（T-n）** ｜ **任务摘要 + DoD + 执行结果链接** ｜ **`subagent_type` = 当前轮次的 acceptor_role** ｜ **结论交付要素清单** ｜ **回填路径**（统一任务列表 §当前在跑 该 T-n 行 `验收结论` 列）。

```text
【委派方式】Task（任务：T-<n>，第 k 位验收）；subagent_type=<acceptor_role>；要求：按 §结论交付要素 提交结论，DoD 对照逐条覆盖。
```

向 Task 写明：**「仅服务本任务 T-<n> 的验收；结论按结构化要素提交；不顺带改其他任务的验收结论列；不跨界写执行 / 质检 / 通晒列」**。

## 阻塞单格式

```text
## 阻塞单 B-<序号>
- 关联条目：T-<n>
- 阻塞类型：角色不响应 / 角色拒绝 / 不通过待裁决 / 多位结论冲突 / 验收材料无法核验
- 触发条件：…
- 已尝试：…
- 升级路径：→ manager（由 manager 决定打回执行 / 调整 DoD / 调整验收人）
- 截止预期：…
```

阻塞单同步追加到统一任务列表 §阻塞单。

## 输出格式（默认）

```text
## 【accept 模式】公示
## 【调用 acceptor / 接收结论】公示
## §结论交付要素 摘要
## 【守门检查】结果（5 项）
## §当前在跑 T-n 行更新（验收结论列追加、状态扭转）
## 【accept 摘要】
## 阻塞单（若有）
```

## 注意事项（task-accept 专属）

- **守门是核心职责**：宁可保持当前状态等补齐，不可放水扭转 `已验收`
- **多人语义守门**：必须严格按 §多人验收语义 表判断状态扭转，第 1 位提交即扭转 `验收中`，全部通过才扭转 `已验收`
- **不跨列写**：仅写 §当前在跑 该 T-n 的 `验收结论` 列与 `状态` 列；不写 `执行结果 / 质检结论 / 通晒回填` 列
- **不跨任务**：单次验收只服务本 T-n
- **角色守门**：提交人必须在 `acceptor_role[]` 范围内
- **不通过不直接打回**：不通过 / 多位冲突 → 升级到 manager，由 manager 决定是否调用质检环节或直接打回执行；本 skill 不主动改 §当前在跑 状态为 `待执行`
- **信息边界**：本 skill 不修改任务请求文件 / 上游产物 / `task_request` 字段
