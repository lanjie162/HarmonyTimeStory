# Skill: todo-drive

> **Skill 元信息**
>
> - **属于**：[`manager`](../../manager.md)
> - **目的**：在跟踪源（默认 `document/TODO-跟踪.md`）上把 `[ ]` 推进到**证据齐备的可验证完成态**：盘点 → 推导 → 派发 → 驱动 → 回收 → `[x]`。
> - **入口**：`document/TODO-跟踪.md`、`document/todo/`，或用户提及「TODO / 待办 / 派发 / 跟催 / 关单 / 收口」。
> - **常见参与角色**：`pm` / `architect` / `dev` / `qa` / `ued`，按条目类型动态推导。
> - **专属编号**：`T-n`（条目）/ `B-x`（阻塞 / 待裁决）；升级到 `CR-x` 由 `version-planning` 收口。详见 [`manager.md`](../../manager.md) §决策点编号 - 总集。
> - **来源选型触发**：**每条目开始**（阶段二步骤 1）。规则按 [`manager.md`](../../manager.md) §来源选型 § 节奏 § 暂停语总集 执行，本文不重述。
> - **产物**：`【TODO 本轮范围】` + 执行表 + 派发记录 / 回收记录 + 阻塞单（若有），并就地维护跟踪源（写 `[x]`、状态、证据）。

## 与 review-meeting / version-planning 的衔接

| 关系 | 说明 |
|------|------|
| `version-planning` → 本 skill | 定稿计划中「移交 todo-drive 包」标题级清单，由本 skill 落条目并驱动收口 |
| `review-meeting` → 本 skill | 纪要 📅 行动项 → 落跟踪行 → 由本 skill 推进 |
| 本 skill → `version-planning`（反向） | 条目阻塞或与定稿冲突 → 升级请求 `CR-x` 修订计划，本 skill 不私自改里程碑 |
| 本 skill → `review-meeting`（反向） | 条目暴露方案级争议 → 申请新开评审议题 |

**边界**：本 skill 不做里程碑 / 路线图（属于 `version-planning`）；不替代评审节奏（属于 `review-meeting`）。本 skill 是 **DoD 与证据完整性守门**。

| 子代理 | 典型条目 |
|--------|----------|
| `pm` | A 区、范围 / 验收 / 版本号、CR 评估、纪要产品类 |

### 推导优先级（自上而下命中即补强）

1. **用户显式**（最高）：用户指定「本条只走 `dev`」等 → 以用户为准。
2. **跟踪表「源」列路径**（默认）

   | 路径或目录特征 | 建议驱动顺序（本条内串行） |
   |----------------|----------------------------|
   | `requirement/`、`需求 / PRD / MVP 需求` | `pm` → `architect` → `dev` →（涉验收则）`qa` |
   | `techdoc/`、`技术方案 / HarmonyOS 平台能力调研` | `architect` → `dev` → `qa` |
   | `design/`、`原型 / UI` | `ued` → `pm` → `dev` |
   | `meeting/`、纪要 Action | 按主题判断；不明朗时只问一句「本条偏产品 / 方案 / 实现 / 测试？」再映射上表 |
   | 单一路径且明显仅测试 / 仅实现 | 可收缩为单一 `qa` / `dev` |

3. **区段启发式**（与 `TODO-跟踪.md` 章节一致时）：A → 产品牵头；B → 架构 → 研发；C → 测试牵头（必要拉研发排障）；D / E / F 按「源」微调。
4. **仍模糊**：最短一问定主责角色，勿罗列所有子代理让用户选。

### 推导公示（开场 / 每条目必出，可合并）

`【推导】条目：<# 或摘录>；本子代理序列：<name1 → name2 → …>；依据：<源列 / 路径 / 区段 / 用户原话>。`

> 角色缺失处理按 manager.md §子代理正文绑定 第 5 条执行。

## 证据四要素（守门标准）

每个条目关单必须满足：**证据 ID + 来源工具 + 证据路径 + 结论**。

- **证据 ID**：在跟踪源中可被引用的稳定标识（如 `EV-V1-T2-state-check`）。
- **来源工具**：MCP / `hdc` / 编辑器 / 评审纪要 / 计划终稿等明确名称。
- **证据路径**：可点击或可定位的相对路径（如 `document/evidence/V1/V1-T2/[研发]V1-T2-state-check-….md`）。
- **结论**：通过 / 有条件通过 / 不通过；以及是否覆盖派发 DoD。

证据不齐 → **不得 `[x]`**；技术对错由 `architect` / `qa` 等项目惯例兜底，本 skill 是**完整性守门**。

## 核心工作流

### 阶段一 · 情境、盘点与编排公示

1. 读跟踪源，拉出**本轮要推的条目**（含待派发、进行中、待验收）。
2. 对每条给出**推导子序列**（见上表）；合并输出 `【TODO 本轮范围】`：N 条，阻塞 K，待验收 M。
3. 编排模式 / 全流程执行方式按 manager.md §节奏 §来源选型 §全流程默认 处理；未公示则各条目在阶段二步骤 1 单独选择。

### 执行前预检（硬性）

每条目进入驱动前必须通过预检：

1. 是否存在明确 DoD（无则退回补齐）。
2. 是否声明证据字段（至少：证据 ID、来源工具、路径、结论）。
3. 是否确认执行来源（按 manager.md §来源选型）。
4. 依赖是否可执行（依赖未满足则标阻塞，不派发）。

### 阶段二 · 逐条目驱动（主循环）

对每个待推进条目，**不跳步**（批量模式按 manager.md §节奏 执行）：

1. **【条目 Tn · 来源】**（必达）  
   - 触发节点：每条目开始。  
   - 选项与暂停语：使用 manager.md §暂停语总集 第 1 行（`【来源】` 模板，前缀写为 `【条目 Tn · 来源】`）；其余规则按 §来源选型 全文执行。
2. **呈现条目**：原文摘要、源链、**DoD**、依赖、截止。
3. **驱动子代理完成**（遵从步骤 1）：  
   - **模拟**：按推导顺序读 md → `【子代理名 · 本条产出】`。  
   - **Task**：按顺序逐角色 Task；主会话汇编为连贯本条结果；不完整则按 §来源选型 失败回退。
4. **回收检入**：对照证据四要素与派发 DoD；够则 `[x]` + 源文档同步，输出 `【回收记录】`；缺则退回**进行中**并一条说清缺啥。
5. **【条目 Tn 完毕】**：使用 manager.md §暂停语总集 第 4 行；批量模式可声明跳过。

#### 状态口径（在汇总表维护）

| 状态 | 含义 |
|------|------|
| 待派发 | 未进入步骤 3 |
| 进行中 | 已驱动或已派 Task，尚未过检入 |
| 待验收 | 待用户或 manager 确认是否关单 |
| 已关闭 | 证据齐、`[x]` 已写 |
| 阻塞 | 等人 / 决策 → **阻塞单 B-x** |
| 待裁决 | 产品与测试结论冲突，或重大阻塞需用户裁决 |

### 阶段三 · 跟催与例会顺序

- **T-3 / 超期**：单条要点名；**清点进度** → 进行中一览。
- 建议顺序：**盘点 → 先关待验收 → 下发 / 驱动 → 跟催**。

## Task 委派模板

单条提示词模板要素：**条目编号** ｜ **摘录** ｜ **输入路径** ｜ **唯一 DoD** ｜ **`subagent_type`** ｜ **禁止改其他勾选**。

`【委派方式】Task（TODO：<条目标识>，角色：<name>）；回退：按 manager.md §来源选型 失败回退。`

向 Task 写明：**「仅服务本条目；禁止顺带改其他 `[ ]`；输出限于本条 DoD 的证据与结论」**。

## 阻塞单格式

```text
## 阻塞单 B-<序号>
- 关联条目：T-<…>
- 阻塞类型：等人 / 等决策 / 工具 / 数据 / 上游
- 触发条件：…
- 已尝试：…
- 升级路径：→ version-planning（CR-x） / review-meeting / 用户裁决
- 截止预期：…
```

## 输出格式（默认）

```text
## 【TODO 本轮范围】与 【推导】摘要
## 【执行来源】 / 【编排模式】 / 【全流程执行方式】（若有）
## 执行表（条目 ｜ 状态 ｜ 子代理序列 ｜ 截止 ｜ 最后证据）
## 本条：【子代理 · 产出】汇编（若多角色）
## 派发记录 / 回收记录
## 阻塞单（若有）
```

## 注意事项（todo-drive 专属）

- **证据不齐不 `[x]`**：技术对错由专业角色兜底，本 skill 是 DoD 与证据完整性守门。
- **不代替 `version-planning` 改里程碑**：条目与计划冲突 → 升级 `CR-x` 或 @manager（version-planning skill）。
- **冲突冻结**：若 `pm` 与 `qa` 对同条目结论冲突，必须冻结为**待裁决**并抛出决策点，不得自行折中关单。
- **不篡改已定稿纪要 / 附录的裁定**。

> 通用注意事项（保持中立 / 灵活适配 / 上下文管理 / 引用规范 / 完整性兜底）见 manager.md §注意事项。

风格：中文；结论先行；表格优先。

**状态（manager 自用）**：`[待派发 / 推进中 / 已本轮收口 / 阻塞升级]`
