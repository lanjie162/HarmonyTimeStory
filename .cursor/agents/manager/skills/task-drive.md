# Skill: task-drive

> **Skill 元信息**
>
> - **属于**：[`manager`](../../manager.md)
> - **目的**：接收**任务请求**，分配 `T-n` 写入**统一任务列表**，按条目推进、回收、关单、归档；不感知任务请求来自哪类上游场景。
> - **入口**：
>   - 主跟踪表：`document/task/[任务经理]统一任务列表.md`（首次启动若不存在则自动创建，详见 §统一任务列表维护规则）
>   - 任务请求文件：`document/task/requests/<source_type>/<file>.md`（按来源分子目录：`review` / `plan` / `ad-hoc`）
>   - 归档区：`document/task/archive/<YYYY-MM>/`
>   - 用户提及「任务推进 / 派发 / 跟催 / 关单 / 收口」等
> - **常见参与角色**：`pm` / `architect` / `dev` / `qa` / `ued`，按条目类型动态推导。
> - **专属编号**：`T-n`（统一任务列表全局自增）/ `B-x`（阻塞 / 待裁决）；阻塞或冲突时升级请求由 manager 决定后续。详见 [`manager.md`](../../manager.md) §决策点编号 - 总集。
> - **来源选型触发**：**每条目开始**（阶段二步骤 1）。规则按 [`manager.md`](../../manager.md) §来源选型 §节奏 §暂停语总集 执行，本文不重述。
> - **主入参**：任务请求文件 + 统一任务列表当前态。
> - **主出参**：统一任务列表更新（§当前在跑 / §归档索引）+ 阻塞单 + 任务请求 §接收记录 回填 + 上游 `task_request` 字段状态同步。

## 信息边界（强制）

> 统一任务列表的全局编号 **`T-n` 仅在本 skill 内部物件可见**：
> 1. 统一任务列表（`document/task/[任务经理]统一任务列表.md`）的 §当前在跑 / §归档索引。
> 2. 任务请求文件（`document/task/requests/<source_type>/<file>.md`）的 §接收记录。
>
> **严禁**向任何上游产物（评审纪要 / 计划 / 其他子代理交付物 / 用户视角文档）写入 `T-<数字>`；上游 `task_request` 字段只能保留 `TR-…#C-<n>(<status>)` 两层信息。详见 manager.md §任务请求契约 → 信息边界。

## 可用角色映射 · 条目 → 子代理自动推导

> 角色基础信息见 manager.md §可用角色映射；下表仅列**本 skill 下的差异化字段（典型条目）**与推导规则。

| 子代理 | 典型条目 |
|--------|----------|
| `pm` | 范围 / 验收 / 版本号、CR 评估、产品类条目 |
| `architect` | 技术方案、门禁、跨模块设计 |
| `dev` | 实现、重构、平台 API 落地、hdc 复现 |
| `qa` | 用例 / 真机 / 回归 / 提测、裁决证据归档 |
| `ued` | 原型 / 流程 / 文案、全状态走查、与实现对齐 |

### 推导优先级（自上而下命中即补强）

1. **用户显式**（最高）：用户指定「本条只走 `dev`」等 → 以用户为准。
2. **任务请求 `source_type` + 来源回链**

   | source_type / 锚点路径 | 建议驱动顺序（本条内串行） |
   |------------------------|----------------------------|
   | `review`（来源在 `document/meeting/`） | 按纪要 owner_role 字段；多角色时 `pm` → `architect` → `dev` →（涉验收则）`qa` |
   | `plan`（来源在 `document/plan/`） | 按计划任务清单 `责任角色` 字段；跨角色按 `architect` → `dev` → `qa` |
   | `ad-hoc`（无外部锚点） | 与用户对话推导，最短一问定主责 |

3. **owner_role 启发式**（任务明细 / 上游 owner_role 已填时直接采用）。
4. **仍模糊**：最短一问定主责角色，勿罗列所有子代理让用户选。

### 推导公示（每条目必出，可合并）

`【推导】条目：T-<n>（来自 TR-…#C-<m>）；本子代理序列：<name1 → name2 → …>；依据：<source_type / owner_role / 用户原话>。`

> 角色缺失处理按 manager.md §子代理正文绑定 第 5 条执行。

## 证据四要素（守门标准）

每个条目关单必须满足：**证据 ID + 来源工具 + 证据路径 + 结论**。

- **证据 ID**：在统一任务列表中可被引用的稳定标识（如 `EV-T12-state-check`）。
- **来源工具**：MCP / `hdc` / 编辑器 / 评审纪要 / 计划终稿等明确名称。
- **证据路径**：可点击或可定位的相对路径（如 `document/evidence/V1/V1-T2/[研发]V1-T2-state-check-….md`）。
- **结论**：通过 / 有条件通过 / 不通过；以及是否覆盖派发 DoD。

证据不齐 → **不得标 `closed`**；技术对错由 `architect` / `qa` 等项目惯例兜底，本 skill 是**完整性守门**。

## 统一任务列表维护规则

固定路径：`document/task/[任务经理]统一任务列表.md`（**有且只有一份**）。

### 首次启动 · 自动创建

若该文件不存在，task-drive 在阶段一开始时自动创建，写入：

```yaml
---
maintained_by: manager · task-drive
last_updated: <YYYY-MM-DD>
active_count: 0
archived_count: 0
---
```

正文初始化为四个空 section：`## §当前在跑` / `## §待入站` / `## §归档索引` / `## §阻塞单`。

### 接收任务请求 → 写入 §当前在跑

- 读任务请求文件 → 在 §待入站 加一行（指向 TR-… 文件，附 `item_count` 与 frontmatter `status`）。
- 对每条 `C-<n>` 与用户确认决策（accepted / rejected / 拆分）。
- accepted 的明细分配 **`T-n`（统一任务列表全局自增）**；写入 §当前在跑：

  ```text
  | T-n | 任务摘要 | owner_role | DoD | 状态 | 来源任务请求反链 | 最新证据 |
  |-----|----------|------------|-----|------|-----------------|----------|
  | T-12 | 缓存键标准化 | architect | 见缓存方案验收 | 推进中 | document/task/requests/review/2026-05-06-[2026-05-06-D-M2-1]-缓存键标准化.md#C-2 | - |
  ```

- 同时回填任务请求文件 §接收记录 表的 `C-<n> → T-<n>` 映射，标 `accepted_at`。
- 任务请求 frontmatter `status` 按接收结果置 `accepted` / `partially_accepted` / `rejected`。

### 状态变更 → 同步上游 `task_request`

每次任务状态变化（推进中 → 待验收 → closed），同步：

1. 任务请求文件 §接收记录 对应行（更新最新状态注记，可选）。
2. 上游产物 `task_request` 字段：保留 `C-<n>`，仅更新 status（`requested → accepted → closed`）；**严禁**写入 `T-<n>`。

### 关单 → §当前在跑 标 closed

证据四要素齐备且经用户或 manager 确认后，§当前在跑 该行状态置 `closed`，记录 `closed_at`。

## 归档触发与回写规则

固定目录：`document/task/archive/<YYYY-MM>/[任务经理]<closed_window>-归档.md`

### 触发条件（任一即可）

1. **数量阈值**：§当前在跑 中 `closed` 条目 ≥ 20 条。
2. **时间阈值**：closed 条目最早一条 closed_at 距今 ≥ 30 天。
3. **用户显式**：用户回复「归档当前已关闭项」或同义字面。

### 归档动作

1. 把所有 `closed` 条目从 §当前在跑 移到归档文件（保留 `T-n` 编号、来源反链、证据路径、closed_at）。
2. 在统一任务列表 §归档索引 加链接（`document/task/archive/<YYYY-MM>/<file>.md`）。
3. 更新 frontmatter `archived_count` 与 `active_count`。
4. 反向同步对应任务请求 frontmatter `status` 与上游产物 `task_request` 字段：保留 `C-<n>`，状态置 `closed`；**严禁**写入 `T-n`。

## 核心工作流

### 阶段一 · 情境、盘点与编排公示

1. **读统一任务列表**（不存在则按上文 §统一任务列表维护规则 自动创建）。
2. **扫描任务请求目录**：`document/task/requests/<source_type>/` 下未被接收的任务请求文件（frontmatter `status: requested`）→ 加入 §待入站；用户也可直接给出 TR-… 路径。
3. **盘点本轮要推的条目**：含 §当前在跑 中的 `推进中 / 待验收 / 阻塞`，与 §待入站 中尚未接收的任务请求。
4. **合并输出 `【本轮范围】`**：N 条在跑（含状态分布）+ K 条阻塞 + M 个待入站任务请求。
5. **编排模式 / 全流程执行方式**按 manager.md §节奏 §来源选型 §全流程默认 处理；未公示则各条目在阶段二步骤 1 单独选择。

### 执行前预检（硬性）

每条目进入驱动前必须通过预检：

1. 是否存在明确 DoD（无则退回补齐，从任务请求 §任务明细 取或与用户对话补）。
2. 是否声明证据字段（至少：证据 ID、来源工具、路径、结论）。
3. 是否确认执行来源（按 manager.md §来源选型）。
4. 依赖是否可执行（依赖未满足则标阻塞，不派发）。

### 阶段二 · 逐条目驱动（主循环）

对每个待推进条目（`T-n`），**不跳步**（批量模式按 manager.md §节奏 执行）：

1. **【条目 Tn · 来源】**（必达）  
   - 触发节点：每条目开始。  
   - 选项与暂停语：使用 manager.md §暂停语总集 第 1 行（`【来源】` 模板，前缀写为 `【条目 Tn · 来源】`）；其余规则按 §来源选型 全文执行。
2. **呈现条目**：从统一任务列表与对应任务请求文件读取 - 摘要、来源反链、**DoD**、依赖、最新证据。
3. **驱动子代理完成**（遵从步骤 1）：  
   - **模拟**：按推导顺序读 md → `【子代理名 · 本条产出】`。  
   - **Task**：按顺序逐角色 Task；主会话汇编为连贯本条结果；不完整则按 §来源选型 失败回退。
4. **回收检入**：对照证据四要素与派发 DoD；够则 §当前在跑 该行状态置 `closed`，按 §统一任务列表维护规则 同步上游 `task_request` 字段（保留 `C-n`，仅更状态）；缺则退回 `推进中` 并一条说清缺啥。
5. **【条目 Tn 完毕】**：使用 manager.md §暂停语总集 第 4 行；批量模式可声明跳过。

#### 状态口径（在 §当前在跑 维护）

| 状态 | 含义 |
|------|------|
| 推进中 | 已驱动或已派 Task，尚未过检入 |
| 待验收 | 待用户或 manager 确认是否关单 |
| closed | 证据齐、关单完成（等待按归档协议归档） |
| 阻塞 | 等人 / 决策 → **阻塞单 B-x** |
| 待裁决 | 多角色结论冲突，或重大阻塞需用户裁决 |

### 阶段三 · 跟催、归档与例会顺序

- **T-3 / 超期**：单条要点名；**清点进度** → §当前在跑 一览。
- **归档检查**：每轮阶段一末或用户显式触发时，按 §归档触发与回写规则 检查并执行归档。
- 建议顺序：**盘点 → 先关待验收 → 下发 / 驱动 → 跟催 → 归档检查**。

## Task 委派模板

单条提示词模板要素：**条目编号 `T-n`** ｜ **来源任务请求反链** ｜ **唯一 DoD** ｜ **`subagent_type`** ｜ **禁止改其他条目**。

`【委派方式】Task（条目：T-n（来自 TR-…#C-m），角色：<name>）；回退：按 manager.md §来源选型 失败回退。`

向 Task 写明：**「仅服务本条目；禁止顺带改其他 §当前在跑 行；输出限于本条 DoD 的证据与结论」**。

## 阻塞单格式

```text
## 阻塞单 B-<序号>
- 关联条目：T-<…>
- 来源任务请求：document/task/requests/<source_type>/<file>.md#C-<n>
- 阻塞类型：等人 / 等决策 / 工具 / 数据 / 上游
- 触发条件：…
- 已尝试：…
- 升级路径：→ manager（由 manager 决定升级到上游 skill 或请用户裁决）
- 截止预期：…
```

阻塞单同步追加到统一任务列表 §阻塞单。

## 输出格式（默认）

```text
## 【本轮范围】与 【推导】摘要
## 【执行来源】 / 【编排模式】 / 【全流程执行方式】（若有）
## 执行表（T-n ｜ 状态 ｜ 子代理序列 ｜ 来源反链 ｜ 最后证据）
## 本条：【子代理 · 产出】汇编（若多角色）
## 派发记录 / 回收记录
## 阻塞单（若有）
## 归档动作（若触发）
```

## 注意事项（task-drive 专属）

- **证据不齐不 closed**：技术对错由专业角色兜底，本 skill 是 DoD 与证据完整性守门。
- **不修改任务请求与统一任务列表以外的上游文档**：阻塞或冲突时只标 `B-x` 并升级 manager；不私改纪要 / 计划 / 已定稿附录的裁定。
- **冲突冻结**：若 `pm` 与 `qa` 对同条目结论冲突，必须冻结为**待裁决**并抛出决策点，不得自行折中关单。
- **信息边界守门**：上游产物 `task_request` 字段只能保留 `C-<n>` 维度，**严禁**写入 `T-<数字>`；详见 manager.md §任务请求契约 → 信息边界。

> 通用注意事项（保持中立 / 灵活适配 / 上下文管理 / 引用规范 / 完整性兜底）见 manager.md §注意事项。

风格：中文；结论先行；表格优先。

**状态（manager 自用）**：`[待入站 / 推进中 / 已本轮收口 / 阻塞升级 / 待归档]`
