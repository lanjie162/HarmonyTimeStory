# Skill: version-planning

> **Skill 元信息**
>
> - **属于**：[`manager`](../../manager.md)
> - **目的**：在 AI 闭环项目下产出**版本计划 + 版本内任务计划**：策略 → 多角色征询 → 决策点 P-x → **《最终工作计划》**（标准化产物，含计划任务清单与 §可选 · 打包推荐）经用户定稿。
> - **入口**：`document/plan/`、文件名含 `计划 / 策略 / 路线图 / 版本`，或用户显式声明做计划。
> - **常见参会角色**：`pm`、`sa`、`dev`、`qa`、`ued`。
> - **专属编号**：`S-x`（策略级分叉）/ `P-x`（计划主决策点）/ `CR-x`（已定稿计划的变更控制；其他流程升级到此）。详见 [`manager.md`](../../manager.md) §决策点编号 - 总集。
> - **来源选型触发**：**多角色断点 / 轮二开场 / 计划修订轮**。规则按 [`manager.md`](../../manager.md) §来源选型 §节奏 §暂停语总集 执行，本文不重述。
> - **硬约束**：**§决策守门**（置于「轮一」之前）— 禁止跳 `S-/P-/CR-` 后代拍板、单决策点独占回合、`draft/confirmed` 与「【终稿待你确认】」绑定；与本文其他条款冲突时以该节为准。
> - **主入参**：任意目标命题（用户开放表达 / 已定稿需求 / 已评审纪要 等）。
> - **主出参**：标准化《最终工作计划》（单一产物，内含计划策略、分角色征询摘要、决策点 P-x 记录与计划任务清单；用户定稿后冻结，路径与字段见下文 §计划产物标准化）。

## 计划口径（硬性）

- 不采用传统**按日期排期**作为主输出（允许仅作备注锚点）。
- 不使用**人日 / 人周**作为主估算。
- 工作量统一使用**复杂度分级**（S / M / L / XL 或 1~5 级）。
- 计划核心产物必须是：**版本计划**（Version Plan）+ **版本内任务计划**（Task Plan by Version）。

## 计划产物标准化（独立可读 · 无下游依赖）

### 文件命名

```text
document/plan/[项目经理]<YYYY-MM-DD>-<目标短语>-最终工作计划.md
```

### frontmatter（《最终工作计划》必填）

```yaml
---
plan_target: <一句话目标>
created_at: <YYYY-MM-DD>
created_by: manager
status: draft | confirmed
version_count: <版本数量>
---
```

### 计划任务清单（标准化输出 · 强制列）

每个版本下输出「版本内任务计划」表，作为本 skill 的**核心标准化产物字段**：

| 任务编号 | 任务摘要 | 复杂度 | 前置依赖 | 责任角色 | 验收口径 | task_request |
|----------|----------|--------|----------|----------|----------|--------------|
| `Vx-Ty` | …      | S/M/L/XL | `Vx-Tz` 或 - | dev / qa / … | 一句话验收 | 留空 / `TR-<YYYYMMDD>-<seq>#C-<n>(<status>)` |

字段约束：

- `任务编号`：`Vx-Ty`，按版本内自增；保留作为追溯锚点。
- `责任角色`：仅取自 manager.md §可用角色映射 中的 `subagent_type`。
- `task_request`：**派发追溯字段**，由 manager 在打包模式下回写；规则见 manager.md §上游产物 `task_request` 字段。**严禁**写入 `T-<数字>`（信息边界，详见 manager.md §任务请求契约 → 信息边界）。

### 产物落档原则

- 计划本身**独立可读**：用户、其他子代理拿到《最终工作计划》即可作为一份完整工作纲领；不依赖任何下游产物。
- 是否将 `Vx-Ty` 任务打包为任务请求或进入推进流程，由 manager 在用户主动发起时决定（详见 manager.md §上游产物 → 任务请求 转换协议）；本 skill 不做任何下游切换决策。

## 主工作流程（必须由 manager 驱动）

全流程分**四轮**；节奏按 manager.md §节奏 「交互式默认」执行。

### 多角色断点（来源选型必达）

凡出现以下任一场景，均视为「多角色断点」，按 manager.md §来源选型 处理（暂停语使用 §暂停语总集 第 1 行，前缀写为 `【多角色启动方式确认】`）：

- 轮二「分角色建议与风险」启动前；
- 任一轮内新增第 2 个及以上子代理参与时；
- 计划修订轮中再次发起跨角色征询时。

> 沿用 / 失效条件 / 失败回退按 manager.md §来源选型 全文执行。

### §决策守门（硬约束，优先于叙事惯性）

以下条款与 [`manager.md`](../../manager.md) §节奏「交互式（默认）」一致并**加严**；与本 skill 其他段落冲突时，**以本节为准**。

1. **禁止代拍板**  
   - 在用户未就该 `S-x`/`P-x`/`CR-x` 给出 **明确选项或约束**（当轮助手输出后的下一则用户回复，或你已声明且可追溯的同轮字面）之前：**禁止**在《最终工作计划》草案、本会话或 frontmatter 中对该编号使用 **终局绑定**措辞（包括但不限于「✅已定」「P-x 已定案」「策略已封口」「本节为最终版」「计划已定稿」「已对齐」等）。  
   - **「主持人备选（pending）」≠ 用户选择**；仅能标 **待用户确认**，**禁止**默认为采纳或在未获回复前写入 `P-x 结论：…`。

2. **单决策点独占回合**  
   - 若本助手回合须用户裁决，须在输出 **恰好一个** `【决策点 S-x】`、`【决策点 P-x】` 或 `【决策点 CR-x】`（使用 manager.md §暂停语总集 第 2 行字面，`X-y` 换成对应编号）并提出选项后 **立即结束本轮输出**。**禁止**在同回合后再输出：`《最终工作计划》` 正文大段、`§计划任务清单` 全文、frontmatter 定稿、`§可选 · 打包推荐`、下一轮次的编排长篇、以及对其它决策点的 **`P-x 结论`/预收敛**行文。  
   - **例外**：用户已声明 **批量模式**且本场已公示 `【批量模式】`（manager.md §节奏）；纪要/草案中仍须标注哪些决策未逐项经用户确认。

3. **必须进入 `P-x` 收敛的触发条件**（不得以「备选已写明」收口而跳过 §轮三暂停）  
   自轮二《分角色征询摘要》可归并判断：若满足 **任一**，则须在 §轮三 至少列出一个待决策编号（可归并为一条多维 `P-*`，但仍须 **逐条按守门第 2 款**与用户收敛）：  
   - 征询摘要中存在 **与其它角色的冲突点**非空或未消解；  
   - 任一角色条目含 **条件 / 前提**（「若…则…」「取决于…」「待…确认」）；  
   - **版本切分、范围、门禁、复杂度口径、优先级** 等在角色间表述 **未收口到同一取值**；  
   - 轮一 **`S-x` 未定**或未声明不产生 `S-*`（见下款）；  
   - 本条 skill **其它条款要求升级为 `P-x`** 的情形（如信息不足不许自行裁定）。

4. **策略级不产生 `S-x` 的显式声明**  
   若轮一末尾判断 **无策略级分叉**：须在进入轮二前单列一行 **`本轮不产生 S-*（依据：<一句话事实>）。`** **禁止**静默跳过轮一秒末分叉判定。

5. **轮三不产生 `P-x` 的显式声明（少见）**  
   仅在轮二摘要 **客观无冲突**、无未收口维度、且无本 skill「须升级 P-x」事项时适用：须在启动轮四汇总前单列 **`本轮不产生 P-*（依据：<一句话事实>）。`** **禁止**在无此声明情况下直接产出含「已定稿口吻」的计划正文。

### 轮一 · 计划策略（写入《最终工作计划》内的「计划策略」章节）

在读取用户目标 + 关键文档（已定稿需求 / 技术方案 / 评审纪要等任意路径）后，先形成计划策略草案，并在轮四汇总时落到《最终工作计划》的「计划策略」章节，至少含：

- **目标与范围假设**（含明确**非目标**）
- **阶段划分逻辑**（探索 / 骨架 / 核心功能 / 联调发版 等可改名）
- **协调策略**：拟征询哪些角色、为何、何顺序
- **版本切分原则**：按业务价值、风险前置、依赖顺序拆为 V1 / V2 / …
- **复杂度评估口径**：采用 S / M / L / XL（或 1~5），并给判定规则
- **已知信息缺口**（需用户补一句或下一文件锚点）
- **本轮是否拟启用 Task 征询**（默认否；启用则公示范围）

固定一行：`【编排推断】计划目标：<…>；拟征询子代理：<序列>。`

**轮一末（若存在策略级分叉）**：使用 manager.md §暂停语总集 第 2 行（`【决策点 S-x】`，通常 `S-1`），并遵守 **§决策守门**（独占回合、禁止代拍板）。**无分叉**则输出 **`本轮不产生 S-*（依据：…）。`**（见守门第 4 条）。

### 轮二 · 组织协调：分角色建议与风险

按轮一公布的顺序，对每个已配置子代理执行（须先完成上文「多角色断点」）：

- **读** `.cursor/agents/<name>.md`
- **输出** `【<name> · 计划征询】`：**建议（What/Why）** + **风险（含触发条件）** + **依赖他角色的前提**（若有）

**轮二末**形成《分角色征询摘要》表：角色 ｜ 关键建议 ｜ 关键风险 ｜ 与其他角色的冲突点（并在轮四汇总进《最终工作计划》）。

### 轮三 · 决策点清单（用户确认）

由 manager 从**策略** + **分角色征询摘要**中抽出**待用户决策项**，按 **§决策守门第 3、5 条**判断是否必须立项；编号 `P-1 … P-n`。每条须含：**背景**、**选项**、**各角色立场摘要**、**主持人备选（pending，非定案）（可选）** —— **禁止使用「manager 倾向」等易被误读为已定案的标题**；若写主持侧倾向，仅能置于「主持人备选（pending）」名下。

每助手回合 **只推进一个** `P-x`（manager.md §节奏「交互式默认」+ **§决策守门第 2 条**），暂停语使用 §暂停语总集 第 2 行（`【决策点 P-x】`）。**用户回复之前**禁止写出 `P-x 结论：` 或任何终局定案句。

用户确认后，**下一回合或以新小节**仅以 **一行**追加：`P-x 结论：<…>`（记入终局计划伏笔），再处理下一 `P-x`。

### 轮四 · 《最终工作计划》汇总与定稿

在全部 `S-x`（若有）与 `P-x` 已与用户逐项收敛，或满足 **守门第 5 条「本轮不产生 P-*」声明** ，或用户声明「接受暂缓项」并排入风险且 **逐项记清暂缓编号**之后，产出《最终工作计划》草稿。**禁止**在仍有未闭环 `S-*`/`P-*`/`CR-*`（未声明暂缓且未裁决）时，输出带有 **已定稿/已对齐终局口吻**的正文或大段清单。

草稿落档规则补充（与下文「定稿门禁」一致）：
1. **与用户决策对齐**：仅对用户已确认的 `S-*`/`P-*`/`CR-*` 写 **结论性**措辞；未定项仅能标 **待定 / 待用户确认**或与「主持人备选（pending）」并列。  
2. **`frontmatter.status`**：**首次及更新落档默认为 `draft`**。仅在场外完成 manager.md §暂停语总集 **「【终稿待你确认】」**（可前缀「【《最终工作计划》待你确认】」）且用户 **`确认定稿`**（或逐项修正闭环）之后，下一次落档可将 `status` 置为 **`confirmed`**；此前 **禁止**写 `confirmed`。  
3. **禁止话术**：未完成上述终稿确认前，**禁止**对用户输出「本节为最终版」「计划已定稿」「可冻结执行」等暗示 **无需再等用户**的说法。

继而按 §计划产物标准化 落档。**默认正文结构**：

- **计划目标与范围**（草案：与已收敛 `S-*`/`P-*`/`CR-*` 对齐；未定处标 **待定**；终稿口径以 **【终稿待你确认】** 后为准）
- **计划策略**（由轮一沉淀并在此固化：目标与范围假设、阶段划分逻辑、版本切分原则、复杂度评估口径、信息缺口处理结论）
- **分角色征询摘要**（由轮二沉淀并在此固化）
- **决策点 P-x 记录**（由轮三沉淀并在此固化）
- **版本计划**（版本编号 ｜ 版本目标 ｜ 准入条件 ｜ 退出条件 ｜ 关键风险）
- **§计划任务清单**（按 §计划产物标准化 → 计划任务清单 标准列；含 `task_request` 列留空待 manager 后续回写）
- **关键路径与并行轨**（按版本）
- **风险登记与升级触发**
- **暂缓 / 留档项**（与决策记录对齐）
- **§可选 · 打包推荐**（manager 末尾固定追加，见下）

#### 版本模板（固定字段）

| 字段 | 必填说明 |
|------|----------|
| 版本编号 | `V1 / V2 / …` |
| 版本目标 | 单句可验收目标 |
| 准入条件 | 开始该版本前必须满足 |
| 退出条件 | 完成该版本必须满足 |
| 任务清单 | 见 §计划产物标准化 → 计划任务清单（强制列） |
| DoD 映射 | `Vx-Dy` 对应任务与证据 |

#### 复杂度分级（默认）

| 级别 | 含义（默认） | 计划用途 |
|------|--------------|----------|
| S | 单模块小改，依赖少，验证面窄 | 可并入当前版本 |
| M | 跨 2~3 模块或中等验证成本 | 作为版本内主任务 |
| L | 跨层改造、依赖多、验证面大 | 作为版本关键路径任务 |
| XL | 高不确定 / 高风险 / 跨团队 | 拆分子任务并前置风险验证 |

复杂度仅用于**排序、切分版本、识别风险**，不等同于人日换算。

### 定稿门禁（硬性）

- **不得**在未获用户明示前标注计划状态为「已对齐 / 已定稿」；与本 skill **§决策守门**及上文 **`draft`/`confirmed`** 规则冲突时 **以本节 + 守门为准**。
- 终稿暂停：**单独一轮**输出 manager.md §暂停语总集 第 5 行（`【终稿待你确认】`）；可前缀为 `【《最终工作计划》待你确认】`。该轮 **不得以**大块清单或打包节 **冲淡**暂停——打包节仍可出现于草案，但须在用户 **确认定稿之后再**在同一文件升版收敛表述（或在确认轮只呈现差异摘要）。
- 用户回复 **`确认定稿`**（或逐项修正闭环）后，下一轮可标记正文内 `状态：计划已定稿（用户确认于 <日期或可略>）`，并将 **`frontmatter.status`** 升为 **`confirmed`**；此前保持 **`draft`**。此后**以《最终工作计划》终稿为准**；后续是否进入任务推进由 manager 在用户主动发起时决定。

### §可选 · 打包推荐（manager · 轮四末尾固定追加）

> 即使用户未声明意图，也须输出本节作为可选打包候选；**不暂停、不询问、不切换** skill；用户后续主动发起打包后由 manager 进入打包模式（见 manager.md §上游产物 → 任务请求 转换协议 阶段 B）。

```text
## 可选 · 打包推荐（manager）

> 本节为可选打包候选，由用户主动决定是否打包。
> 当前没有正在打包的会话上下文。

| 候选编号 | 摘要 | 推荐 source_type | 推荐 owner_role | DoD 充分度 | 已派发 task_request |
|----------|------|------------------|-----------------|-----------|---------------------|
| V1-T1    | ...  | plan             | dev / qa / …    | 充分 / 待补 | 留空 / TR-…#C-… |

**推荐使用方式**：
- `打包候选 V1-T1,V1-T3 为 plan 任务请求` — 选择性打包并入档
- `先不打包` — 跳过本轮，所有候选保留待后续
```

候选行直接来自 §计划任务清单；`已派发 task_request` 列复制自该表对应字段。

## 变更控制（CR）

对已定稿计划的任何修改，必须走 `CR-x`：

1. 标注变更来源（需求变更 / 风险触发 / 工具阻塞 / 质量裁决）。
2. 关联原决策点（`P-x`）与受影响版本（`Vx`）。
3. 输出影响评估：范围、复杂度、DoD、风险。
4. 以**新决策点**请求用户确认后才生效（按 §轮三 节奏与 **§决策守门**，使用 `【决策点 CR-x】`；**独占回合**，禁止与《最终工作计划》大段改写同回合混写）。

## 注意事项（version-planning 专属）

- **决策守门优先**：§决策守门与本 skill 其余条款不一致时，**以 §决策守门为准**。
- **计划口径硬性**：不按日期 / 不按人日；工作量只用复杂度分级；若被外部材料推着用日期或人日，须显式回退到本 skill 口径。
