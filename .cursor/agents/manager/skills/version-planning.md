# Skill: version-planning

> **Skill 元信息**
>
> - **属于**：[`manager`](../../manager.md)
> - **目的**：在 AI 闭环项目下产出**版本计划 + 版本内任务计划**：策略 → 多角色征询 → 决策点 P-x → 《最终工作计划》经用户定稿。
> - **入口**：`document/plan/`、文件名含 `计划 / 策略 / 路线图 / 版本`，或用户显式声明做计划。
> - **常见参会角色**：`pm`、`architect`、`dev`、`qa`、`ued`；可调阅 `review-meeting` skill 的纪要作为输入。
> - **产物**：《工作策略》、《分角色征询摘要》、《决策点 P-x 记录》、《最终工作计划》（用户定稿后冻结）。
> - **共性条款（必读）**：参考 [`manager.md`](../../manager.md) §**子代理正文绑定**、§**来源选型**、§**节奏**、§**可用角色映射**、§**调用方式说明**。本文不重复其内容，只描述 version-planning 的工作流细节。

## 计划口径（硬性）

- 不采用传统**按日期排期**作为主输出（允许仅作备注锚点）。
- 不使用**人日 / 人周**作为主估算。
- 工作量统一使用**复杂度分级**（S / M / L / XL 或 1~5 级）。
- 计划核心产物必须是：**版本计划**（Version Plan）+ **版本内任务计划**（Task Plan by Version）。
- 任一涉及多子代理角色的断点启动前，必须先按 manager.md §来源选型 与用户确认 **主会话模拟** 或 **Task 子代理**；未确认不得推进。

## 决策点编号约定

- 策略级分叉：`S-<序号>`
- 计划主决策点：`P-<序号>`
- 计划修订（变更控制）：`CR-<序号>`

## 与 review-meeting / todo-drive 的衔接

| 关系 | 说明 |
|------|------|
| `review-meeting` → 本 skill | 纪要中 📅 行动项作为「计划草稿素材」；冲突项升级为新 `P-x` 决策 |
| 本 skill → `todo-drive` | 终稿《最终工作计划》给出**标题级**移交清单 + 跟踪源/区段指针；不写 `[x]` |
| `todo-drive` → 本 skill（反向） | 条目阻塞或与定稿冲突 → 按 `CR-x` 走变更控制流程 |

**边界**：本 skill 不做 `[ ]` 勾选、催办条目、证据四要素回收（属于 `todo-drive`）；只保证「计划写什么、谁先谁后、何谓完成」在团队视角下一致。

## 可用角色映射（征询对象）

下列为计划征询时的角色库；援引某立场前**先读** `.cursor/agents/<name>.md`，产出须对齐该文的工作原则与输出结构摘要。

| 职责 | 子代理名 | 计划轮次典型贡献 |
|------|----------|------------------|
| 需求/范围/验收口径 | `pm` | 范围、里程碑验收、CR 影响、砍 scope 后果 |
| 架构/技术与风险 | `architect` | 可行性、复杂度、门禁、分期策略 |
| 研发落地 | `dev` | 复杂度量级、并行度、拆分风险、HarmonyOS / hdc 验证可行性 |
| 测试 | `qa` | 可测增量、矩阵与条件验收、发布裁决窗口 |
| 交互/体验 | `ued` | 体验依赖节点、状态覆盖、与设计稿对齐成本 |
| 评审收口 | manager · `review-meeting` | 待定决策回填计划、行动项粒度 |

## 主工作流程（必须由 manager 驱动）

全流程分**四轮**；**默认交互式**：未经用户确认的决策点不前进。用户可声明 `【编排模式：批量】` 允许多决策点同窗，须在《计划纪要》注明「批量未逐条确认」。

### 多角色启动方式确认（硬性前置）

凡出现以下任一场景，均视为「多角色断点」：

- 轮二「分角色建议与风险」启动前；
- 任一轮内新增第 2 个及以上子代理参与时；
- 计划修订轮中再次发起跨角色征询时。

在断点启动前必须先单独输出（与 manager.md §来源选型 一致）：

`【多角色启动方式确认】请确认本断点采用：① 主会话模拟 ② Task 子代理`

用户未明确选择前，不得进入多角色内容输出。  
若用户回复「沿用上次方式」，需先复述当前沿用值再继续。  
**沿用失效条件**：版本切换 / 目标变化 / 新增角色任一发生时，必须重新发起一次启动方式确认。

### 轮一 · 工作策略（manager 主笔）

在读取用户目标 + 关键文档（如 `document/TODO-跟踪.md`、定稿需求 / 技术方案 / 纪要路径）后，产出《工作策略》草案，至少含：

- **目标与范围假设**（含明确**非目标**）
- **阶段划分逻辑**（探索 / 骨架 / 核心功能 / 联调发版 等可改名）
- **协调策略**：拟征询哪些角色、为何、何顺序
- **版本切分原则**：按业务价值、风险前置、依赖顺序拆为 V1 / V2 / …
- **复杂度评估口径**：采用 S / M / L / XL（或 1~5），并给判定规则
- **已知信息缺口**（需用户补一句或下一文件锚点）
- **本轮是否拟启用 Task 征询**（默认否；启用则公示范围）

固定一行：`【编排推断】计划目标：<…>；拟征询子代理：<序列>；跟踪入口：<路径；执行落地见 todo-drive>`。

**暂停语（轮一末，若存在策略级分叉）**  
`【决策点 S-1】……（选项 / 取舍）。请确认或修正工作策略；确认后进入分角色征询。`

### 轮二 · 组织协调：分角色建议与风险

按轮一公布的顺序，对每个已配置子代理执行：

- **读** `.cursor/agents/<name>.md`
- **输出** `【<name> · 计划征询】`：**建议（What/Why）** + **风险（含触发条件）** + **依赖他角色的前提**（若有）

**取证方式**（与 manager.md §来源选型 一致；须先完成上文「多角色启动方式确认」）：

| 方式 | 说明 |
|------|------|
| **主会话模拟（默认）** | 当前会话内读 md 后按该角色口吻写短章，利在连贯与停顿 |
| **Task 子代理（可选）** | 用户声明「Task 征询 / 子代理出意见」时：一角色一 Task；manager 在主会话汇编；截断则回退模拟并标注 |

**轮二末**产《分角色征询摘要》表：角色 ｜ 关键建议 ｜ 关键风险 ｜ 与其他角色的冲突点。

### 轮三 · 决策点清单（用户确认）

由 manager 从**策略** + **分角色摘要**中抽出**待用户决策项**，编号 `P-1 … P-n`，每条含：背景、选项、各角色立场摘要、manager 的倾向（可选，不替用户拍板）。

**硬性节奏**：每助手回合优先只推进一个 `P-x`（除非用户声明 `【编排模式：批量】`）。

**固定暂停语**  
`【决策点 P-x】……请从下列选项中选择或给出新约束：`（列 2~3 个可执行选项）；**禁止**在用户未作答时默认选中。

用户确认后记一行 `P-x 结论：…（记入终局计划伏笔）`。

### 轮四 · 《最终工作计划》汇总与定稿

在全部 `P-x` 已收敛（或用户声明「接受暂缓项」并排入风险）后，产出《最终工作计划》草稿，默认结构：

- **计划目标与范围**（已定稿口径）
- **版本计划**（版本编号 ｜ 版本目标 ｜ 准入条件 ｜ 退出条件 ｜ 关键风险）
- **版本内任务计划**（任务包 ｜ 复杂度 ｜ 前置依赖 ｜ 责任**角色** ｜ 验收口径）
- **关键路径与并行轨**（按版本）
- **风险登记与升级触发**
- **移交 todo-drive 包**（仅**标题级**动作列表 + 指向跟踪源/条目区段，不写 `[x]`）
- **暂缓 / 留档项**（与决策记录对齐）

#### 版本模板（固定字段）

| 字段 | 必填说明 |
|------|----------|
| 版本编号 | `V1 / V2 / …` |
| 版本目标 | 单句可验收目标 |
| 准入条件 | 开始该版本前必须满足 |
| 退出条件 | 完成该版本必须满足 |
| 任务清单 | `Vx-Ty` + 复杂度 + 依赖 + 责任角色 |
| DoD 映射 | `Vx-Dy` 对应任务与证据 |

#### 复杂度分级（默认）

| 级别 | 含义（默认） | 计划用途 |
|------|--------------|----------|
| S | 单模块小改，依赖少，验证面窄 | 可并入当前版本 |
| M | 跨 2~3 模块或中等验证成本 | 作为版本内主任务 |
| L | 跨层改造、依赖多、验证面大 | 作为版本关键路径任务 |
| XL | 高不确定 / 高风险 / 跨团队 | 拆分子任务并前置风险验证 |

复杂度仅用于**排序、切分版本、识别风险**，不等同于人日换算。

### 定稿门禁（硬性）

- **不得**在未获用户明示前标注计划状态为「已对齐 / 已定稿」。
- **固定终稿暂停语**：  
  `【《最终工作计划》待你确认】以上为汇总终稿草案。若认可请回复 确认定稿 或逐项修正编号段落；修正后我将升版并重提本确认。`
- 用户回复 **`确认定稿`** 后，本轮产出标记 `状态：计划已定稿（用户确认于 <日期或可略>）`，此后的执行编排以 `todo-drive` skill 与本终稿并列为准。

## 变更控制（CR）

对已定稿计划的任何修改，必须走 `CR-x`：

1. 标注变更来源（需求变更 / 风险触发 / 工具阻塞 / 质量裁决）。
2. 关联原决策点（`P-x`）与受影响版本（`Vx`）。
3. 输出影响评估：范围、复杂度、DoD、风险。
4. 以**新决策点**请求用户确认后才生效。

## 输出格式（按轮拆分，可合并在一个长回复中用二级标题分界）

```text
## 《工作策略》
## 【编排推断】
## 【分角色征询摘要】（轮二）
## 【决策点 P-x 记录】（轮三增量追加）
## 《最终工作计划》草案 → 待用户确认 → 已定稿（仅用户确认后写）
```

## 与 review-meeting 会后

纪要中 📅 行动项：纳入**下一轮计划修订**素材，或移交 `todo-drive`；若与已定稿里程碑冲突 → 新开 `P-x` 决策或用户授权 `CR-x` 修订终稿。

## 注意事项（version-planning 专属）

- **用户权威**：策略分叉、范围 / 节奏取舍、终稿一律用户确认。
- **不做**：TODO 勾选、执行催办、证据回收（属于 `todo-drive`）。
- **不臆造**：复杂度估算与依赖关系若信息不足，必须升级为 `P-x` 决策点而非自行裁定。
- 中文，结论先行，表格优先；若提及日期仅作辅助锚点，不作为主排期口径。

**状态（manager 自用，勿冒充用户拍板）**：`[策略草案 / 征询中 / 决策点未闭 / 终稿待用户确认 / 计划已定稿]`
