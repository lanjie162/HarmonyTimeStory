# Skill: review-meeting

> **Skill 元信息**
>
> - **属于**：[`manager`](../../manager.md)
> - **目的**：主持评审会议，按议程逐模块、逐决策点收敛多角色意见，产出**《评审决策纪要》**（标准化产物，含 📅 改进 TODO 表与 §可选 · 打包推荐）。
> - **入口**：`document/meeting/`、文件名含 `评审 / 评议 / PRD / 技术方案 / 原型 / UI`，或用户显式声明评审。
> - **常见参会角色**：`pm`、`architect`、`dev`、`qa`、`ued`。
> - **专属编号**：`D-MN-y`（模块内）/ `G-x`（跨模块全局）。详见 [`manager.md`](../../manager.md) §决策点编号 - 总集。
> - **来源选型触发**：**每模块开场**（阶段二步骤 1）。规则按 [`manager.md`](../../manager.md) §来源选型 §节奏 §暂停语总集 执行，本文不重述。
> - **主入参**：任意上游材料（需求文档 / 技术方案 / 原型 / UI 稿等）+ 评审命题。
> - **主出参**：标准化《评审决策纪要》（路径与字段见下文 §评审产物标准化）。

## 评审类型与参会角色自动推导

用户只需给出 **评审材料路径** 和/或 **一句话意图**；按下表推导评审类型与参会子代理列表，**不必**让用户逐个写出 `pm`、`architect` 等名称。

### 推导优先级（自上而下命中即停）

1. **用户显式类型**（最高）：用户写「需求评审 / PRD」「技术方案评审 / 架构评审」「交互评审 / UI 评审 / 原型评审」时，直接映射下表。

2. **路径启发式**（默认）

   | 路径或文件特征 | 评审类型 | 参会子代理（建议发言顺序） |
   |----------------|----------|----------------------------|
   | `document/requirement/`、`requirement/`、文件名含 `需求 / PRD / MVP 需求` | PRD/需求评审 | `pm` → `architect` → `dev` → `qa` → `ued` |
   | `document/techdoc/`、`techdoc/`、文件名含 `技术方案 / HarmonyOS 平台能力调研` | 技术方案评审 | `architect` → `dev` → `qa` |
   | `document/design/`、`design/`、文件名含 `UI / 原型 / 线框` | 交互/视觉评审 | `ued` → `pm` → `dev` |
   | `document/meeting/` | 会议纪要跟进 | 默认按主题推断；不明朗时**追问一句**「本次按需求 / 方案 / 交互哪类收口？」 |

3. **文件名/标题启发式**：含「需求说明 / 功能 / 验收」→ 偏需求；含「技术方案 / 架构 / 表结构」→ 偏方案；含「原型 / 交互 / UI」→ 偏交互。

4. **仍模糊**：仅追问 **一句最短问题**（例如「这次审需求还是审交互稿？」），不要罗列子代理名让用户选。

### 推导公示（开场第一段必出）

`【自动推导】评审类型：<类型>；本轮子代理：<name1, name2, …>；依据：<路径/关键词/用户原话>。若有误请直接纠正类型或勾选去掉的角色。`

> 角色缺失处理按 manager.md §子代理正文绑定 第 5 条执行（跳过该角色 + 中立维度补位 + 标注「未配置子代理：`name`」）。

## 评审产物标准化（独立可读 · 无下游依赖）

### 文件命名

```text
document/meeting/[主持人]<YYYY-MM-DD>-<评审类型>-<标题>.md
```

### frontmatter（必填）

```yaml
---
review_type: requirement | architecture | interaction | follow-up
source_doc: <被评审材料相对路径>
participants: [pm, architect, dev, qa, ued]
created_at: <YYYY-MM-DD>
created_by: manager
status: draft | confirmed
---
```

### 正文统一字段（强制章节）

| 分类 | 说明 |
|------|------|
| ✅ 已确认项 | 无争议，可直接推进 |
| ⚠️ 需修改项 | 需按建议调整后确认 |
| ❌ 暂缓 / 拒绝项 | 当前版本不推进 |
| 📅 改进 TODO | 待派发执行的改进项（详见下表） |

### 📅 改进 TODO 表（强制列）

| 编号 | 摘要 | owner_role | dod_hint | priority | task_request |
|------|------|------------|----------|----------|--------------|
| A-1  | …    | dev / qa / ued / architect / pm | 一句话验收提示 | P1 / P2 / P3 | 留空 / `TR-<YYYYMMDD>-<seq>#C-<n>(<status>)` |

字段约束：

- `编号`：`A-<序号>`，纪要内自增；保留作为追溯锚点。
- `owner_role`：建议责任角色，仅取自 manager.md §可用角色映射 中的 `subagent_type`。
- `dod_hint`：一句话「完成意味着什么」的提示，便于后续打包时填 DoD；缺失则填 `TBD`。
- `priority`：`P1` / `P2` / `P3`。
- `task_request`：**派发追溯字段**，由 manager 在打包模式下回写；规则见 manager.md §上游产物 `task_request` 字段。**严禁**写入 `T-<数字>`（信息边界，详见 manager.md §任务请求契约 → 信息边界）。

### 产物落档原则

- 评审纪要本身**独立可读**：不依赖任何下游产物即可作为评审结论档案。
- 是否将 📅 改进 TODO 打包为任务请求或纳入计划修订，由 manager 在用户主动发起时决定（详见 manager.md §上游产物 → 任务请求 转换协议）；本 skill 不做任何下游切换决策。

## 核心工作流

### 阶段一 · 情境识别与团队组建

1. **接收指令**：用户给出评审材料路径和/或评审意图即可；评审类型与参会子代理按上文「自动推导」确定。  
   推导完成后输出 **【自动推导】** 一行公示；用户可说「按推断执行」或纠正类型/增减角色。
2. **动态匹配专家**（与推导表一致，供复核）：
   - **PRD/需求评审**：`pm`、`architect`、`dev`、`qa`、`ued`
   - **技术方案评审**：`architect`、`dev`、`qa`（若存在 `security-reviewer.md` 加入安全审查）
   - **交互/视觉评审**：`ued`、`pm`、`dev`
3. **确认议程**：输出拟邀请专家列表（子代理名）、模块划分与评审重点，等待用户确认或调整。整场来源倾向 / 全流程发言方式（可选公示）按 manager.md §来源选型 §全流程默认 处理；未公示则各模块在阶段二步骤 1 单独选择。

### 阶段二 · 逐模块循环评审

对每个逻辑模块，严格按以下顺序执行，**不跳步、不批量**（批量模式按 manager.md §节奏 执行）。

1. **【模块 MN · 来源】**（必达）  
   - 触发节点：每模块开场。  
   - 选项与暂停语：使用 manager.md §暂停语总集 第 1 行（`【来源】` 模板，前缀写为 `【模块 MN · 来源】`）；其余规则（独占回合 / 沿用 / 失败回退）按 §来源选型 全文执行。  
   - **与步骤 3 绑定**：步骤 3 的角色发言路径必须遵从本步裁定。

2. **呈现当前模块**：输出模块编号、名称、核心内容摘要与评审重点。

3. **分发角色视角**（遵从步骤 1 裁定）：  
   - **主会话模拟路径**：每角色输出前先读 `.cursor/agents/<name>.md`，再输出 **【某某视角】**（详见 manager.md §子代理正文绑定）。  
   - **Task 子代理路径**：按 manager.md §来源选型 对该模块逐个派发 Task；不完整则按其失败回退条款处理。  
   - 示例（模拟时）：「【架构师视角】请评审 [模块 1.x]，重点关注技术可行性与扩展性。」

4. **汇总本模块意见**：按角色逐一输出评审结论，标注来源以便追溯。

5. **识别分歧并列出待决策点**：列出本模块待决策清单 `D-MN-1 / D-MN-2 …`；每条含**争议或不确定性简述**、**各角色立场（若有）**、**主持人建议（可选）**。  
   若 **无任何待决策点**，直接进入步骤 7（模块整体结论），再进入步骤 8（模块级暂停）。

6. **逐个决策点收敛**：使用 manager.md §暂停语总集 第 2 行（`【决策点 D-MN-y】`），按 §节奏 「交互式默认」执行：每回合最多推进一个决策点。  
   用户回复后用一行记录结论（✅ 采纳 / ⚠️ 附加条件 / ❌ 否决 / 📌 暂缓留档）。

7. **模块整体结论**：本模块已无待定决策点（或已暂缓）后，单独一回输出《模块评审小结》：✅ 已确认项 / ⚠️ 需修改项 / 📌 已决策事项摘要；**不得引入新的待定分歧**（新分歧列入下一模块或新开议题）。

8. **【模块 N 完毕】**：使用 manager.md §暂停语总集 第 3 行；用户确认后再进入下一模块（下一模块从步骤 1 开始）。**进入下一模块时不得默认沿用上一模块的来源**，除非用户在该模块步骤 1 中明确回复「沿用」（且议程已公示全流程发言方式）。

> **批量模式**：按 manager.md §节奏 「批量（显式例外）」执行；本 skill 额外约束：**向 Task 委派时必加「仅限本模块、禁止跨模块一气呵成」**。

### 阶段三 · 争议汇总与全局决策引导

在所有模块评审完成（或用户中止）后：

1. **交叉分析**：汇总各模块的争议项，识别跨模块关联冲突（如体验 vs 性能 vs 进度）。  
2. **生成全局决策清单**：

   ```text
   🔴 待决策事项汇总
   1. [模块名] - [争议点]
      - 产品经理: 观点...
      - 架构师: 观点...
      - 建议: ...
   ```

3. **引导逐项决策**：编号 `G-1 / G-2 …`，使用 manager.md §暂停语总集 第 2 行（`【决策点 G-x】`）；**默认每次只呈现 1 个**；用户明确说「本条一起定」时可合并 2~3 条，须在纪要注明。

### 阶段四 · 纪要生成

1. 根据用户决策，更新评审状态。  
2. 按 §评审产物标准化 落档：文件命名、frontmatter、正文统一字段、📅 改进 TODO 表（含 `task_request` 列留空待 manager 后续回写）。
3. 在纪要末尾**追加** §可选 · 打包推荐（见下）。

### §可选 · 打包推荐（manager · 阶段四末尾固定追加）

> 即使用户未声明意图，也须输出本节作为可选打包候选；**不暂停、不询问、不切换** skill；用户后续主动发起打包后由 manager 进入打包模式（见 manager.md §上游产物 → 任务请求 转换协议 阶段 B）。

```text
## 可选 · 打包推荐（manager）

> 本节为可选打包候选，由用户主动决定是否打包。
> 当前没有正在打包的会话上下文。

| 候选编号 | 摘要 | 推荐 source_type | 推荐 owner_role | DoD 充分度 | 已派发 task_request |
|----------|------|------------------|-----------------|-----------|---------------------|
| A-1      | ...  | review           | dev / qa / …    | 充分 / 待补 | 留空 / TR-…#C-… |

**推荐使用方式**：
- `打包候选 A-1,A-3 为 review 任务请求` — 选择性打包并入档
- `先不打包` — 跳过本轮，所有候选保留待后续
```

候选行直接来自 §阶段四 的 📅 改进 TODO 表；`已派发 task_request` 列复制自该表对应字段。

## 会议类型常见检查清单

### 需求评审（PRD Review）

- 评审重点：需求完整性、范围边界、验收标准、风险与冲突。  
- 关注维度：用户价值、技术可行性、测试覆盖、体验一致性。  
- 常见问题：目标用户和场景是否清晰；验收标准是否可测试；异常流程是否覆盖；是否有遗漏或冲突的规则。

### 技术方案评审（Architecture Review）

- 评审重点：设计正确性、扩展性、性能、安全与迁移方案。  
- 关注维度：模块边界、数据模型、接口合约、异常处理、回滚方案。  
- 常见问题：是否有更简单的方案；关键失败点是什么；性能瓶颈和数据一致性如何保证；是否支持灰度、渐进式切换。

### 交互评审（UX Review）

- 评审重点：任务流完整性、状态覆盖、可达性与一致性。  
- 关注维度：主路径、异常路径、空态、加载态、错误态、无障碍。  
- 常见问题：用户能否在 N 步内完成核心任务；所有页面状态是否已设计；行为与反馈是否符合用户预期；文案是否清晰一致、无歧义。

## 注意事项（review-meeting 专属）

- **两级暂停**：模块内有多个待决策点时**逐项停顿**；勿在同一回合塞入多项待裁决事项或「模块小结 + 下一模块」。
- **Task 委派必加范围词**：向 Task 委派时必须写明「**仅限本模块、禁止跨模块一气呵成**」，否则按 §来源选型 失败回退处理。
- **产物独立可读**：评审纪要落档后即作为本 skill 终结产物；是否进入下游执行流由 manager 在用户主动发起时决定，本 skill 不做切换决策、不引用下游 skill 名。
- **信息边界**：📅 改进 TODO 表与 §可选 · 打包推荐 中的 `task_request` 字段**严禁**写入 `T-<数字>`（统一任务列表全局编号）；只能填 `TR-…#C-<n>(<status>)`。详见 manager.md §任务请求契约 → 信息边界。

> 通用注意事项（保持中立 / 灵活适配 / 上下文管理 / 引用规范 / 完整性兜底）见 manager.md §注意事项。
