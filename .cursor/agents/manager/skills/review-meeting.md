# Skill: review-meeting

> **Skill 元信息**
>
> - **属于**：[`manager`](../../manager.md)
> - **目的**：主持评审会议，按议程逐模块、逐决策点收敛多角色意见，产出《评审决策纪要》。
> - **入口**：`document/meeting/`、文件名含 `评审 / 评议 / PRD / 技术方案 / 原型 / UI`，或用户显式声明评审。
> - **常见参会角色**：`pm`、`architect`、`dev`、`qa`、`ued`。
> - **产物**：《评审决策纪要》+《会后移交标准包》。
> - **共性条款（必读）**：参考 [`manager.md`](../../manager.md) §**子代理正文绑定**、§**来源选型**、§**节奏**、§**可用角色映射**、§**调用方式说明**。本文不重复其内容，只描述 review-meeting 的工作流细节。

## 评审类型与参会角色自动推导

用户只需给出 **评审材料路径** 和/或 **一句话意图**；按下表推导评审类型与参会子代理列表，**不必**让用户逐个写出 `pm`、`architect` 等名称。

### 推导优先级（自上而下命中即停）

1. **用户显式类型**（最高）：用户写「需求评审 / PRD」「技术方案评审 / 架构评审」「交互评审 / UI 评审 / 原型评审」时，直接映射下表。

2. **路径启发式**（默认）

   | 路径或文件特征 | 评审类型 | 参会子代理（建议发言顺序） |
   |----------------|----------|----------------------------|
   | `document/requirement/`、`requirement/`、文件名含 `需求 / PRD / MVP 需求` | PRD/需求评审 | `pm` → `architect` → `dev` → `qa` → `ued` |
   | `document/techdoc/`、`techdoc/`、文件名含 `技术方案 / HarmonyOS 平台能力调研` | 技术方案评审 | `architect` → `dev` → `qa` |
   | `document/design/`、`design/`、文件名含 `UI / 原型 / 线框` | 交互/视觉评审 | `ued` → `pm` → `dev` |
   | `document/meeting/` | 会议纪要跟进 | 默认按主题推断；不明朗时**追问一句**「本次按需求 / 方案 / 交互哪类收口？」 |

3. **文件名/标题启发式**：含「需求说明 / 功能 / 验收」→ 偏需求；含「技术方案 / 架构 / 表结构」→ 偏方案；含「原型 / 交互 / UI」→ 偏交互。

4. **仍模糊**：仅追问 **一句最短问题**（例如「这次审需求还是审交互稿？」），不要罗列子代理名让用户选。

### 推导公示（开场第一段必出）

`【自动推导】评审类型：<类型>；本轮子代理：<name1, name2, …>；依据：<路径/关键词/用户原话>。若有误请直接纠正类型或勾选去掉的角色。`

若某角色对应的 `.cursor/agents/<name>.md` 不存在，自动跳过该角色，并以中立视角补充该维度的检查项，同时在纪要中标注「未配置子代理：`name`」。

## 决策点编号约定

- 模块内：`D-<模块号>-<序号>`（示例：`D-M2-1` = 模块 2 第 1 个决策点）
- 全局争议：`G-<序号>`
- 暂缓 / 留档：`📌 D-..-.. 暂缓` 或 `📌 G-.. 暂缓`

## 核心工作流

### 阶段一 · 情境识别与团队组建

1. **接收指令**：用户给出评审材料路径和/或评审意图即可；评审类型与参会子代理按上文「自动推导」确定。  
   推导完成后输出 **【自动推导】** 一行公示；用户可说「按推断执行」或纠正类型/增减角色。
2. **动态匹配专家**（与推导表一致，供复核）：
   - **PRD/需求评审**：`pm`、`architect`、`dev`、`qa`、`ued`
   - **技术方案评审**：`architect`、`dev`、`qa`（若存在 `security-reviewer.md` 加入安全审查）
   - **交互/视觉评审**：`ued`、`pm`、`dev`
3. **确认议程**：输出拟邀请专家列表（子代理名）、模块划分与评审重点，等待用户确认或调整。  
   - **整场来源倾向**：若用户已声明「子代理模式 / 启用子代理发言」，按 manager.md §来源选型 公示 `【发言来源】子代理 Task …`；否则默认主会话模拟，可省略此行。**最终仍以阶段二每模块步骤 1** 为准。  
   - **可选 · 全流程发言方式**：希望减少重复选型时，可公示一行 `【全流程发言方式】主会话模拟` 或 `【全流程发言方式】Task 子代理`；未公示则不写，各模块在阶段二步骤 1 单独选择。

### 阶段二 · 逐模块循环评审

对每个逻辑模块，严格按以下顺序执行，**不跳步、不批量**（批量模式见下文「阶段二（附）」）。

1. **来源确认（模块开场，必达）**  
   - **输出**：`【模块 MN · 来源】` — 请任选其一：① **主会话模拟**（在主会话读取 `.cursor/agents/<name>.md` 后输出视角）；② **Task 子代理**（按 manager.md §来源选型 逐角色 Task 取证，主会话汇编）；③ **沿用【全流程发言方式】**（仅当议程已公示时可用）。  
   - **交互式**：本步原则上 **独占一个助手回合**（只输出选项与暂停语），用户未选型不得进入步骤 2。用户可回复 `1 / 2 / 3` 或 `模拟 / 子代理 / 沿用`。  
   - **快捷**：议程已公示 `【全流程发言方式】` 时可缩写为 `【模块 MN · 来源】沿用全流程：<模拟|Task>。若改为另一方式请回复 改模拟 / 改子代理。`，仍须用户应答后才进步骤 2。  
   - **与步骤 3 绑定**：步骤 3 必须遵从本步裁定。

2. **呈现当前模块**：输出模块编号、名称、核心内容摘要与评审重点。

3. **分发角色视角**（遵从步骤 1 裁定）：  
   - **主会话模拟路径**：每角色输出前先读 `.cursor/agents/<name>.md`，再输出 **【某某视角】**。  
   - **Task 子代理路径**：按 manager.md §来源选型 对该模块逐个派发 Task；不完整则回退模拟，并标注「子代理输出不完整，以下为模拟补强」。  
   - 示例（模拟时）：「【架构师视角】请评审 [模块 1.x]，重点关注技术可行性与扩展性。」

4. **汇总本模块意见**：按角色逐一输出评审结论，标注来源以便追溯。

5. **识别分歧并列出待决策点**：列出本模块待决策清单 `D-MN-1 / D-MN-2 …`；每条含**争议或不确定性简述**、**各角色立场（若有）**、**主持人建议（可选）**。  
   若 **无任何待决策点**，直接进入步骤 7（模块整体结论），再进入步骤 8（模块级暂停）。

6. **逐个决策点收敛（硬性节奏）**  
   - **每一助手回合最多只推进一个待决策点**。只呈现下一个尚未收敛的条目，展开选项或权衡，邀请用户裁定。  
   - **固定暂停语**：`【决策点 D-MN-y】……（选项 / 利弊 / 依赖）。请对本决策点做出选择或补充约束；确认后我将记录结论并进入下一决策点。`  
   - 用户回复后用一行记录结论（✅ 采纳 / ⚠️ 附加条件 / ❌ 否决 / 📌 暂缓留档）；**不得在同一回合内自动展开下一个决策点**。  
   - 重复直至清单全部收敛，或用户指示「跳过 / 会后决策」（须记入纪要待跟踪）。

7. **模块整体结论**：本模块已无待定决策点（或已暂缓）后，单独一回输出《模块评审小结》：✅ 已确认项 / ⚠️ 需修改项 / 📌 已决策事项摘要；**不得引入新的待定分歧**（新分歧列入下一模块或新开议题）。

8. **模块级暂停确认**：  
   - **固定暂停语**：`【模块 N 评审完毕】以上为模块整体结论，请确认或修正。回复「继续」进入下一模块；若要重议某决策点请说明编号（如 D-N-1）。`  
   - **等待用户确认**后再进入下一模块（下一模块从步骤 1 开始）；未确认前不得输出下一模块任何内容。**进入下一模块时不得默认沿用上一模块的来源**，除非用户在该模块步骤 1 中明确回复「沿用」（且议程已公示全流程发言方式）。

### 阶段二（附） · 节奏：交互式（默认）与批量（例外）

> 此节作为对 manager.md §节奏 的具体化补充。

| 模式 | manager 行为 |
|------|--------------|
| **交互式（默认）** | **模块开场**：每个模块的步骤 1（来源）结束后本轮即止，等待用户。**决策点级**：步骤 6 每呈现一个决策点后结束，等待用户；全部收敛后再单独一回步骤 7。**模块级**：步骤 8 后再进入下一模块。**禁止**：未选来源就开评；同一回合叠发多决策点或模块小结+下一模块。 |
| **批量（显式例外）** | 用户明确说「一次性全文评审 / 批量模式 / 不要每模块暂停 / 不要每决策点暂停」时，开场公示 `【会议模式：批量】`，须声明：是否跳过各模块步骤 1（若跳过须给 `【批量默认来源】模拟｜Task`）；是否跳过决策点级 / 模块级暂停。纪要注明「批量产出，未逐项经用户确认的层级」。 |

向 Task 委派时须在指令中写明「**仅限本模块、禁止跨模块一气呵成**」。未指定批量时仍须遵守「每回合最多处理一个决策点并暂停」。

### 阶段三 · 争议汇总与全局决策引导

在所有模块评审完成（或用户中止）后：

1. **交叉分析**：汇总各模块的争议项，识别跨模块关联冲突（如体验 vs 性能 vs 进度）。  
2. **生成全局决策清单**：

   ```text
   🔴 待决策事项汇总
   1. [模块名] - [争议点]
      - 产品经理: 观点...
      - 架构师: 观点...
      - 建议: ...
   ```

3. **引导逐项决策**：**默认每次只呈现 1 个全局争议项**，编号 `G-1 / G-2 …`，格式与阶段二步骤 6 相同；**暂停等待用户**后再呈现下一项。**例外**：用户明确说「本条一起定」时可合并 2~3 条，须在纪要注明。  
4. **暂停等待**：明确提示「请对本决策点做出决策，或指示跳过/调整；完成后进入下一全局决策点。」

### 阶段四 · 纪要生成

1. 根据用户决策，更新评审状态。  
2. 输出《评审会议纪要》，包含以下字段：

   | 分类 | 说明 |
   |------|------|
   | ✅ 已确认项 | 无争议，可直接推进 |
   | ⚠️ 需修改项 | 需按建议调整后确认 |
   | ❌ 暂缓/拒绝项 | 当前版本不推进 |
   | 📅 后续行动项 | 负责人、截止时间、依赖说明 |

### 会后移交标准包（给 version-planning / todo-drive）

当纪要包含可执行行动项时，补充输出：

| 字段 | 说明 |
|------|------|
| 行动项ID | 建议 `A-1 / A-2 …` |
| 建议归属版本 | 如 `V2 / V3`，不确定则标「待 PM 定版」 |
| 建议复杂度 | `S / M / L / XL` |
| 建议 Owner 角色 | 子代理角色名（pm / architect / dev / qa / ued） |
| 是否 MCP / hdc 必选 | 是 / 否（真机验证类建议为是） |

后续：

- 行动项进入 **版本计划修订** → 切到 `version-planning` skill；  
- 行动项作为 **待办条目** → 切到 `todo-drive` skill 落跟踪行并驱动收口。

## 会议类型常见检查清单

### 需求评审（PRD Review）

- 评审重点：需求完整性、范围边界、验收标准、风险与冲突。  
- 关注维度：用户价值、技术可行性、测试覆盖、体验一致性。  
- 常见问题：目标用户和场景是否清晰；验收标准是否可测试；异常流程是否覆盖；是否有遗漏或冲突的规则。

### 技术方案评审（Architecture Review）

- 评审重点：设计正确性、扩展性、性能、安全与迁移方案。  
- 关注维度：模块边界、数据模型、接口合约、异常处理、回滚方案。  
- 常见问题：是否有更简单的方案；关键失败点是什么；性能瓶颈和数据一致性如何保证；是否支持灰度、渐进式切换。

### 交互评审（UX Review）

- 评审重点：任务流完整性、状态覆盖、可达性与一致性。  
- 关注维度：主路径、异常路径、空态、加载态、错误态、无障碍。  
- 常见问题：用户能否在 N 步内完成核心任务；所有页面状态是否已设计；行为与反馈是否符合用户预期；文案是否清晰一致、无歧义。

## 注意事项（review-meeting 专属）

- **保持中立**：不偏袒任何角色，客观呈现技术权衡。  
- **灵活适配**：若某角色未配置，自动跳过并补充该维度检查项。  
- **上下文管理**：若文件过大，主动建议分批次评审。  
- **两级暂停**：模块内有多个待决策点时**逐项停顿**，勿在同一回合塞入多项待裁决事项。**每模块**须先落实步骤 1（批量模式豁免须在开场写明）。  
- **引用规范**：所有建议必须指明来源角色，便于追溯；若该段来自 Task，可注明「子代理」。  
- **完整性兜底**：Task 返回过短或与模块无关时，**必须回退模拟或重试单次 Task**，不得空白进入决策点。
