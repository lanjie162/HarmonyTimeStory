# Agent 治理护栏落地 — 细化差异说明（通用原则版）

> 本文档为计划的 **逐文件改动清单**，便于 diff 式审阅；**尚未对仓库做任何修改**。
>
> **原则升级**：不以「DB / 落库」为中心，而以 **权威基线 ↔ 任务声称（DoD）↔ 可复核证据** 的一致性为中心；持久化仅是 **高约束域** 的一类示例（同类还有性能门禁、安全合规、平台 API 契约、交互定稿、可用性指标等）。

---

## 0. 通用治理原则（专条与各处引用的统一语义）

| 概念 | 通用定义（不限领域） |
|------|----------------------|
| **权威基线** | 仓库内已定稿、可被引用的段落：`document/techdoc/*`、`document/requirement/*`、已定稿 ADR、`document/design/*` 等；任务必须 **显式锚点**，禁止只有执行侧口头复述。 |
| **声称** | 任务 `DoD` / 计划「验收口径」对「完成态」的描述（功能、非功能、边界均可）。 |
| **证据层级（L1 / L2）** | **L1**：低成本演示级证据（桩、Mock、单路径脚本、抽象上下文演练、开发期环境）—— **仅当 DoD 明文允许 L1 且与权威基线不冲突**。**L2**：与声称 **同级** 的可复核证据（真机/真数据/平台 API 实测、指标日志、回归自动化、符合定稿交互稿的走查材料等）—— **当权威基线对该能力规定了验收强度，或 DoD 使用了「完成/上线/符合定稿」等强声称时默认需要**。 |
| **高约束域** | **不是关键词表绑死**，而是判断规则：**若用 L1 级证据会通过、但权威基线或产品 Must 实际要求更强验证**，则该任务落入高约束域，必须 **要么** DoD 升级为 L2 证据 **要么** **`CR-x`** 砍 scope / 降级基线并留档。 |
| **占位** | 任何「临时实现 / 空壳 / 非最终真相源」须在 DoD 登记 **到期与替换条件**；跨版本延续须 **`CR-x`**。 |
| **冲突** | 计划、任务与权威基线 **字面或强度不一致** → 禁止执行侧自行收窄 → **`CR-x`** 修订其一。 |

**示例（非穷尽）**：持久化、迁移、加密存储；**性能与稳定性**（启动时延、内存、ANR）；**安全与隐私**（权限、数据出境）；**HarmonyOS 能力契约**（PhotoPicker、后台任务、Vision）；**交互与可用性**（与 `UI v1.2` 关键句对齐）；**合规证据**（日志留存、崩溃上报接入）。均适用同一套 **锚点 + 声称强度 + 证据同级** 逻辑。

## 1. 新增文件（pure addition）

**路径（建议）**：`.cursor/agents/GOVERNANCE-权威基线与可验证一致性.md`  
（若团队偏好短名：`GOVERNANCE-基线DoD一致性.md`；下文简称 **「专条」**。）

**建议目录结构（全文约 90～130 行）**：

| 章节 | 内容要点 |
|------|----------|
| 1 目的 | 防止 **任务 DoD / 验收口径** 在强度或范围上 **窄于或偏离** 权威基线；与 **`CR-x`**、统一任务列表契约配套。 |
| 2 适用范围 | `task-dispatch` DoD、`version-planning` 验收口径、`task-quality`、`task-accept` 守门 |
| 3 权威锚点 | `DoD` 须含 **`权威：` `<路径>#锚点`**（适当时多条）；无锚点则仅限纯工程债/文档债且须在摘要标明 |
| 4 声称与证据同级 | L1/L2 定义同 §0；**禁止 L1 证据支撑 L2 声称**（除非 DoD 显式写成「L1 演示通过」且 pm/计划已确认砍 scope） |
| 5 高约束域判定 | 采用 §0 规则 + 简短决策树（执行/质检时可.walk） |
| 6 占位三要素 | **到期**（`Vx-Ty` 或版本）、**替换条件**、延期路径 **`CR-x`** |
| 7 冲突与升级 | 计划/任务 vs 基线 → **`CR-x`**；禁止静默缩水 |

**说明**：专条内可保留 **一小段「示例锚点」**，列举 2～3 类（如技术方案 §4、需求 §x、UI 原型 §y），**不应**把半篇篇幅写给 DB。

## 3. `.cursor/agents/manager/skills/task/task-dispatch.md`

**改动类型**：扩展 **`DoD` 字段约束**。

**现状（摘录）**：

```markdown
- `DoD`：**必填**，「完成意味着什么」的可验证标准（一段话或编号清单）；缺失则填 `TBD` ...
```

**细化后（在原 bullet 下追加）**：

| 新增约束 | 通用说明 |
|----------|----------|
| **权威锚点** | 凡依赖已定稿需求/技术/设计/ADR 的：`DoD` 含 **`权威：`** 行（路径+锚点）；多来源可多条 |
| **证据层级** | 凡 **强声称**（上线就绪、符合定稿、通过门禁、满足 §x 非功能指标）：`DoD` 标明 **`证据：L1`** 或 **`证据：L2`**（定义见专条）；未标明且摘要含强声称时，默认按 **L2** 理解或由派发轮澄清为 **TBD** |
| **占位** | 交付物为临时/桩/单路径：`DoD` 含 **`占位到期：`** + **`CR-x` 续期规则**；否则维持 **TBD** / **🟡 待补充**，不建议直接接收 |

**差异语义**：不增加表格列；约束写在 **DoD 单元格写法**。

**示例（说明性质，可写入专条附录而非 dispatch 正文）**：  
- 性能类任务：`权威：调研表 §4` + `证据：L2（真机指标+log）`  
- 交互对齐：`权威：UI 原型 §4.5` + `证据：L2（走查表或截图链）`  
- 持久化类：`权威：技术方案 §4` + `证据：L2` —— **仅作高约束域的一类**，与其它类别平列。

**改动类型**：扩展 **`验收口径`** 字段约束。

**追加（通用表述）**：

- 当任务触及 **已定稿基线**（需求章、技术方案、交互稿、调研表结论）：`验收口径` 须 **引用锚点** 或指向 **「见 Vx 计划 Out」**，不得仅写「完成开发/功能 OK」。

## 7. `.cursor/agents/dev.md`

**草案（通用）**：

```markdown
- **声称与交付同级**：`DoD` 引用权威锚点时，代码与自测证据须与该锚点 **同一强度**；禁止用演示桩覆盖交付级声称。
```
